# Module 3: Blockchain Transaction Structure

## Introduction

In blockchain technology, a transaction is far more than just a transfer of cryptocurrency. It is the fundamental atomic unit that represents a state change on the distributed ledger. For  engineering students, understanding the intricate structure of a transaction is crucial, as it forms the basis for building secure and reliable decentralized applications (dApps). This module deconstructs the anatomy of a standard transaction, using Bitcoin as the primary example due to its well-defined structure, while also noting how this evolves in other platforms like Ethereum.

## Core Concepts of a Transaction Structure

A transaction is a digitally signed data structure that encodes the transfer of value or information between participants. Its primary purpose is to prove that a sender intends to transfer something they own to a new owner.

### 1. Transaction Inputs

Inputs are references to previous transaction outputs (UTXOs - Unspent Transaction Outputs). They prove the origin of the funds or data being spent. Think of them as the "source" of the coins.

*   **Previous Tx Hash (Pointer):** The hash of the transaction where the UTXO was originally created.
*   **Index (vOut):** The specific output index within the referenced transaction (e.g., Output #0, #1).
*   **ScriptSig (Unlocking Script):** This contains two critical components:
    *   **Signature:** A cryptographic signature generated with the sender's private key. This proves ownership of the funds being spent.
    *   **Public Key:** The full public key corresponding to the private key used to create the signature.

**Example:** If Alice received 1 BTC in a past transaction (hash: `abc123...`) at output index `0`, she must reference `abc123...` and index `0` as an input to send those coins to Bob.

### 2. Transaction Outputs

Outputs define the new ownership of the transferred value and the amount. They become new UTXOs, ready to be spent in a future transaction.

*   **Value:** The amount of cryptocurrency (e.g., in Satoshis for Bitcoin) being assigned to this output.
*   **ScriptPubKey (Locking Script):** A script that specifies the conditions that must be met to spend this output in the future. It typically contains the recipient's public key hash (address). The ScriptPubKey acts as a cryptographic lock that can only be opened by the correct private key.

**Example:** Alice creates an output that sends 0.8 BTC to a ScriptPubKey containing a hash of Bob's public key. Only Bob, with his corresponding private key, can create a signature to unlock and spend this output later.

### 3. Metadata

These fields provide essential information about the transaction itself.

*   **Version Number:** Indicates which set of rules the transaction follows. This allows the network protocol to evolve over time.
*   **Transaction ID (TxID):** A unique hash identifier for the transaction, generated by hashing the transaction data (double SHA-256 in Bitcoin). This is the primary reference for the transaction.
*   **Input Count & Output Count:** The number of inputs and outputs in the transaction.
*   **Locktime:** Defines the earliest time or block number at which a transaction can be added to the blockchain. If set to zero, the transaction is valid immediately.

## The Transaction Lifecycle (Simplified)

1.  **Creation:** A user's wallet software constructs a transaction, specifying inputs (sources of funds) and outputs (recipients and amounts).
2.  **Signing:** The wallet cryptographically signs the transaction with the private keys corresponding to the inputs, creating the `ScriptSig`.
3.  **Propagation:** The signed transaction is broadcast to the peer-to-peer network.
4.  **Validation:** Network nodes (miners/validators) verify the transaction's validity: checking signatures, ensuring inputs are unspent, and confirming the sum of inputs ≥ sum of outputs.
5.  **Mining:** Valid transactions are grouped into a block by a miner.
6.  **Confirmation:** Once the block is mined and added to the chain, the transaction is confirmed. The outputs of the transaction become new UTXOs on the ledger.

## Beyond Bitcoin: Ethereum's Account-Based Model

It's important to note that not all blockchains use the UTXO model. Ethereum uses an **account-based model**, similar to a bank ledger. Transactions don't reference previous outputs but instead update the state of sender and recipient accounts. An Ethereum transaction includes fields like `nonce` (preventing replay attacks), `gasPrice`, and `gasLimit` to manage computation fees on the network.

## Key Points & Summary

*   **Atomic Unit:** A transaction is the basic element that changes the state of the blockchain ledger.
*   **Inputs vs. Outputs:** Inputs are references to previous unspent outputs (proof of source), while outputs define new ownership and value (creating new UTXOs).
*   **Scripts are Key:** The `ScriptSig` (unlocking script) and `ScriptPubKey` (locking script) work together to cryptographically prove ownership and authorize transfers.
*   **Cryptographic Security:** Digital signatures, created with private keys, are fundamental to proving authority and preventing fraud.
*   **Hashing for Integrity:** The Transaction ID is a unique hash that serves as a fingerprint for the transaction data.
*   **Model Difference:** Understand the distinction between Bitcoin's **UTXO model** (like cash) and Ethereum's **account-based model** (like bank accounts).
*   **Purpose:** This structure ensures that only the rightful owner can spend funds and that all network participants can independently verify the validity of every transaction, achieving decentralization and trustlessness.
