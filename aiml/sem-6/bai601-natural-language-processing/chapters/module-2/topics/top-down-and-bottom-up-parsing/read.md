## Module 2: Top-down and Bottom-up Parsing

### 1. Introduction

In the compilation process, **parsing** (or **syntax analysis**) is the phase where the compiler checks the syntactic structure of a string of tokens (from the lexer) against the rules of a **context-free grammar** (CFG). Its goal is to determine if the token string can be generated by the grammar and, if so, to produce a **parse tree**. Two fundamental strategies for constructing this parse tree are **Top-down Parsing** and **Bottom-up Parsing**. Understanding the difference between them is crucial for compiler design.

### 2. Core Concepts

#### Top-down Parsing

**Concept:** A top-down parser starts with the grammar's start symbol (at the root of the parse tree) and repeatedly applies production rules to rewrite non-terminal symbols until it derives the entire input string of tokens. It constructs the parse tree from the top (root) down to the leaves (tokens).

*   **Analogy:** It's like solving a problem by starting with a high-level goal and recursively breaking it down into smaller sub-problems until you reach the base facts.
*   **Key Characteristic:** It is a **goal-directed** or **predictive** approach. The parser predicts which production rule to apply next to match the input.
*   **Common Type:** **Recursive Descent Parsing** is a classic example, often implemented manually. A more efficient and popular method is **LL(1) Parsing**, which uses a parsing table to make predictions based on the next input token (the 1 stands for one token lookahead).

**Example:**
Grammar: `S â†’ aS | b`
Input: `a a b`

**Top-down Parse Steps:**
1.  Start with `S`.
2.  To match the first `a`, apply `S â†’ aS`. Now we have `a S`.
3.  Look at the next `a`, apply `S â†’ aS` again. Now we have `a a S`.
4.  Look at the next `b`, apply `S â†’ b`. Now we have `a a b`, which matches the input.
The parse tree is built from `S` down to `a a b`.

#### Bottom-up Parsing

**Concept:** A bottom-up parser starts with the input string of tokens (the leaves of the parse tree) and uses production rules in reverse to reduce the string back to the start symbol of the grammar. It constructs the parse tree from the bottom (leaves) up to the root.

*   **Analogy:** It's like building a structure by first assembling small components into larger modules, which are then joined together to form the final product.
*   **Key Characteristic:** It is a **shift-reduce** approach. The parser shifts (reads) tokens onto a stack and then reduces a sequence of tokens on the stack to a non-terminal symbol (using the bodyâ†’head of a production rule) when it recognizes a right-hand side (RHS).
*   **Common Type:** The most prevalent type is **LR Parsing** (e.g., LR(0), SLR(1), CLR(1), LALR(1)), which is powerful and can handle a wide range of grammars. It's often generated automatically by tools like YACC or Bison.

**Example:**
Grammar: `S â†’ S + T | T` and `T â†’ id`
Input: `id + id`

**Bottom-up Parse Steps (Simplified):**
1.  Shift `id` onto the stack. Reduce `id` to `T` (using `T â†’ id`). Stack: `T`.
2.  Reduce `T` to `S` (using `S â†’ T`). Stack: `S`.
3.  Shift `+`. Stack: `S +`.
4.  Shift `id`. Stack: `S + id`. Reduce `id` to `T`. Stack: `S + T`.
5.  Reduce `S + T` to `S` (using `S â†’ S + T`). Stack: `S` (start symbol). Input is accepted.

### 3. Comparison and Key Differences

| Feature | Top-down Parsing | Bottom-up Parsing |
| :--- | :--- | :--- |
| **Starting Point** | Start symbol (Root of tree) | Input string (Leaves of tree) |
| **Direction** | Expands tree from root to leaves | Builds tree from leaves to root |
| **Primary Operation** | Expand non-terminals using productions | Reduce terminals/non-terminals to their parent non-terminal |
| **Grammar Type** | Prefers **LL(k)** grammars. Often cannot handle left-recursion. | More powerful. Handles **LR(k)** grammars, including left-recursive ones. |
| **Lookahead** | Typically uses one token (LL(1)) | Can use more lookahead, but common methods use one (LR(1)) |
| **Ease of Implementation**| Easier to implement manually (Recursive Descent) | Generally more complex, best done with parser generators |
| **Power & Flexibility** | Less powerful | More powerful and flexible |

### 4. Key Points & Summary

*   **Purpose:** Both techniques aim to validate syntax and build a parse tree, but they approach the problem from opposite ends.
*   **Top-down** parsers are **predictive**. They begin with a goal and work forward, making predictions about the structure. They are intuitive but can be limited by grammar features like left recursion.
*   **Bottom-up** parsers are **reductive**. They start with the data and work backward, grouping tokens until the original goal is met. They are more powerful and are the basis for most automated parser generators.
*   **Choice:** The choice between them often depends on the grammar of the language and the tools available. For simple languages, a hand-written top-down parser might suffice. For complex programming languages, an automatically generated bottom-up (LALR) parser is standard.

**In essence:** Top-down parsing asks, "What rules can I apply to generate this input?" while bottom-up parsing asks, "What rules could have been applied to produce this sequence of tokens?"
