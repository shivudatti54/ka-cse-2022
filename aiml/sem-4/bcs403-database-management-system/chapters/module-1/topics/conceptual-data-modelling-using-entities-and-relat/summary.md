# **Conceptual Data Modelling using Entities and Relationships**

**Module: DATABASE MANAGEMENT SYSTEM, 8 Hours**

### Key Concepts

- **Entity Types:**
  - A collection of related data that have a unique identifier (PID)
  - Define the concept of a data entity
- **Entity Sets:**
  - A group of related entities of the same type
  - Can be related to other entity sets through relationships
- **Structural Constraints:**
  - Define the relationships between entities and sets
  - Can be defined using cardinality (e.g., 1:N, M:N)
- **Weak Entity Types:**
  - Entities that require the existence of a stronger entity to exist
  - Typically have a partial identifier (PID)
- **ER Diagrams:**
  - A visual representation of the data model
  - Used to illustrate the relationships between entities and sets
- **Specifications:**
  - Define the meaning of the data model
  - Includes entity-relationship diagrams, cardinality rules, and data types

### Important Formulas and Definitions

- **Cardinality:** The number of entities that can be related
  - 1:N (one-to-many)
  - M:N (many-to-many)
- **Relationships:** A connection between two or more entities
- **Generalization-Specialization (ISA):** A relationship where a weaker entity is a specialization of a stronger entity
- **Part-Whole (IS-A):** A relationship where a part is a component of a whole

### Theorems and Concepts

- **Entity Integrity:** The principle of ensuring that each entity has a unique identifier
- **Referential Integrity:** The principle of ensuring that relationships between entities are consistent
- **Functional Dependency:** A relationship where one entity determines the value of another entity

### Quick Revision Tips

- Focus on understanding the key concepts of entity types, entity sets, and structural constraints
- Practice drawing ER diagrams to visualize relationships between entities
- Review cardinality rules and data types to ensure accurate implementation
- Familiarize yourself with theorems and concepts related to entity integrity, referential integrity, and functional dependency
