# Module 4: Activities Covered by Software Project Management

**Subject:** Software Engineering & Project Management
**Semester:** V
**Duration:** 10 Hours

---

## Introduction

Software Project Management (SPM) is the application of knowledge, skills, tools, and techniques to project activities to meet the project requirements. Unlike conventional project management, SPM has a unique focus on the intangible nature of its product, the high rate of change, and the complex logical relationships. It is a structured and disciplined approach to ensure that software is delivered on time, within budget, and with the required quality. The core activities of SPM form a lifecycle that guides a project from inception to completion.

## Core Activities of Software Project Management

The activities of SPM are interconnected and often iterative. They can be broadly categorized into the following key areas:

### 1. Project Planning
This is the foundational activity where the roadmap for the project is created. It involves defining the scope, objectives, and the course of action to achieve them.
*   **Scope Management:** Defining and documenting the project's goals, deliverables, tasks, costs, and deadlines. A clear scope prevents "scope creep."
*   **Effort, Cost, and Schedule Estimation:** Using techniques like Function Point Analysis, Use-Case Points, or COCOMO to predict the effort required (in person-months), which directly influences cost and project schedule.
*   **Resource Planning:** Identifying and allocating human resources, hardware, software, and other tools needed for the project.
*   **Risk Planning:** Proactively identifying potential risks (technical, managerial, organizational) and creating mitigation strategies.
*   **Quality Planning:** Defining the quality standards and how they will be achieved.
*   **Example:** For a new "Student Portal" project, planning would involve listing features (scope), estimating it will take 6 developers 4 months (effort/schedule), budgeting for licenses (cost), and identifying a risk like "integration with legacy university systems."

### 2. Project Monitoring and Control
This is an ongoing activity that runs concurrently with project execution. It ensures the project stays on track according to the plan.
*   **Tracking Progress:** Regularly comparing actual progress (completed tasks, spent effort) against the project plan using tools like Gantt charts or Burn-down charts.
*   **Change Management:** Handling requests for changes to the project scope in a controlled manner, assessing their impact on schedule and cost before approval.
*   **Risk Monitoring:** Continuously tracking identified risks and watching for new ones.
*   **Example:** During the development of the portal, the project manager notices that the "fee payment module" is taking 20% longer than estimated. This triggers a control action, such as reallocating resources to get back on schedule or officially revising the timeline.

### 3. Project Scheduling
This activity translates the project plan into a timeline. It involves sequencing tasks, assigning durations, and allocating resources to create a concrete schedule.
*   **Task Breakdown:** Using a Work Breakdown Structure (WBS) to decompose the project into smaller, manageable tasks.
*   **Task Sequencing:** Defining the order of tasks and their dependencies (e.g., Task B cannot start until Task A is finished) using network diagrams.
*   **Milestone Identification:** Defining key checkpoints or major deliverables (e.g., "Completion of UI Prototype") to measure progress.
*   **Example:** The "Develop Login Module" task is broken down into: (1) Design UI, (2) Code backend, (3) Implement security, (4) Test. These tasks are sequenced and assigned to specific team members with start and end dates.

### 4. Resource Management
This involves efficiently utilizing the people, equipment, and technology assigned to the project.
*   **Human Resource Management:** Acquiring, developing, and managing the project team. This includes resolving conflicts, providing training, and ensuring motivation.
*   **Equipment & Tool Management:** Ensuring necessary hardware, software licenses, and development tools are available when needed.
*   **Example:** A senior developer is a critical resource. The manager must ensure they are not over-allocated to multiple tasks simultaneously, which could cause a bottleneck.

### 5. Risk Management
This is a proactive and continuous process of identifying, analyzing, and responding to project risk.
*   **Risk Identification:** Brainstorming potential problems (e.g., key personnel leaving, technology failing to meet performance needs).
*   **Risk Analysis:** Assessing the probability and impact of each risk.
*   **Risk Mitigation Planning:** Creating plans to avoid, minimize, or control the impact of high-priority risks.
*   **Example:** The risk of "database performance being too slow" is mitigated by planning for performance testing early in the cycle and having a database expert on call.

### 6. Configuration Management
This activity manages changing requirements and evolving software. It systematically controls changes to the configuration (code, documents, etc.) to maintain integrity.
*   **Version Control:** Using systems like Git to manage different versions of code and documents.
*   **Change Control:** Formal process for evaluating, approving, and implementing changes.
*   **Example:** When a client requests a new feature, it is not directly added to the code. Instead, it goes through a Change Control Board (CCB) which assesses its impact on budget and schedule before approval.

## Key Points / Summary

| Activity | Primary Focus | Key Outcome |
| :--- | :--- | :--- |
| **Project Planning** | Defining the "what, why, and how" | Project Plan (Scope, Budget, Schedule) |
| **Project Monitoring & Control** | Tracking progress and managing changes | Keeping the project on track (On Time, On Budget) |
| **Project Scheduling** | Creating a timeline and sequence | Visual Schedule (Gantt Chart, Network Diagram) |
| **Resource Management** | Efficient use of people and tools | Optimized team and resource allocation |
| **Risk Management** | Proactive problem identification | Risk Register and Mitigation Plans |
| **Configuration Management** | Managing change and versions | Controlled evolution of the software product |

**In essence, Software Project Management is a multi-faceted discipline that balances competing constraints—scope, time, cost, and quality—through these interconnected activities to guide a software project to successful completion.**
