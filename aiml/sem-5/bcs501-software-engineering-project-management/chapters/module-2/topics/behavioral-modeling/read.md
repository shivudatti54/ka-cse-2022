# Behavior Monitoring Tools

## Introduction

Behavior Monitoring Tools are a cornerstone of dynamic malware analysis. While static analysis examines code at rest, dynamic analysis observes malware in action within a controlled environment (a sandbox). These tools capture the runtime behavior of malicious software, providing a real-world view of its capabilities, such as file system modifications, network communications, registry changes, and process manipulation. This hands-on approach is crucial for understanding the true intent of malware that employs obfuscation, packing, or anti-analysis techniques designed to evade static detection.

## Core Concepts of Behavior Monitoring

### 1. The Principle of Execution Monitoring

The fundamental premise is simple: run the malware and watch what it does. However, doing this safely and capturing all relevant actions requires specialized tools. These tools typically operate by intercepting system calls (APIs) made by the malicious process. By hooking into these APIs, the monitoring tool can log, and sometimes block, every interaction the malware attempts with the operating system.

### 2. System Call Interception (API Hooking)

Most modern operating systems provide a structured way for applications to request services from the kernel, known as system calls or APIs. Malware must use these APIs to perform actions. Monitoring tools use techniques like:

- **In-line Hooking:** Modifying the first few instructions of an API function in memory to jump to the analyst's code, which logs the call and its parameters before executing the original function.
- **SSDT Hooking (Windows Specific):** Modifying the System Service Descriptor Table, which is a kernel structure that holds pointers to system call functions. This is a more advanced and powerful technique, often used by rootkits and thus a key area to monitor for malicious activity.

### 3. The Analysis Environment: Sandbox

Behavior monitoring is never performed on a live production system. It is conducted within an isolated environment, commonly referred to as a **sandbox**.

- **Virtual Machines (VMs):** Tools like VMware or VirtualBox are used to create disposable analysis environments. Snapshots are taken before execution and can be reverted to a clean state after analysis.
- **Purpose-Built Sandboxes:** Systems like Cuckoo Sandbox automate the entire process: they run the sample, monitor its behavior using multiple tools, and generate a comprehensive report.

## Key Behavior Monitoring Tools

### 1. Process Monitor (ProcMon)

**ProcMon** is an advanced monitoring tool for Windows that shows real-time file system, Registry, and process/thread activity. It is the first tool many analysts reach for.

- **Key Features:**
  - **Captures five classes of operation:** File System, Registry, Network, Process, and Profiling events.
  - **Powerful Filtering:** Indispensable for cutting through the noise of thousands of events generated by the OS itself. Analysts filter by process name, operation type (e.g., `RegSetValue`), or path.
  - **Process Tree:** Visually displays the relationship between processes, crucial for identifying child processes spawned by the malware.

- **Use Case:** Quickly identifying where a piece of malware drops its files (`WriteFile` operations to `C:\Users\AppData\Local\Temp\`) and what registry keys it modifies to achieve persistence (`RegSetValue` to `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`).

### 2. Process Explorer

**Process Explorer** is like a super-powered version of Windows Task Manager. It provides detailed information about which processes have opened which files, DLLs, and handles.

- **Key Features:**
  - **Process Hierarchy:** A color-coded tree view showing parent-child relationships.
  - **DLL View:** Reveals all libraries loaded by a process, helping to identify suspicious or injected DLLs.
  - **Handle View:** Shows all open handles (files, registry keys, mutexes, etc.) for a selected process. A malware sample often creates a mutex to ensure only one instance runs on a victim machine.
  - **String Search:** Allows you to search all running processes for a specific string of text (e.g., a command-and-control server URL).

- **Use Case:** Discovering a suspicious process that has an unknown company name listed in its properties or that has an unexpected TCP connection open.

### 3. API Monitor

**API Monitor** is a tool designed specifically to monitor API calls made by applications. It allows you to selectively monitor calls from specific DLLs (e.g., `kernel32.dll`, `advapi32.dll`, `ws2_32.dll`).

- **Key Features:**
  - **Selective Monitoring:** Choose which APIs and which modules to monitor, reducing noise.
  - **Parameter Decoding:** It doesn't just log that `CreateFileW` was called; it decodes and displays the parameters (filename, desired access, share mode, etc.) in a human-readable format.
  - **Call Stack:** Shows the chain of function calls that led to the API being called, which can be invaluable for understanding the context within the malware's code.

- **Use Case:** Tracing exactly how a ransomware sample encrypts files by monitoring the sequence of `FindFirstFileW` -> `FindNextFileW` -> `CreateFileW` -> `ReadFile` -> `CryptEncrypt` -> `WriteFile` API calls.

### 4. Wireshark

**Wireshark** is the world's foremost network protocol analyzer. It captures network traffic in real-time, allowing you to see exactly what data is being sent and received by the malware.

- **Key Features:**
  - **Deep Inspection:** Decodes hundreds of protocols, from common DNS/HTTP to more obscure ones.
  - **Powerful Filtering:** Use filters like `ip.src==<malware_ip>` or `http contains "command"` to isolate relevant traffic.
  - **Follow Stream:** Reconstructs the entire conversation between the malware and its C2 server.

- **Use Case:** Identifying the DNS query to a domain generation algorithm (DGA) domain, followed by an HTTP POST request that exfiltrates stolen data to a remote server.

### 5. Regshot

**Regshot** is a simple, open-source registry comparison tool. It takes a "before" and "after" snapshot of the Windows registry and highlights the differences.

- **Key Features:**
  - **Simplicity:** It's lightweight and focused on a single task.
  - **Clear Output:** Generates a list of added, deleted, and modified registry keys and values.

- **Use Case:** Quickly confirming that a piece of malware added a Run key for persistence (`HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Run\Malware`).

## Analysis Workflow

A typical behavior monitoring session follows this sequence:

```
+-------------------+     +---------------------+     +-------------------------+     +-----------------+
|   Preparation     | --> |   Execution and     | --> |   Data Analysis and    | --> |   Documentation |
|   (Snapshot VM,   |     |   Monitoring        |     |   Correlation          |     |   and Reporting |
|   Start Tools)    |     |   (Launch Malware)  |     |   (Filter logs,        |     |   (IOCs, TTPs)  |
+-------------------+     +---------------------+     |   find key events)     |     +-----------------+
                                                       +-------------------------+
```

## Exam Tips

1.  **Tool Selection:** Know which tool is best for which task. ProcMon is for broad system activity, API Monitor is for deep API tracing, and Wireshark is for network activity.
2.  **Noise Reduction:** The most critical skill is filtering out normal system activity to find the malicious events. Practice building effective filters in ProcMon and Wireshark.
3.  **Persistence Mechanisms:** Be able to recognize common registry and file system locations used for persistence (Run keys, service registration, scheduled tasks, etc.).
4.  **Indicators of Compromise (IOCs):** From the monitored behavior, you should be able to extract IOCs like file paths, created mutex names, C2 server IPs/domains, and registry keys.
5.  **Correlation:** A single event is rarely conclusive. Look for sequences of events (e.g., a file is written, then a registry key is modified to reference that file).
