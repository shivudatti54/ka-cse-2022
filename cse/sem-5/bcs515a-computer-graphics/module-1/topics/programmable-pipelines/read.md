# Programmable Pipelines

## Introduction

The programmable pipeline represents a fundamental shift in computer graphics architecture, transforming how graphics processing units (GPUs) handle rendering tasks. Unlike the fixed-function pipelines of earlier graphics hardware, programmable pipelines allow developers to write custom programs (called shaders) that execute at specific stages of the rendering process. This flexibility has revolutionized real-time graphics, enabling visual effects and rendering techniques that were previously impossible or extremely computationally expensive.

The transition from fixed-function to programmable pipelines began in the early 2000s with the introduction of vertex and fragment shaders, and has evolved to include geometry shaders, tessellation shaders, and compute shaders. Understanding programmable pipelines is essential for modern graphics programming, game development, and any application requiring real-time visual rendering. This topic forms the foundation for understanding modern GPU architecture and shader-based rendering techniques.

## Key Concepts

### Evolution of Graphics Pipelines

The graphics pipeline has undergone significant transformation since the early days of computer graphics. The fixed-function pipeline, which dominated the 1990s and early 2000s, provided a predetermined set of operations that transformed 3D geometry into 2D screen pixels. Each stage had specific, unchangeable functionality - vertex transformation, lighting calculations, rasterization, and fragment processing were all hardwired into the hardware.

The introduction of programmable pipelines began with the GeForce 3 (2001) and the Xbox console, which provided limited programmability through vertex and pixel shaders. Over time, the programmability expanded, allowing developers to replace or modify the behavior of pipeline stages with custom code. This shift gave graphics programmers unprecedented control over the rendering process, enabling effects like per-pixel lighting, normal mapping, shadow mapping, and post-processing effects.

### Architecture of the Programmable Graphics Pipeline

The modern programmable graphics pipeline consists of several stages, each serving a specific purpose in the rendering process:

**Input Assembler**: This stage collects geometric data (vertices and indices) from memory and assembles them into primitives (points, lines, triangles) for processing by subsequent stages.

**Vertex Shader**: The first programmable stage, the vertex shader processes each vertex individually. It performs operations such as coordinate transformations (from model space to world space to view space to screen space), vertex lighting calculations, and passing data to subsequent stages through varying variables.

**Hull Shader and Tessellator**: Optional stages used for tessellation, which subdivides primitives to add detail dynamically based on distance or other factors.

**Geometry Shader**: This stage can add, modify, or delete primitives. It processes entire primitives (not just individual vertices) and can generate new vertices to create effects like billboard particles, shadow volumes, or detailed wireframe representations.

**Rasterizer**: The rasterizer converts primitives into fragments (potential pixels). It performs interpolation of vertex attributes across the primitive and handles clipping and scissor testing.

**Fragment Shader**: Also known as the pixel shader, this stage processes each fragment generated by the rasterizer. It performs per-fragment operations such as texture sampling, per-pixel lighting, fog calculations, and transparency handling. This is where most visual effects are implemented.

**Output Merger**: The final stage combines fragment data with the existing framebuffer contents, handling depth testing, stencil testing, blending, and write masks.

### Shader Programming Fundamentals

Shaders are written in specialized high-level languages such as GLSL (OpenGL Shading Language), HLSL (High-Level Shading Language for DirectX), orCg/CgFX. These languages provide constructs similar to C programming but include built-in data types and functions specifically designed for graphics operations.

**Data Types**: Shader languages include scalar types (float, int, bool), vector types (vec2, vec3, vec4 in GLSL; float2, float3, float4 in HLSL), matrix types (mat3, mat4), and sampler types for texture access.

**Built-in Variables and Functions**: Shaders have access to built-in variables (like gl_Position in GLSL) and functions for common operations (normalize, dot, cross, reflect, refract, texture sampling, etc.).

**Uniform Variables**: These are global variables that remain constant across all invocations of a shader during a draw call. They are used for passing data that doesn't change per-vertex or per-fragment, such as transformation matrices, light properties, and texture samplers.

### Vertex Shader Operations

The vertex shader is the first programmable stage in the pipeline and operates on each vertex independently. Its primary responsibilities include:

**Coordinate Transformation**: Converting vertex positions from object space through world space, view space, and projection space to normalized device coordinates and finally to screen coordinates. This transformation pipeline is fundamental to positioning objects correctly in the 3D scene.

**Normal Transformation**: Transforming normal vectors using the inverse-transpose of the world matrix to ensure correct lighting calculations regardless of non-uniform scaling.

**Lighting Calculations**: While simple lighting can be computed in the vertex shader (resulting in Gouraud shading), more sophisticated per-pixel lighting is typically deferred to the fragment shader.

**Data Output**: Passing interpolated data to the fragment shader through varyings or outputs, including texture coordinates, normals, view direction vectors, and any custom data needed for fragment processing.

### Fragment Shader Operations

The fragment shader determines the final color of each pixel and is where most visual complexity is implemented:

**Texture Sampling**: Fetching color values from textures using interpolated texture coordinates. Multiple texture lookups can be combined using various blending modes to create complex surface materials.

**Per-Pixel Lighting**: Computing lighting at each fragment rather than at each vertex results in higher quality lighting (Phong shading) with smooth highlights and proper specular response.

**Material Properties**: Implementing various material types through combinations of ambient, diffuse, specular, and emissive components, potentially modulated by texture values.

**Fog and Atmospheric Effects**: Applying distance-based fog to blend objects with the background, creating depth perception and atmospheric perspective.

**Alpha Testing and Transparency**: Handling transparent or semi-transparent surfaces through alpha blending or alpha testing operations.

## Examples

### Example 1: Basic Vertex Transformation

Consider a simple vertex shader that performs coordinate transformation from model space to screen space:

```glsl
// GLSL Vertex Shader
#version 330 core

layout(location = 0) in vec3 aPosition;
layout(location = 1) in vec3 aNormal;
layout(location = 2) in vec2 aTexCoord;

uniform mat4 modelMatrix;
uniform mat4 viewMatrix;
uniform mat4 projectionMatrix;
uniform mat3 normalMatrix;

out vec3 vNormal;
out vec3 vFragPos;
out vec2 vTexCoord;

void main() {
 // Calculate world position
 vec4 worldPosition = modelMatrix * vec4(aPosition, 1.0);
 vFragPos = worldPosition.xyz;

 // Transform normal to world space
 vNormal = normalMatrix * aNormal;

 // Pass texture coordinates
 vTexCoord = aTexCoord;

 // Final position in clip space
 gl_Position = projectionMatrix * viewMatrix * worldPosition;
}
```

**Step-by-step explanation**:

1. Input attributes are declared with layout qualifiers specifying attribute locations
2. Uniform matrices are received from the application (model, view, projection)
3. The vertex position is transformed through the complete pipeline: model → world → view → projection → clip space
4. Normals are transformed using the normal matrix (inverse-transpose of model matrix) to handle non-uniform scaling correctly
5. Data is passed to the fragment shader through output variables

### Example 2: Simple Phong Lighting Fragment Shader

```glsl
// GLSL Fragment Shader
#version 330 core

in vec3 vNormal;
in vec3 vFragPos;
in vec2 vTexCoord;

out vec4 fragColor;

uniform vec3 lightPosition;
uniform vec3 lightColor;
uniform vec3 objectColor;
uniform vec3 viewPosition;

void main() {
 // Normalize interpolated normal
 vec3 normal = normalize(vNormal);

 // Ambient lighting
 float ambientStrength = 0.1;
 vec3 ambient = ambientStrength * lightColor;

 // Diffuse lighting
 vec3 lightDir = normalize(lightPosition - vFragPos);
 float diff = max(dot(normal, lightDir), 0.0);
 vec3 diffuse = diff * lightColor;

 // Specular lighting
 float specularStrength = 0.5;
 vec3 viewDir = normalize(viewPosition - vFragPos);
 vec3 reflectDir = reflect(-lightDir, normal);
 float spec = pow(max(dot(viewDir, reflectDir), 0.0), 32);
 vec3 specular = specularStrength * spec * lightColor;

 // Combine all lighting components
 vec3 result = (ambient + diffuse + specular) * objectColor;
 fragColor = vec4(result, 1.0);
}
```

**Step-by-step explanation**:

1. Interpolated inputs are received from the vertex shader
2. The surface normal is normalized (interpolation can cause non-unit length)
3. Ambient component provides base illumination
4. Diffuse component is calculated using Lambert's cosine law (dot product of normal and light direction)
5. Specular component uses the Blinn-Phong model with halfway vector reflection
6. All components are combined and multiplied by the object color to produce the final fragment color

### Example 3: Texture Mapping with Multi-texturing

```glsl
// Fragment shader with texture sampling
#version 330 core

in vec2 vTexCoord;
in vec3 vNormal;

out vec4 fragColor;

uniform sampler2D diffuseTexture;
uniform sampler2D normalTexture;
uniform sampler2D specularTexture;

void main() {
 // Sample diffuse color from texture
 vec4 diffuseColor = texture(diffuseTexture, vTexCoord);

 // Sample specular map (grayscale determines shininess)
 float specularStrength = texture(specularTexture, vTexCoord).r;

 // If alpha is zero, discard the fragment (transparent surface)
 if(diffuseColor.a < 0.1)
 discard;

 fragColor = diffuseColor;
 fragColor.rgb *= (1.0 + specularStrength * 0.5);
}
```

**Step-by-step explanation**:

1. Multiple texture samplers are declared as uniform variables
2. The diffuse texture provides the base color/modulates the surface color
3. The specular texture controls the specular intensity at each point
4. Alpha testing discards fragments for transparency handling
5. Final color is modulated based on specular map values

## Exam Tips

1. **Understand Pipeline Order**: Remember the exact sequence of programmable stages: Input Assembler → Vertex Shader → Hull Shader → Tessellator → Geometry Shader → Rasterizer → Fragment Shader → Output Merger. This is frequently tested in exams.

2. **Vertex vs Fragment Shader Distinction**: The vertex shader operates on vertices and outputs data that gets interpolated across primitives, while the fragment shader operates on fragments (potential pixels) and determines final pixel colors. Know when to perform operations in each.

3. **Coordinate Spaces**: Be thoroughly familiar with the transformation pipeline: Local/Model Space → World Space → View/Camera Space → Clip Space → NDC → Screen Space. Understand what each transformation matrix does.

4. **Shader Language Syntax**: Know basic GLSL syntax including data types (vec3, mat4), qualifiers (uniform, in, out), and built-in functions (normalize, dot, reflect). DirectX HLSL differences may also be tested.

5. **Fixed vs Programmable Pipeline**: Be able to explain the advantages of programmable pipelines over fixed-function pipelines, including flexibility, visual quality improvements, and computational efficiency for complex effects.

6. **Uniform Variables**: Understand that uniforms remain constant across all vertices/fragments during a draw call and are used for transformation matrices, light properties, and texture samplers.

7. **Tessellation and Geometry Shaders**: These are optional stages in modern pipelines. Understand their purposes: tessellation adds geometry detail dynamically, geometry shaders can create or destroy primitives.

8. **Practical Applications**: Be prepared to explain how programmable pipelines enable specific effects like shadow mapping, bump mapping, and post-processing, as these are commonly asked application-based questions.
