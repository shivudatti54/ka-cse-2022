**Requirement Analysis** bridges problem definition and solution design in the SDLC. It involves eliciting stakeholder needs, analyzing for conflicts, specifying in formal documents (SRS), and validating completeness.

**Requirements Engineering** has four activities: Elicitation (gathering through interviews, observation, prototyping), Analysis (refining, detecting conflicts), Specification (IEEE 830 SRS document), and Validation (ensuring accuracy).

**Functional Requirements** specify what the system does (behaviors, inputs, outputs). **Non-Functional Requirements** specify how the system performs (performance, security, reliability, usability, maintainability, portability).

**Elicitation Techniques**: Interviews, questionnaires, observation, document analysis, prototyping, JAD workshops.

**Analysis Techniques**: Use case modeling (actors, use cases, include/extend relationships), prioritization methods (MoSCoW, AHP, Hundred-Dollar), traceability matrices (forward/backward), conflict resolution (negotiation, trade-off analysis).

**SRS per IEEE 830**: Introduction, Overall Description, Functional Requirements, Non-Functional Requirements, Appendices.

Key insight: 40-60% of software project failures trace to poor requirements engineering. Thorough analysis prevents costly downstream errors.
