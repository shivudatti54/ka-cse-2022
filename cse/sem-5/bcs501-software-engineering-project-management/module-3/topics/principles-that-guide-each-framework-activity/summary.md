# Principles That Guide Each Framework Activity - Summary

## Key Definitions

- **Framework Activities**: The five fundamental software engineering processes: Communication, Planning, Modeling, Construction, and Deployment.
- **Agile Principles**: The twelve principles underlying the Agile Manifesto that guide software development practices.
- **Time-Boxing**: Constraining activities to fixed time periods to create focus and urgency.
- **Just-in-Time (JIT) Modeling**: Creating models only when needed and discarding them after use.
- **Test-Driven Development (TDD)**: A construction practice where tests are written before code following the Red-Green-Refactor cycle.
- **Continuous Integration (CI)**: Integrating code frequently with automated builds and tests.
- **Continuous Delivery (CD)**: Automatically preparing code changes for production release.
- **Work In Progress (WIP) Limits**: Constraints on the amount of work in progress in a Kanban system.

## Important Formulas

- **Velocity Calculation**: Velocity = Sum of completed story points across sprints (used for planning)
- **WSJF Priority**: WSJF = (Business Value + Time Criticality + Risk Reduction) / Job Size (used for backlog prioritization)

## Key Points

1. Agile transforms all five framework activities into iterative, incremental processes rather than linear ones.

2. Communication principles emphasize face-to-face interaction, embedded communication, and continuous stakeholder engagement.

3. Planning in agile uses time-boxing, iterative planning at multiple levels, and velocity-based forecasting.

4. Modeling shifts from comprehensive upfront modeling to just-in-time, collaborative modeling practices.

5. Construction principles prioritize quality through TDD, continuous integration, pair programming, and refactoring.

6. Deployment principles focus on continuous delivery/deployment to maximize feedback and minimize time-to-market.

7. Each agile framework (Scrum, XP, Kanban) applies these principles with different emphases and practices.

8. The Agile Manifesto's value of "Working software over comprehensive documentation" strongly influences all activities.

9. Empirical process control (transparency, inspection, adaptation) underlies agile planning and execution.

10. Feature toggles and incremental delivery reduce risk while enabling frequent releases.

## Common Mistakes

1. Confusing continuous delivery with continuous deployment—only the latter automatically deploys to production.

2. Assuming agile means "no planning"—agile planning is iterative and adaptive, not absent.

3. Treating TDD as merely "writing tests after code"—TDD requires writing failing tests first.

4. Misunderstanding time-boxing—it's not about rushing but about creating sustainable pace and focus.

5. Applying waterfall principles (like extensive upfront documentation) within an agile framework, contradicting agile values.

6. Ignoring WIP limits in Kanban—they are essential for flow optimization, not optional.

7. Treating the Product Owner as optional—in Scrum, the Product Owner is essential for backlog prioritization and communication.
