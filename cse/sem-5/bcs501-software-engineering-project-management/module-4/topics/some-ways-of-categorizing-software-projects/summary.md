# Categorization of Software Projects - Summary

## Key Definitions and Concepts

- **Project Categorization:** The process of classifying software projects based on specific criteria to aid in planning, resource allocation, and methodology selection
- **Small Projects:** Less than 6 months, 1-5 team members
- **Medium Projects:** 6-12 months, 5-15 team members
- **Large Projects:** 1-3 years, 15-50 team members
- **Very Large Projects:** 3+ years, 50+ team members
- **Real-Time Software:** Must respond within strict time constraints
- **Embedded Software:** Software integrated within hardware devices

## Important Formulas and Theorems

No specific formulas apply to this topic. The categorization is qualitative and based on project characteristics.

## Key Points

- Software project categorization directly influences methodology, documentation, and quality assurance approaches
- Projects can be simultaneously classified under multiple categories (size, domain, methodology, complexity)
- Real-time and embedded software represent distinct domains with specific technical requirements
- Agile methodology suits projects with evolving requirements; Waterfall suits stable requirements
- Technical complexity assessment considers novelty of technology, algorithm complexity, and integration challenges
- Team structure (in-house, outsourced, offshore) affects communication, control, and project outcomes
- Large projects require formal processes, extensive documentation, and rigorous oversight

## Common Mistakes to Avoid

1. **Confusing real-time with embedded:** Real-time refers to timing constraints; embedded refers to software within hardware. They can overlap but are distinct concepts.

2. **Assuming single categorization:** A project can belong to multiple categories simultaneously. Don't force a project into just one category.

3. **Ignoring methodology fit:** Choosing a methodology without considering project category leads to project failure. Size and requirements stability determine methodology suitability.

4. **Overlooking team structure impact:** Failing to consider how team structure (especially in offshore development) affects communication and coordination.

## Revision Tips

1. **Create a comparison table:** List all categorization types with their sub-categories and characteristics for quick revision.

2. **Practice scenario classification:** Given a project description, identify all applicable categories. This is commonly asked in exams.

3. **Remember real-world examples:** Associate each category with known software (e.g., Microsoft Office as application software, Windows as system software).

4. **Focus on relationships:** Understand how size relates to methodology choice, how domain affects technical complexity, and how team structure influences communication overhead.
