# Traditional versus Modern Project Management Practices - Summary

## Key Definitions and Concepts

- **Traditional Project Management**: A linear, sequential approach following distinct phases (Requirements → Design → Implementation → Testing → Deployment → Maintenance). Also known as Waterfall model.

- **Modern Project Management**: An adaptive, iterative approach that embraces change and continuous feedback. Primarily represented by Agile methodologies.

- **Agile**: A project management philosophy emphasizing flexibility, collaboration, and iterative development, guided by the Agile Manifesto (2001).

- **Scrum**: An Agile framework using time-boxed sprints, with roles including Product Owner, Scrum Master, and Development Team.

- **Kanban**: A visual Agile method focusing on continuous delivery and workflow optimization without fixed time boxes.

## Important Formulas and Principles

- **Agile Manifesto Principles**: 12 principles emphasizing customer satisfaction, welcoming changing requirements, frequent delivery, business-developer collaboration, and sustainable development.

- **Scrum Events**: Sprint Planning, Daily Scrum, Sprint Review, Sprint Retrospective.

- **Scrum Artifacts**: Product Backlog, Sprint Backlog, Increment (Done).

## Key Points

- Traditional approach follows strict sequential phases; modern approach uses iterative cycles.

- Traditional requires comprehensive upfront documentation; Agile focuses on working software over documentation.

- Change control in traditional is rigid and expensive; Agile embraces change as competitive advantage.

- Traditional suits projects with fixed requirements; Agile suits projects with uncertain/dynamic requirements.

- Scrum uses fixed-length sprints (2-4 weeks); Kanban uses continuous flow without time boxes.

- Customer involvement in traditional is periodic; in Agile it is continuous throughout the project.

- Risk management in traditional is upfront; in Agile it is continuous with small batch sizes limiting impact.

## Common Mistakes to Avoid

- Assuming Agile has no planning—Agile has different planning approaches (Sprint Planning, Product Backlog refinement).

- Thinking documentation is completely eliminated in Agile—Agile values working software more, not documentation less.

- Applying Agile to all projects—Traditional still works well for regulated industries and fixed-scope projects.

- Ignoring hybrid approaches—Many organizations successfully combine both methodologies.

## Revision Tips

1. Create a comparison table with columns for Approach, Documentation, Flexibility, Customer Involvement, and Risk Handling.

2. Memorize the four values of Agile Manifesto as this is frequently tested.

3. Understand the Scrum roles and artifacts thoroughly—these are commonly asked in exams.

4. Practice identifying which methodology suits different project scenarios.

5. Remember that both approaches have their place—the key is selecting the right one for the context.
