# Introduction to Signals in Unix System Programming

## Introduction

Signals are a fundamental mechanism in Unix systems that provide a way for the kernel and processes to communicate asynchronous events. A signal is a software interrupt delivered to a process by the operating system, notifying it that some event has occurred. These events can range from user-generated actions like pressing Ctrl+C to exceptional conditions like invalid memory access or timer expiration. Understanding signals is crucial for Unix system programmers as they form the backbone of process management, inter-process communication, and handling asynchronous behavior in applications.

The Unix signal mechanism dates back to early versions of Unix and has evolved significantly over time. Originally, signals were unreliable and inconsistent, but POSIX standardization has provided a robust and portable signal handling interface. Modern Unix systems support both traditional signal semantics (signal()) and the more reliable POSIX sigaction() interface. Signals are identified by small positive integers, and each signal has a default action associated with it, which typically involves terminating the process, stopping the process, or ignoring the signal.

In Unix System Programming, signals play a vital role in writing responsive and robust applications. They allow processes to respond to external events without continuously polling for changes, making programs more efficient. For instance, a long-running computation can be interrupted gracefully, a daemon process can receive configuration reload requests, and network applications can handle connection timeouts. The signal mechanism provides the foundation for building reactive systems that can handle multiple asynchronous events concurrently.

## Key Concepts

### What is a Signal?

A signal is a notification sent to a process (or thread) to indicate that an event has occurred. When a signal is generated, the kernel interrupts the normal flow of execution and delivers the signal to the target process. The process can respond to the signal in several ways: it can execute a user-defined handler function, ignore the signal, or let the default action take place. Signals are asynchronous, meaning they can occur at any point during program execution, which requires careful programming to avoid race conditions and data corruption.

### Signal Generation

Signals can be generated by various sources in a Unix system. Hardware-generated signals occur when the CPU detects exceptional conditions such as illegal memory access (SIGSEGV), floating-point exceptions (SIGFPE), or illegal instructions (SIGILL). Software-generated signals are processes using produced by other the kill() system call or by the process itself using raise(). Terminal-generated signals include those caused by pressing special key combinations like Ctrl+C (SIGINT), Ctrl+Z (SIGTSTP), and Ctrl+\ (SIGQUIT). Additionally, the kernel generates signals for job control operations and timer expiration.

### Signal Disposition

The signal disposition determines how a process will handle a particular signal. Each signal in a process has an associated disposition, which is the action taken when that signal is delivered. The disposition can be set to a custom signal handler function, SIG_IGN (ignore the signal), or SIG_DFL (use the default action). When a signal handler is installed, it remains installed until explicitly changed, unlike the older signal() implementation where handlers could be reset after being invoked.

### Signal Actions

The POSIX standard defines several standard actions for signal handling. The default action for most signals is to terminate the process, sometimes with a core dump for debugging purposes. Some signals, like SIGSTOP and SIGTSTP, stop the process execution temporarily. Other signals, such as SIGCONT, resume a stopped process. Certain signals like SIGCHLD and SIGURG are typically ignored by default. The sigaction() function provides fine-grained control over signal handling, allowing programmers to specify handler functions, signal masks, and flags that modify the signal delivery behavior.

### Signal Types

Unix systems define numerous signal types, each associated with a specific event. Common signals include SIGALRM (alarm clock), SIGCHLD (child process terminated), SIGINT (terminal interrupt), SIGKILL (kill - cannot be caught or ignored), SIGPIPE (write to pipe with no readers), SIGQUIT (terminal quit), SIGSEGV (invalid memory reference), SIGTERM (termination request), and SIGUSR1/SIGUSR2 (user-defined signals). The SIGKILL and SIGSTOP signals are special as they cannot be blocked, ignored, or handled by user-defined functions.

## Examples

### Example 1: Basic Signal Handling

```c
#include <stdio.h>
#include <stdlib.h>
#include <signal.h>

void sigint_handler(int signum) {
 printf("\nReceived SIGINT (Ctrl+C). Performing graceful shutdown...\n");
 /* Perform cleanup operations here */
 exit(0);
}

int main() {
 /* Install signal handler for SIGINT */
 struct sigaction sa;
 sa.sa_handler = sigint_handler;
 sigemptyset(&sa.sa_mask);
 sa.sa_flags = 0;

 if (sigaction(SIGINT, &sa, NULL) == -1) {
 perror("sigaction");
 exit(1);
 }

 printf("Press Ctrl+C to terminate...\n");

 /* Infinite loop waiting for signals */
 while (1) {
 printf("Working...\n");
 sleep(1);
 }

 return 0;
}
```

This example demonstrates how to handle the SIGINT signal (generated by Ctrl+C) using the POSIX sigaction() function. The handler performs cleanup operations before terminating the program, allowing for graceful shutdown rather than abrupt termination.

### Example 2: Ignoring Signals

```c
#include <stdio.h>
#include <signal.h>
#include <unistd.h>

int main() {
 /* Ignore SIGINT (Ctrl+C) */
 struct sigaction sa;
 sa.sa_handler = SIG_IGN;
 sigemptyset(&sa.sa_mask);
 sa.sa_flags = 0;

 if (sigaction(SIGINT, &sa, NULL) == -1) {
 perror("sigaction");
 return 1;
 }

 printf("SIGINT is now ignored. Press Ctrl+C to test...\n");
 printf("This program will run for 10 seconds.\n");

 for (int i = 10; i > 0; i--) {
 printf("Remaining: %d seconds\n", i);
 sleep(1);
 }

 printf("Program completed.\n");
 return 0;
}
```

This example shows how to ignore the SIGINT signal, making the program immune to Ctrl+C. This is useful for critical operations that should not be interrupted by standard termination signals.

### Example 3: Sending Signals Between Processes

```c
#include <stdio.h>
#include <stdlib.h>
#include <signal.h>
#include <unistd.h>
#include <sys/types.h>

void alarm_handler(int signum) {
 printf("Timeout! Received SIGALRM\n");
}

int main() {
 pid_t pid = fork();

 if (pid == -1) {
 perror("fork");
 return 1;
 }

 if (pid == 0) {
 /* Child process - sleep for 3 seconds then send signal to parent */
 sleep(3);
 printf("Child: Sending SIGUSR1 to parent (PID: %d)\n", getppid());
 kill(getppid(), SIGUSR1);
 exit(0);
 } else {
 /* Parent process - set up signal handler */
 struct sigaction sa;
 sa.sa_handler = alarm_handler;
 sigemptyset(&sa.sa_mask);
 sa.sa_flags = 0;

 sigaction(SIGUSR1, &sa, NULL);

 printf("Parent: Waiting for signal from child...\n");
 pause(); /* Wait for signal */
 printf("Parent: Signal received, continuing...\n");
 }

 return 0;
}
```

This example demonstrates inter-process communication using signals. The parent process installs a handler for SIGUSR1 and waits using pause(). The child process sleeps for 3 seconds and then sends SIGUSR1 to the parent using the kill() system call.

## Exam Tips

1. **Understand the difference between signal() and sigaction()**: The sigaction() function is preferred in modern POSIX systems because it provides more reliable and portable signal handling semantics compared to the older signal() function.

2. **Remember that signal handlers must be async-signal-safe**: Functions like printf(), malloc(), and other standard library functions may not be safe to call from within a signal handler. Only use async-signal-safe functions (listed in POSIX standards) in signal handlers.

3. **Know the default actions of common signals**: SIGKILL and SIGSTOP cannot be caught, blocked, or ignored. This is important for exam questions about signal handling limitations.

4. **Understand signal masking**: Using sigprocmask(), processes can block the delivery of certain signals. Blocked signals are queued (in modern systems) and delivered when unblocked.

5. **Remember the relationship between fork() and signals**: Child processes inherit the signal dispositions from their parent. The SIGCHLD signal is sent to the parent when a child terminates.

6. **Practice with sigsuspend()**: This function atomically replaces the signal mask and suspends the process, making it crucial for avoiding race conditions in signal handling code.

7. **Know when to use pause()**: The pause() function suspends execution until a signal is delivered. It is commonly used with signal handlers to wait for asynchronous events.

8. **Understand signal set operations**: Functions like sigemptyset(), sigfillset(), sigaddset(), and sigdelset() are used to manipulate signal sets, which are essential for specifying which signals to block or handle.
