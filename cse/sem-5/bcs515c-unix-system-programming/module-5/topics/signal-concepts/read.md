# Signal Concepts in Unix System Programming

## Introduction

Signals are a fundamental mechanism of inter-process communication in Unix systems, providing a way for the kernel and processes to notify each other of asynchronous events. A signal is a software interrupt that delivers a notification to a process, informing it that a particular event has occurred. The term "asynchronous" is crucial here because signals can arrive at any time during a process's execution, interrupting its normal control flow.

The Unix signal mechanism evolved from early Unix systems and has become one of the oldest and most widely used forms of process communication. Signals can be generated by various sources: the hardware (such as divide-by-zero errors or illegal memory accesses), the kernel (when a child process terminates or when a timer expires), or other processes (using the kill system call). Understanding signals is essential for writing robust Unix applications, particularly for implementing process synchronization, handling exceptional conditions, and building responsive daemon processes.

The signal concept in Unix follows a consistent model across different implementations, though some historical variations exist. POSIX (Portable Operating System Interface for Unix) standardized the signal API, ensuring portability across different Unix variants. Modern Unix systems support both the traditional signal() interface and the more reliable sigaction() interface, with the latter being recommended for new program development due to its deterministic behavior.

## Key Concepts

### Signal Definition and Characteristics

A signal is a bounded message consisting of a signal number (a small positive integer) that identifies the type of signal. Unix systems define approximately 30-60 different signal types, though the exact number varies by implementation. Each signal is associated with a default action that the kernel performs when the signal is delivered to a process. The four possible default actions are: termination of the process (SIGTERM, SIGKILL), termination with core dump (SIGQUIT), stopping the process (SIGSTOP, SIGTSTP), and ignoring the signal (SIGIGN).

Signals are classified into two broad categories: hardware-generated signals (also called synchronous signals) and software-generated signals (asynchronous signals). Synchronous signals are generated as a direct result of executing a particular instruction, such as accessing illegal memory (SIGSEGV), dividing by zero (SIGFPE), or pressing certain key combinations (SIGINT). Asynchronous signals are generated by external events or explicit system calls and can arrive at any point during process execution.

### Signal Generation and Delivery

When a signal is generated, the kernel marks the signal as "pending" for the target process. The actual delivery of the signal to the process occurs when the process next executes a system call or is scheduled to run in user mode. This delivery mechanism means that signals cannot interrupt critical kernel operations and ensures that signal handlers execute at safe points in the program's execution.

The delivery of a signal involves the kernel temporarily interrupting the normal flow of execution, saving the current process context, and invoking the signal handler function. Upon return from the signal handler, the process resumes execution from the point where it was interrupted. If the process has specified SIG_DFL (default action) for a signal, the kernel performs the default action associated with that signal.

### Signal Handling

Each process maintains a signal disposition that defines how the process responds to each type of signal. The disposition can be set to one of three values: a user-defined signal handler function, SIG_IGN (ignore the signal), or SIG_DFL (perform the default action). When a signal handler is installed, it remains installed until explicitly changed, unlike some historical Unix implementations where the disposition was reset to SIG_DFL after each signal delivery.

The prototype for a signal handler follows a specific signature: `void handler(int signo)`. The handler receives the signal number as its sole argument, allowing a single handler function to handle multiple signal types. Signal handlers must be careful about what operations they perform, as they execute in an asynchronous context. Functions that are not async-signal-safe should be avoided within signal handlers.

### Signal Sets and Operations

The POSIX standard provides the sigset_t data type to represent sets of signals, along with functions to manipulate these sets. The sigemptyset() function initializes an empty signal set, sigfillset() creates a set containing all signals, sigaddset() adds a signal to a set, and sigdelset() removes a signal from a set. The sigismember() function tests whether a particular signal is a member of a set.

These signal set operations are fundamental to more advanced signal management functions like sigprocmask() (for blocking signals), sigpending() (for checking pending signals), and sigaction() (for installing signal handlers). Understanding signal sets is crucial for writing programs that need to handle multiple signals with different dispositions.

### Signal Mask and Blocking

Each process has a signal mask that defines which signals are currently blocked from delivery. When a signal is blocked, it remains pending until the signal is unblocked, at which point it is delivered. The sigprocmask() function allows processes to examine and change their signal mask, enabling precise control over when signals are delivered. Blocking signals is essential for protecting critical sections of code from interruption and for implementing synchronized access to shared resources.

The relationship between pending signals and blocked signals is important: when a blocked signal is generated, it remains pending until either the signal is unblocked or the process explicitly handles it. Multiple occurrences of the same signal while it is blocked may result in only one delivery when unblocked, depending on the implementation and signal type.

## Examples

### Example 1: Basic Signal Handling

```c
#include <stdio.h>
#include <signal.h>
#include <unistd.h>

void sigint_handler(int signo) {
 printf("\nReceived SIGINT (Signal %d)\n", signo);
 printf("Cleaning up...\n");
}

int main() {
 /* Install signal handler for SIGINT */
 struct sigaction sa;
 sa.sa_handler = sigint_handler;
 sigemptyset(&sa.sa_mask);
 sa.sa_flags = 0; /* No special flags */

 if (sigaction(SIGINT, &sa, NULL) == -1) {
 perror("sigaction");
 return 1;
 }

 printf("Process ID: %d\n", getpid());
 printf("Press Ctrl+C to send SIGINT\n");
 printf("Press Ctrl+\\ to send SIGQUIT\n");

 /* Infinite loop waiting for signals */
 while (1) {
 pause(); /* Wait for signals */
 }

 return 0;
}
```

This example demonstrates the proper use of sigaction() to install a signal handler. The program installs a handler for SIGINT (generated by pressing Ctrl+C) and enters an infinite loop using pause(), which blocks until a signal is delivered. The handler prints a message and returns, allowing the program to continue.

### Example 2: Signal Blocking for Critical Sections

```c
#include <stdio.h>
#include <signal.h>
#include <unistd.h>

volatile sig_atomic_t counter = 0;
sigset_t block_mask, old_mask;

void handler(int signo) {
 printf("Signal %d received, counter = %d\n", signo, counter);
}

int main() {
 struct sigaction sa;
 sa.sa_handler = handler;
 sigemptyset(&sa.sa_mask);
 sa.sa_flags = 0;

 sigaction(SIGUSR1, &sa, NULL);

 /* Initialize signal set and add SIGUSR1 */
 sigemptyset(&block_mask);
 sigaddset(&block_mask, SIGUSR1);

 /* Block SIGUSR1 */
 if (sigprocmask(SIG_BLOCK, &block_mask, &old_mask) == -1) {
 perror("sigprocmask");
 return 1;
 }

 printf("SIGUSR1 blocked. Sending signal...\n");
 kill(getpid(), SIGUSR1); /* Signal remains pending */

 /* Critical section - signal is blocked */
 counter = 100;
 printf("In critical section, counter = %d\n", counter);
 sleep(2);

 /* Unblock SIGUSR1 - now handler executes */
 printf("Unblocking SIGUSR1...\n");
 if (sigprocmask(SIG_SETMASK, &old_mask, NULL) == -1) {
 perror("sigprocmask");
 return 1;
 }

 return 0;
}
```

This example illustrates signal blocking to protect critical sections. The program blocks SIGUSR1 before entering a critical section where it modifies a shared variable. Even though the signal is sent using kill(), it remains pending until the signal is unblocked after the critical section completes.

### Example 3: Handling Multiple Signals with sigaction

```c
#include <stdio.h>
#include <signal.h>
#include <unistd.h>

void generic_handler(int signo) {
 const char *name;

 switch (signo) {
 case SIGTERM: name = "SIGTERM"; break;
 case SIGUSR1: name = "SIGUSR1"; break;
 case SIGUSR2: name = "SIGUSR2"; break;
 default: name = "UNKNOWN"; break;
 }

 printf("Caught signal %d (%s)\n", signo, name);
}

int main() {
 struct sigaction sa;

 sa.sa_handler = generic_handler;
 sigemptyset(&sa.sa_mask);

 /* Add additional signals to block during handler */
 sigaddset(&sa.sa_mask, SIGUSR1);
 sigaddset(&sa.sa_mask, SIGUSR2);

 sa.sa_flags = 0;

 /* Install handlers for multiple signals */
 sigaction(SIGTERM, &sa, NULL);
 sigaction(SIGUSR1, &sa, NULL);
 sigaction(SIGUSR2, &sa, NULL);

 printf("PID: %d\n", getpid());
 printf("Send signals using: kill -TERM %d, kill -USR1 %d, kill -USR2 %d\n",
 getpid(), getpid(), getpid());

 while (1) {
 pause();
 }

 return 0;
}
```

This example demonstrates handling multiple signals with a single handler function. The sa_mask member is used to specify additional signals to block while the handler executes, preventing the handler from being interrupted by other signals. This is important for maintaining handler integrity and preventing nested signal handling.

## Exam Tips

1. **Understand signal delivery timing**: Signals are delivered when the process returns from kernel mode to user mode, not immediately when generated. Remember that signals cannot interrupt system calls or kernel operations.

2. **Know the difference between synchronous and asynchronous signals**: Synchronous signals (like SIGSEGV, SIGFPE) are generated as a direct result of CPU execution, while asynchronous signals (like SIGALRM, SIGUSR1) come from external sources.

3. **Remember the four default signal actions**: Termination (with and without core dump), stopping a process, ignoring the signal, and continuing a stopped process.

4. **Use sigaction() over signal()**: The sigaction() function provides more reliable and portable signal handling. It allows specification of flags and maintains consistent signal disposition.

5. **Async-signal-safe functions**: Only use async-signal-safe functions (like write(), \_exit(), signal()) within signal handlers. Functions like printf(), malloc(), and strcpy() are not safe.

6. **Signal masking prevents delivery**: A blocked signal remains pending until unblocked. Use sigpending() to check which blocked signals are pending.

7. **volatile sig_atomic_t**: Use this type for variables modified by signal handlers and read in the main program to ensure atomic access.

8. **Remember sigsuspend()**: When you need to atomically unblock signals and wait for them, use sigsuspend() rather than separate sigprocmask() and pause() calls to avoid race conditions.
