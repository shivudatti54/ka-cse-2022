# Extracting Interaction Design Requirements - Summary

## Key Definitions and Concepts

- **Interaction Design Requirements**: Specifications defining how users interact with a system, encompassing functional, non-functional, and interaction requirements viewed through user goals and tasks

- **User Personas**: Fictional representations of target users created from research data, including demographics, goals, motivations, frustrations, and behavioral patterns

- **User Scenarios**: Narrative descriptions of specific situations where users interact with a product to achieve goals, including context, user actions, and system responses

- **User Stories**: Format "As a [user], I want [goal] so that [benefit]" to capture user needs and value delivered

- **Task Analysis**: Systematic examination breaking down how users accomplish goals into steps, decisions, and information needs

- **Use Cases**: Descriptions of system interactions from the user's perspective, specifying actors, triggers, flows, and postconditions

- **User Journey Maps**: Visual representations of the sequence of user interactions with a product over time, capturing emotions and pain points

## Important Formulas and Techniques

- **MoSCoW Prioritization**: Must have, Should have, Could have, Won't have (this time) - for prioritizing requirements

- **User Story Format**: "As a [type of user], I want [goal] so that [benefit]"

- **Research Method Selection**: Qualitative methods (interviews, observation) for depth; Quantitative methods (surveys) for breadth and patterns

## Key Points

1. Requirements extraction is the foundation of user-centered design, preceding all subsequent design work

2. Multiple research methods should be combined for comprehensive understanding—interviews, surveys, observations, and focus groups each provide unique insights

3. Personas must be based on actual research data, not assumptions, and should be limited to 3-5 per product

4. User scenarios and stories keep design teams focused on real user goals rather than feature lists

5. Task analysis reveals the cognitive and physical processes involved in user interactions, enabling better interface design

6. Requirements should be prioritized using techniques like MoSCoW to focus development efforts on highest-impact features

7. The deliverables from requirements extraction (personas, scenarios, journey maps, use cases) directly inform and guide all subsequent design decisions

8. Both qualitative and quantitative research methods are essential—neither alone provides complete understanding

## Common Mistakes to Avoid

1. **Creating personas from assumptions rather than research data** - This leads to designs based on stereotypes rather than actual user needs

2. **Collecting requirements without understanding context** - The environment and situation in which users interact is as important as the interaction itself

3. **Prioritizing features over user goals** - Focus on what users need to achieve, not just what features to include

4. **Ignoring edge cases and alternative flows** - Use cases should cover not just the happy path but also error conditions and exceptions

5. **Treating requirements extraction as a one-time activity** - User needs evolve, and requirements should be revisited throughout the design process

## Revision Tips

1. Practice writing user stories in the standard format until it becomes automatic—this format is frequently tested

2. Create a comparison chart of research methods, their purposes, strengths, and limitations

3. Review the components of a complete persona and practice creating one from sample research data

4. Understand how each deliverable (personas → scenarios → journey maps → use cases → requirements) flows from the previous step

5. Memorize the MoSCoW prioritization categories and be able to apply them to sample requirement lists
