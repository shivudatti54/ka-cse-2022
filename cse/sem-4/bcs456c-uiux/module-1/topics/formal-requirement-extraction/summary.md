# Formal Requirement Extraction - Summary

## Key Definitions and Concepts

- **Requirement Extraction:** Systematic process of gathering, analyzing, and documenting user needs, business objectives, and technical constraints using structured techniques
- **Functional Requirements:** Statements describing specific behaviors and functions the system must perform
- **Non-Functional Requirements:** Quality attributes specifying how the system should perform (performance, usability, security, reliability)
- **User Persona:** Fictional character representing a typical user type based on research data
- **Use Case:** Structured description of system interactions from an actor's perspective
- **User Story:** Brief description of a feature from user's viewpoint in format "As a... I want... So that..."
- **SRS (Software Requirements Specification):** Comprehensive document containing all system requirements

## Important Formulas and Techniques

- **MoSCoW Prioritization:** Must have (critical) → Should have (important) → Could have (desirable) → Won't have (out of scope)
- **User Story Format:** "As a [user], I want [feature], so that [benefit]"
- **Likert Scale:** 5-7 point scale for measuring attitudes or opinions in questionnaires

## Key Points

1. Formal requirement extraction provides a structured foundation for UI/UX design decisions
2. Multiple techniques (interviews, observation, prototyping) should be used for comprehensive understanding
3. Functional requirements describe what the system does; non-functional requirements describe how it performs
4. User personas and scenarios help designers empathize with target users
5. Use cases provide detailed interaction flows between users and the system
6. Requirements must be testable, unambiguous, and traceable
7. Prioritization is essential due to resource and time constraints
8. Stakeholders include end users, business managers, developers, and regulatory bodies
9. Requirements validation ensures the right product is being built
10. Documentation serves as a contract between stakeholders and development teams

## Common Mistakes to Avoid

1. **Confusing requirements with features:** Requirements describe user needs; features are the solutions that fulfill those needs
2. **Ignoring non-functional requirements:** Performance, security, and usability are equally important as functional capabilities
3. **Single-source dependency:** Relying on only one stakeholder or technique leads to incomplete requirements
4. **Vague requirement statements:** Requirements must be specific and measurable, not ambiguous

## Revision Tips

1. Practice writing user stories in the standard format with acceptance criteria
2. Create sample use cases for common applications (banking, e-commerce, healthcare)
3. Review sample SRS documents to understand proper structure and formatting
4. Memorize key terminology and be able to distinguish between requirement types
5. Understand how each requirement gathering technique works and its pros/cons
