## Introduction

Exception priorities constitute a fundamental concept in modern microcontroller systems, particularly in ARM Cortex-M processors extensively deployed in embedded applications ranging from industrial control systems to consumer electronics. An exception in the ARM Cortex-M architecture refers to any event that disrupts the sequential flow of program execution, encompassing hardware interrupts generated by peripheral devices, software interrupts triggered by instructions such as SVC, fault conditions arising from memory violations or bus errors, and system-level events including reset and clock failures. When multiple exceptions manifest simultaneously or when an exception occurs during the servicing of another exception, the processor must determine the order of servicing based on a well-defined priority system that ensures critical real-time events receive immediate attention while maintaining system stability and predictable execution behavior.

The Nested Vectored Interrupt Controller (NVIC) embedded within all ARM Cortex-M processors implements a comprehensive and configurable priority scheme that enables developers to establish precise interrupt handling behavior. Understanding this exception priority mechanism is essential for developing reliable embedded systems capable of meeting real-time constraints, as improper priority configuration can lead to missed deadlines, priority inversion, or system deadlock. This module provides an in-depth examination of the ARM Cortex-M exception model, priority level allocation, priority grouping mechanisms, configuration register interfaces, and practical considerations for embedded system design. Students will gain both theoretical understanding and practical skills necessary for implementing robust interrupt-driven systems.

## Theoretical Foundation

### Exception Model Architecture

ARM Cortex-M processors implement a vector-based exception model wherein each exception type corresponds to an associated handler function whose address resides in a dedicated vector table located in memory. When an exception condition is detected, the processor executes a precisely defined sequence: first, it captures the current processor state including the Program Counter (PC), Program Status Register (xPSR), and general-purpose registers R0-R3, R12, and Link Register (LR) by pushing them onto the stack; second, it determines the exception type and assigns an appropriate exception number; third, it fetches the handler address from the vector table using the exception number as an offset; fourth, it branches to the handler code; and finally, upon handler completion, it executes an exception return sequence that restores the saved state and resumes interrupted program execution.

The Cortex-M exception model supports the following exception types in decreasing priority order: Reset (highest priority), Non-Maskable Interrupt (NMI), Hard Fault, Memory Management Fault, Bus Fault, Usage Fault, Reserved for system extensions, SVCall (Supervisor Call generated by SVC instruction), Debug Monitor, Reserved, PendSV (Pendable Service Call for OS scheduling), SysTick timer interrupt, and up to 240 device-specific interrupt requests (IRQs) numbered from IRQ0 through IRQ239. The priority of system exceptions (Reset through Debug Monitor) is fixed in hardware and cannot be modified by software, whereas the 240 IRQ priorities are fully programmable through the NVIC.

### Priority Level Implementation

The NVIC implements a configurable priority system wherein the number of available priority bits varies according to the specific Cortex-M implementation. Cortex-M0 and Cortex-M0+ processors implement 2 priority bits (4 levels), Cortex-M3, Cortex-M4, and Cortex-M7 typically implement 4 priority bits (16 levels), while high-performance variants may implement up to 8 priority bits (256 levels). The actual number of implemented bits is indicated by the Configuration Control Register (CCR) STKALIGN and PRI_BITS fields, accessible through the SCB (System Control Block).

A critical and often confusing characteristic of the ARM Cortex-M priority system is the numerical interpretation: lower numerical priority values represent _higher_ urgency, meaning priority 0 represents the most critical exception that cannot be preempted by any other exception, while higher numerical values (up to the maximum supported by the implementation) indicate lower urgency. Thus, an interrupt configured with priority 2 will be preempted by any active interrupt with priority 0 or 1, but will preempt any interrupt configured with priority 3 or higher. This convention aligns with hardware priority encoding where the most significant bits of the priority field represent the most significant priority levels.

### Priority Grouping and Sub-Priority

The AIRCR (Application Interrupt and Reset Control Register) within the System Control Block provides a PRIGROUP field that enables developers to partition the priority field into two components: preemption priority (also called group priority) and sub-priority. The preemption priority determines whether an incoming exception can preempt an exception currently being serviced—if the incoming exception has a higher preemption priority (numerically lower value) than the currently executing handler, the processor will suspend the current handler and service the higher-priority exception. Conversely, exceptions with equal or lower preemption priority cannot preempt the current handler.

When multiple exceptions of the same preemption priority are pending simultaneously, the sub-priority field determines the order of servicing. The sub-priority mechanism provides a tiebreaker for exceptions at identical preemption levels, ensuring deterministic behavior when several interrupts share the same group priority. The PRIGROUP field specifies how the available priority bits are divided, with values ranging from 0 (no grouping, all bits used for preemption priority) to 7 (maximum grouping, with fewer bits for preemption and more for sub-priority).

The relationship between PRIGROUP value and bit allocation follows the formula: if the implementation provides N priority bits, then the PRIGROUP value determines how many bits (G) are allocated to group priority, with the remaining (N-G) bits allocated to sub-priority. For a 4-bit implementation (16 priority levels), PRIGROUP values produce the following allocation: PRIGROUP=0 yields 4 bits for preemption (16 groups, 1 sub-priority each); PRIGROUP=1 yields 3 bits for preemption (8 groups, 2 sub-priorities each); PRIGROUP=2 yields 2 bits for preemption (4 groups, 4 sub-priorities each); PRIGROUP=3 yields 1 bit for preemption (2 groups, 8 sub-priorities each); and PRIGROUP=4 yields 0 bits for preemption (1 group, 16 sub-priorities).

### Priority Arbitration Algorithm

The NVIC implements a deterministic priority arbitration algorithm that executes whenever multiple pending exceptions require servicing. The algorithm proceeds as follows: first, the NVIC identifies all pending exceptions by examining the set-pending registers; second, it retrieves the effective priority of each pending exception, which combines both the configured priority value and any active priority masking; third, it selects the pending exception with the highest effective priority (lowest numerical value); fourth, if multiple exceptions share the same effective priority, the one with the lowest exception number is selected; and finally, the selected exception is activated and begins execution. This algorithm ensures completely deterministic behavior essential for real-time systems where worst-case response time must be precisely calculable.

## Hardware Implementation

### NVIC Register Interface

The NVIC provides memory-mapped registers for interrupt configuration, organized to provide byte-accessible priority registers for each interrupt line. The primary registers include: NVIC_ISER (Interrupt Set-Enable Register, read-write) to enable specific interrupts by writing the interrupt number; NVIC_ICER (Interrupt Clear-Enable Register, read-write) to disable interrupts; NVIC_ISPR (Interrupt Set-Pending Register, read-write) to manually set pending status; NVIC_ICPR (Interrupt Clear-Pending Register, read-write) to clear pending status; and NVIC_IPR[0] through NVIC_IPR[59] (Interrupt Priority Registers, read-write) to store the 8-bit priority value for each interrupt, though only the implemented upper bits are significant.

Each interrupt has an associated priority register within the NVIC_IPR array, with the mapping given by interrupt number N mapping to NVIC_IPR[N/4] at byte offset (N%4). For example, IRQ0 maps to NVIC_IPR[0] bits[7:0], IRQ1 maps to bits[15:8], IRQ2 maps to bits[23:16], and IRQ3 maps to bits[31:16]. This byte-aligned organization allows efficient priority configuration using simple byte store instructions.

### CMSIS Configuration Functions

The ARM Cortex Microcontroller Software Interface Standard (CMSIS) provides standardized API functions for interrupt configuration that abstract the underlying register details. The primary functions include: NVIC_EnableIRQ(IRQn_Type IRQn) to enable a specific interrupt by setting the corresponding bit in NVIC_ISER; NVIC_DisableIRQ(IRQn_Type IRQn) to disable an interrupt by setting the corresponding bit in NVIC_ICER; NVIC_SetPriority(IRQn_Type IRQn, uint32_t priority) to configure the priority of an interrupt by writing to the appropriate byte within the NVIC_IPR registers; and NVIC_GetPriority(IRQn_Type IRQn) to retrieve the current priority configuration.

For priority grouping configuration, the CMSIS function NVIC_SetPriorityGrouping(uint32_t PriorityGroup) writes the PRIGROUP field of the AIRCR register. The parameter must be one of the predefined values: NVIC_PRIORITYGROUP_0 (0 bits for preemption, 4 bits for sub-priority in 4-bit implementations), NVIC_PRIORITYGROUP_1 (1 bit for preemption, 3 bits for sub-priority), NVIC_PRIORITYGROUP_2 (2 bits for preemption, 2 bits for sub-priority), NVIC_PRIORITYGROUP_3 (3 bits for preemption, 1 bit for sub-priority), or NVIC_PRIORITYGROUP_4 (4 bits for preemption, 0 bits for sub-priority).

### Programming Example

Consider a practical configuration scenario for an STM32F4 microcontroller (Cortex-M4 with 4 priority bits) requiring three interrupts with specific priority relationships: Timer0 with highest urgency, UART with medium urgency, and Button with lowest urgency among these three. The configuration code using CMSIS would proceed as follows: first, enable the clocks to the peripherals using RCC peripheral clock enable registers; second, configure priority grouping using NVIC_SetPriorityGrouping(NVIC_PRIORITYGROUP_2) to establish 2 bits for preemption and 2 bits for sub-priority; third, set Timer0 priority to 0 (highest) using NVIC_SetPriority(TIMER0_IRQn, 0); fourth, set UART priority to 2 using NVIC_SetPriority(UART_IRQn, 2); fifth, set Button priority to 4 using NVIC_SetPriority(BUTTON_IRQn, 4); and finally, enable each interrupt using NVIC_EnableIRQ for each respective IRQ.

## Nested Exception Handling

### Preemption and Stacking

One of the significant capabilities of the ARM Cortex-M exception model is automatic nested interrupt handling. When a higher-priority exception occurs while a lower-priority exception handler is executing, the processor automatically performs a preemption sequence: it saves the current handler's state (including additional registers R4-R11 and the FPSCR if floating-point context is active) onto the stack, loads the new exception's handler address from the vector table, and begins executing the higher-priority handler. Upon completion, the exception return mechanism automatically restores the preempted handler's state, allowing it to resume execution from the point of preemption.

The stack frame saved during exception entry contains eight registers: R0, R1, R2, R3, R12, LR, PC, and xPSR, occupying 32 bytes on the stack. When nested exceptions occur, each level of nesting requires an additional 32-byte stack frame, plus any additional context saving required by the handler itself. System designers must allocate sufficient stack memory to accommodate the maximum expected nesting depth, typically calculated as (number of priority levels × 32 bytes) plus handler-specific local variables and function call stack usage.

### Priority Masking and Critical Sections

The ARM Cortex-M processor provides the PRIMASK register, a single-bit register that, when set to 1, masks all configurable interrupts (IRQs) while permitting only the NMI and Hard Fault handlers to execute. The BASEPRI register provides more granular control by masking all interrupts at or below a specified priority level—setting BASEPRI to value X prevents any interrupt with priority value ≥ X from being serviced. The BASEPRI_MAX macro allows setting the maximum effective priority blocking value. TheFAULTMASK register extends masking to include the Hard Fault exception, allowing critical code sequences to execute without any interruption.

These masking mechanisms are essential for implementing atomic operations and critical sections in embedded applications. A typical critical section implementation would save the current PRIMASK or BASEPRI state, set the masking register to prevent interrupts, perform the protected operation, and then restore the saved mask state. This ensures that time-critical or data-consistency-critical code sequences execute atomically without possibility of interruption.

## Priority-Related Phenomena

### Priority Inversion and Mitigation

Priority inversion constitutes a significant problem in embedded systems where a high-priority task waits for a resource held by a lower-priority task while a medium-priority task preempts the lower-priority task, effectively inverting the priority relationship. In interrupt-driven systems, priority inversion can manifest when an interrupt handler shares data with a task through a shared resource. The classic solution is priority inheritance, where the lower-priority task temporarily inherits the priority of the waiting high-priority task, preventing medium-priority tasks from preempting the critical section. However, the ARM Cortex-M NVIC does not implement automatic priority inheritance in hardware—software mechanisms or real-time operating systems must provide this functionality.

A related phenomenon is stack overflow in nested interrupt scenarios. If interrupt handlers are poorly designed with excessive stack usage or unbounded recursion, nested interrupts can consume available stack space, potentially corrupting memory or causing hard faults. Best practices include: allocating adequate stack space during system initialization; minimizing stack usage within handlers through deferred processing; using the stack size calculation methodology: Stack_Size = (Maximum_Nesting_Depth × 32 bytes) + (Maximum_Local_Variables) + (Function_Call_Depth × 16 bytes); and employing stack overflow detection mechanisms available on some Cortex-M variants.

## Conclusion

Exception priorities in ARM Cortex-M microcontrollers provide a sophisticated mechanism for managing concurrent asynchronous events in embedded systems. The NVIC implements a flexible priority system supporting multiple priority levels, priority grouping with preemption and sub-priority components, and deterministic arbitration through a well-defined algorithm. Understanding the numerical convention where lower values indicate higher priority, the PRIGROUP configuration for priority field partitioning, and the proper use of priority masking registers is essential for developing reliable real-time embedded applications. The nested exception handling capability enables high-priority events to preempt lower-priority handlers, but requires careful stack planning to prevent overflow conditions. These concepts collectively form the foundation for designing embedded systems that meet stringent real-time performance requirements.
