## Introduction

Exception handling is a fundamental mechanism in ARM microcontrollers that enables the processor to respond to asynchronous events and critical conditions requiring immediate attention. When an exception occurs, the processor suspends normal execution flow and transfers control to a dedicated exception handler routine designed to address the specific event. This mechanism forms the backbone of interrupt-driven systems, enabling real-time response to external stimuli, hardware faults, and software-generated events.

In ARM architecture, exceptions include hardware interrupts (IRQ and FIQ), memory faults, undefined instructions, software interrupts (SWI), and processor reset. Each exception type is associated with a specific mode of operation, and the processor maintains separate banked registers for each exception mode to ensure complete context preservation during exception handling. Understanding exception handling is essential for embedded systems developers as it directly impacts system reliability, interrupt latency, and overall microcontroller performance.

This topic examines the ARM exception handling architecture, including exception types and modes, vector table configuration, priority schemes, and practical implementation considerations for firmware development.

## Key Concepts

### Exception Types and Modes

ARM processors support multiple exception types, each corresponding to a specific operating mode:

1. **Reset**: Occurs when the reset signal is asserted. The processor enters Supervisor mode and execution begins at the reset vector address (0x00000000 or 0xFFFF0000 depending on configuration).

2. **Undefined Instruction**: Generated when the processor attempts to execute an instruction that is not in the ARM or Thumb instruction set. This exception enables software emulation of coprocessor instructions.

3. **Software Interrupt (SWI)**: Triggered by executing the SWI instruction. Commonly used to implement system calls, allowing user-mode programs to request privileged operating system services.

4. **Prefetch Abort**: Occurs when the processor attempts to fetch an instruction from a non-permissible memory location. The memory management unit (MMU) or memory protection unit (MPU) generates this exception.

5. **Data Abort**: Generated when a data memory access violates memory protection or translation rules. This exception handles both read and write access violations.

6. **IRQ (Interrupt Request)**: A standard priority interrupt generated by peripheral devices. The processor enters IRQ mode and executes the IRQ handler.

7. **FIQ (Fast Interrupt Request)**: A high-priority, low-latency interrupt designed for time-critical operations. FIQ mode has dedicated registers (R8-R12) to minimize context saving overhead.

### Exception Vector Table

The exception vector table is a fixed memory location containing branch instructions or addresses for each exception handler. In ARM architecture, the vector table typically occupies 32 bytes (8 vectors × 4 bytes each) and is located at either the low memory address (0x00000000) or high address (0xFFFF0000) depending on the configuration.

The standard vector table layout is:

- 0x00: Reset
- 0x04: Undefined Instruction
- 0x08: Software Interrupt (SWI)
- 0x0C: Prefetch Abort
- 0x10: Data Abort
- 0x14: Reserved
- 0x18: IRQ
- 0x1C: FIQ

The vector table can be implemented using either branch instructions (relative offsets) or vector table entries containing absolute handler addresses. Modern ARM systems often use the "vector table offset" feature to relocate the vector table dynamically.

### Exception Handling Sequence

When an exception occurs, the processor follows a precise sequence:

1. **Mode Change**: The processor switches to the corresponding exception mode, bank switching registers as defined for that mode.

2. **Link Register Saving**: The processor saves the return address (current PC + 4 for ARM state) to the Link Register (LR) of the new mode.

3. **SPSR Saving**: The Current Program Status Register (CPSR) is copied to the Saved Program Status Register (SPSR) of the new mode.

4. **PC Update**: The processor disables IRQ interrupts (sets the I bit in CPSR) and updates the Program Counter (PC) to the exception vector address.

5. **Handler Execution**: Control transfers to the exception handler, which performs necessary processing.

6. **Return**: The handler executes appropriate return instructions (SUBS PC, LR, #4 or similar) to restore the saved state and return to the interrupted program.

### Exception Priority

ARM processors implement a fixed priority scheme for exceptions. From highest to lowest priority:

1. Reset (highest)
2. Data Abort
3. FIQ
4. IRQ
5. Prefetch Abort
6. Undefined Instruction
7. SWI (lowest)

When multiple exceptions occur simultaneously, the processor services them according to this priority order. The FIQ handler can execute with IRQs disabled, ensuring deterministic latency for critical interrupts.

### Interrupt Latency

Interrupt latency refers to the time between the assertion of an interrupt request and the execution of the first instruction of the interrupt handler. Key factors affecting latency include:

- **Current processor state**: Interrupt disable bits (I and F flags in CPSR)
- **Memory access times**: Vector table and handler code location in memory
- **Pipeline depth**: ARM processors with deeper pipelines may require additional cycles
- **Priority**: Higher priority interrupts preempt lower priority ones

FIQ interrupts provide lower latency than IRQ due to dedicated banked registers and the ability to place the FIQ handler at the end of the vector table (immediately following the other vectors), eliminating the need for a branch instruction.

## Examples

### Example 1: Basic IRQ Handler Implementation in Assembly

```assembly
; IRQ Handler in ARM assembly
; Assumes vector table is configured and IRQ handler address is set

IRQ_Handler:
 ; Save context - push registers onto stack
 SUB LR, LR, #4 ; Adjust LR for correct return address
 STMFD SP!, {R0-R3, LR} ; Save LR and caller-saved registers

 ; Read the interrupt source from the VIC (Vectored Interrupt Controller)
 LDR R0, =0xFFFFF000 ; VIC base address
 LDR R1, [R0, #0x0C] ; VICirqStatus register

 ; Check and handle specific interrupt sources
 CMP R1, #0x01 ; Check Timer 0 interrupt
 BEQ Timer0_Handler

 CMP R1, #0x02 ; Check UART interrupt
 BEQ UART_Handler

 ; Restore context and return
 LDMFD SP!, {R0-R3, PC}^ ; Restore registers and return

Timer0_Handler:
 ; Handle Timer 0 interrupt
 LDR R2, =0xE0005000 ; Timer0 base address
 LDR R3, [R2, #0x00] ; Read Timer interrupt register
 STR R3, [R2, #0x00] ; Clear interrupt
 ; Additional timer handling...
 B IRQ_Handler_End

UART_Handler:
 ; Handle UART interrupt
 LDR R2, =0xE0004000 ; UART base address
 LDR R3, [R2, #0x00] ; Read UART interrupt ID
 ; Process UART data...
 B IRQ_Handler_End

IRQ_Handler_End:
 LDMFD SP!, {R0-R3, PC}^
```

### Example 2: Nested Interrupt Handler in C with Context Saving

```c
// Nested IRQ handler in C for ARM
// Requires proper stack configuration and priority settings

extern void VIC_SetIRQHandler(void (*handler)(void));

void (* volatile irq_handler)(void) = NULL;

// Main IRQ handler dispatcher
void IRQ_Handler(void) {
 // Get the interrupt source from VIC
 volatile unsigned int *vic_irq = (unsigned int *)0xFFFFF00C;
 unsigned int irq_status = *vic_irq;

 // Check for Timer 0 interrupt (bit 0)
 if (irq_status & (1 << 0)) {
 handle_timer0_interrupt();
 }

 // Check for UART 0 interrupt (bit 3)
 if (irq_status & (1 << 3)) {
 handle_uart0_interrupt();
 }

 // Clear interrupt at peripheral level
 clear_peripheral_interrupts(irq_status);
}

// Timer 0 interrupt handler
void handle_timer0_interrupt(void) {
 volatile unsigned int *timer_reg = (unsigned int *)0xE0005000;

 // Clear timer interrupt
 timer_reg[0] = timer_reg[0]; // Read to clear

 // Process timer event
 timer_tick++;

 // Optional: Re-enable interrupts if nested handling needed
 // __enable_irq();
}

// Nested exception handler function
void enable_nested_irq(void) {
 // Enable IRQ in CPSR (clear I bit)
 __asm volatile ("CPSIE i");
}
```

### Example 3: Vector Table Configuration

```c
// Vector table definition and relocation for ARM

// Standard vector table structure
typedef struct {
 void (*reset)(void);
 void (*undefined_instruction)(void);
 void (*software_interrupt)(void);
 void (*prefetch_abort)(void);
 void (*data_abort)(void);
 void (*reserved)(void);
 void (*irq)(void);
 void (*fiq)(void);
} vector_table_t;

// Weak default handlers that can be overridden
__attribute__((weak, at(0x00000000)))
vector_table_t __vector_table = {
 .reset = Reset_Handler,
 .undefined_instruction = Default_Handler,
 .software_interrupt = Default_Handler,
 .prefetch_abort = Default_Handler,
 .data_abort = Default_Handler,
 .reserved = Default_Handler,
 .irq = IRQ_Handler,
 .fiq = FIQ_Handler,
};

// Function to relocate vector table
void vector_table_init(unsigned int new_address) {
 volatile unsigned int *vtcr = (unsigned int *)0xE000ED08;
 unsigned int offset = new_address - 0x00000000;

 // Set Vector Table Offset Register (SCB->VTOR)
 *vtcr = offset & 0xFFFFFE00; // Must be aligned to 128-byte boundary
}

// Configure and enable specific interrupts
void interrupt_enable(unsigned int irq_num, unsigned int priority) {
 volatile unsigned int *vic_enable = (unsigned int *)0xFFFFF010;
 volatile unsigned int *vic_priority = (unsigned int *)0xFFFFF400;

 // Enable the interrupt
 *vic_enable = (1 << irq_num);

 // Set priority (lower value = higher priority)
 vic_priority[irq_num] = priority;
}
```

## Exam Tips

1. **Memorize the Exception Vector Order**: The vector table order (Reset, Undefined, SWI, Prefetch Abort, Data Abort, Reserved, IRQ, FIQ) is frequently tested. Remember the offset addresses: 0x00, 0x04, 0x08, 0x0C, 0x10, 0x14, 0x18, 0x1C.

2. **Understand Banked Registers**: Be clear about which registers are banked in each exception mode (SP, LR, SPSR) versus which are common across all modes.

3. **Know the LR Offset Calculation**: For exception returns, remember that LR is saved as "address of the next instruction + 4" in ARM state. The exact offset subtraction depends on the exception type (typically -4 or -8).

4. **Distinguish IRQ vs FIQ**: FIQ has higher priority, lower latency, dedicated registers (R8-R12), and can be placed directly after the vector table to avoid branch overhead.

5. **CPSR vs SPSR**: Remember that CPSR is the current status register (accessible in all modes), while SPSR is the saved status register (one per exception mode) used to restore CPSR on exception return.

6. **Exception Priority Order**: Know the priority sequence: Reset > Data Abort > FIQ > IRQ > Prefetch Abort > Undefined > SWI.

7. **Context Saving Overhead**: Be aware of the trade-offs between fully interruptible handlers (nested interrupts) and simple handlers that complete quickly but block other interrupts.
