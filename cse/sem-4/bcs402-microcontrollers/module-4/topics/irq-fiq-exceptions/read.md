## Introduction

Exception handling constitutes a fundamental mechanism in microcontroller systems that enables the processor to respond to asynchronous events and urgent conditions with minimal latency. In ARM-based microcontrollers, exceptions represent events that disrupt the normal sequential flow of program execution and necessitate special processing actions by the processor. The two primary interrupt-driven exception mechanisms in ARM architecture are the Interrupt Request (IRQ) and Fast Interrupt Request (FIQ), which provide distinct pathways for peripheral devices to signal the processor and request service.

Understanding IRQ and FIQ exceptions is essential for embedded systems developers because these mechanisms facilitate interaction between the microcontroller and the external environment, enabling response to user inputs, handling of time-critical events, and efficient management of multiple peripheral devices. The absence of proper interrupt handling would constrain microcontrollers to polling-based approaches, which consume excessive processing resources and introduce unacceptable latency in real-time applications.

This content addresses the foundational concepts of ARM exception handling as applied in the BCS402 Microcontrollers course, forming the theoretical basis for interrupt-driven programming paradigms essential in real-time embedded system development. Modern ARM Cortex-M processors implement these fundamental concepts through the Nested Vectored Interrupt Controller (NVIC), which provides sophisticated interrupt management including configurable priority levels, interrupt nesting, and vectored exception handling.

## Theoretical Foundation

### Exception Classification in ARM Architecture

ARM processors implement a comprehensive exception handling mechanism that categorizes exceptions into distinct types, each serving a specific purpose and employing unique handling procedures. The exception model in ARM architecture (specifically ARMv4T and subsequent versions) classifies exceptions with defined priority levels and vector addresses.

The primary exception types in ARM processors include:

1. **Reset Exception**: Invoked during system power-on, hardware reset, or software-initiated reset. This exception has the highest priority and initializes the processor to a known state.

2. **Undefined Instruction Exception**: Generated when the processor encounters an instruction that it cannot decode or execute, enabling software emulation of unsupported instructions.

3. **Software Interrupt (SWI) Exception**: Triggered by the SWI instruction, providing a mechanism for user-mode programs to request privileged operating system services.

4. **Prefetch Abort Exception**: Raised when a memory access failure occurs during instruction fetch, indicating an invalid memory address in the instruction stream.

5. **Data Abort Exception**: Generated when a data memory access fails, typically due to memory protection violations or alignment errors.

6. **IRQ (Interrupt Request) Exception**: Generated by external peripheral devices requesting processor attention through the IRQ line.

7. **FIQ (Fast Interrupt Request) Exception**: High-priority interrupt designed for time-critical operations requiring minimum latency.

### IRQ (Interrupt Request) Mechanism

The IRQ exception serves as the standard interrupt mechanism in ARM processors, optimized for handling general-purpose peripheral interrupts with moderate latency requirements. When an IRQ exception is triggered, the processor executes a deterministic sequence of operations to preserve state and transfer control to the appropriate interrupt service routine.

The IRQ exception handling sequence proceeds as follows: First, the processor completes the currently executing instruction to maintain instruction atomicity. Second, the processor saves the return address (address of the next instruction to be executed) onto the IRQ mode stack. Third, the CPSR (Current Program Status Register) contents are copied to SPSR_irq (Saved Program Status Register for IRQ mode), preserving the processor state. Fourth, the processor switches to IRQ mode with IRQ interrupts disabled. Fifth, the program counter is loaded with the IRQ vector address (0x00000018 in ARM7TDMI), transferring control to the IRQ handler.

Key characteristics of the IRQ mechanism include:

- **Priority**: IRQ possesses lower priority relative to FIQ, enabling FIQ interrupts to preempt IRQ processing when both occur simultaneously.

- **Masking Capability**: The I bit (bit 7) in the CPSR controls IRQ masking. When set to 1, IRQ interrupts are globally disabled; when cleared to 0, IRQ interrupts are enabled.

- **Nesting Support**: IRQ supports nested interrupts when properly configured, requiring the handler to explicitly enable IRQ interrupts (with care taken to prevent unbounded nesting).

- **Vector Address**: The IRQ vector is located at address 0x00000018 in the exception vector table.

- **Typical Applications**: Non-time-critical peripheral interrupts including keyboard input, timer overflow, UART receive/transmit complete, and ADC conversion complete.

The IRQ handler must implement proper register preservation by saving all registers it modifies onto the stack and restoring them before returning to the interrupted program. This ensures the interrupted program resumes execution without corruption of its register state.

### FIQ (Fast Interrupt Request) Mechanism

The FIQ exception provides a high-priority, low-latency interrupt pathway specifically designed for time-critical applications requiring rapid response. The FIQ mechanism achieves reduced latency through architectural optimizations including dedicated register banking and strategic vector placement.

The FIQ mechanism incorporates the following architectural features:

- **Dedicated Register Bank**: FIQ mode provides private copies of registers R8 through R12 (R8_fiq through R12_fiq), enabling the FIQ handler to access these registers without saving them to stack memory. This eliminates the context saving overhead that contributes significantly to interrupt latency.

- **Priority Level**: FIQ possesses the highest priority among asynchronous exceptions (excluding Reset), ensuring it can preempt IRQ handlers and achieve minimum response time.

- **Non-Preemptability**: When FIQ is enabled (F bit in CPSR is cleared), FIQ interrupts cannot be interrupted by IRQ interrupts, guaranteeing deterministic service time for the FIQ handler.

- **Vector Address**: The FIQ vector is positioned at address 0x0000001C in the exception vector table, immediately following the IRQ vector. This placement allows the FIQ handler to be placed at this address directly, eliminating the branch instruction overhead required for other exceptions.

- **Masking Capability**: The F bit (bit 6) in the CPSR controls FIQ masking. When set to 1, FIQ interrupts are disabled.

- **Typical Applications**: High-speed data transfer operations (DMA completion), real-time control systems, time-critical sensor data acquisition, and audio processing.

### CPSR and SPSR in Exception Handling

The Current Program Status Register (CPSR) contains critical flags and control bits that govern interrupt handling behavior. Understanding the bit-level structure of the CPSR is essential for proper exception management.

The relevant CPSR bits for interrupt handling include:

| Bit Position | Designation | Function Description                                                                            |
| ------------ | ----------- | ----------------------------------------------------------------------------------------------- |
| 5            | T           | Thumb state indicator (1 = Thumb, 0 = ARM)                                                      |
| 6            | F           | FIQ disable bit (1 = FIQ disabled, 0 = FIQ enabled)                                             |
| 7            | I           | IRQ disable bit (1 = IRQ disabled, 0 = IRQ enabled)                                             |
| [4:0]        | Mode        | Processor mode bits (10000=User, 10001=FIQ, 10010=IRQ, 10011=SVC, 10111=Abort, 11011=Undefined) |

When an exception occurs, the CPSR is automatically copied to the SPSR (Saved Program Status Register) of the target exception mode, preserving the processor state. Upon exception return, the SPSR is restored to CPSR, reinstating the pre-exception processor state.

### Exception Vector Table

The exception vector table contains fixed memory addresses where the processor transfers control when specific exceptions occur. The vector table layout for ARM7TDMI is presented below:

| Exception Type        | Vector Address | Exception Mode |
| --------------------- | -------------- | -------------- |
| Reset                 | 0x00000000     | Supervisor     |
| Undefined Instruction | 0x00000004     | Undefined      |
| Software Interrupt    | 0x00000008     | Supervisor     |
| Prefetch Abort        | 0x0000000C     | Abort          |
| Data Abort            | 0x00000010     | Abort          |
| IRQ                   | 0x00000018     | IRQ            |
| FIQ                   | 0x0000001C     | FIQ            |

The vector placement at 0x0000001C for FIQ enables direct handler placement at this address without requiring an additional branch instruction, reducing latency by one instruction fetch cycle.

### Interrupt Latency Analysis

Interrupt latency represents the time interval between the occurrence of an interrupt condition and the commencement of its service routine. This metric is critical in real-time embedded systems where meeting timing constraints is essential.

The total interrupt latency in ARM processors comprises multiple components:

1. **Instruction Completion Latency**: The processor must complete the currently executing instruction before recognizing the interrupt. Load and store multiple instructions may require multiple cycles.

2. **Exception Recognition Latency**: The processor requires additional cycles to recognize the interrupt condition and initiate the exception entry sequence.

3. **State Saving Latency**: Time consumed in saving return address to the stack and copying CPSR to SPSR.

4. **Vector Fetch Latency**: Time required to fetch the instruction at the exception vector address.

5. **Handler Prologue Latency**: Time spent in the handler entry code saving callee-saved registers.

The FIQ mechanism achieves lower latency compared to IRQ through multiple optimizations: dedicated register banking eliminates saving R8-R12 to the stack, vector placement at 0x0000001C allows direct handler execution, and FIQ's higher priority prevents IRQ-induced delays.

## Practical Implementation

### ARM7TDMI IRQ Handler Implementation

The following assembly code demonstrates a complete IRQ handler implementation:

```assembly
; IRQ Handler for ARM7TDMI
IRQ_Handler
 ; Calculate correct return address (LR points to next instruction + 4)
 SUB lr, lr, #4

 ; Save required registers onto IRQ mode stack
 STMFD sp!, {r0-r3, r12, lr}

 ; Switch to Supervisor mode for potential nested calls
 MRS r0, SPSR
 STMFD sp!, {r0, spsr, pc}^

 ; Enable IRQ for nested interrupts (if desired)
 MSR cpsr_c, #0x12

 ; Now in Supervisor mode, save additional registers
 STMFD sp!, {r4-r11, lr}

 ; Call C interrupt handler
 BL Timer_ISR

 ; Restore Supervisor mode registers
 LDMFD sp!, {r4-r11, lr}

 ; Return to IRQ mode
 MSR cpsr_c, #0x92

 ; Restore IRQ mode registers and SPSR
 LDMFD sp!, {r0-r3, r12, pc}^
```

### FIQ Handler Implementation

The FIQ handler benefits from reduced overhead due to banked registers:

```assembly
; FIQ Handler for ARM7TDMI
FIQ_Handler
 ; R8_fiq through R12_fiq available without saving
 ; Process high-speed data transfer completion

 LDR r0, =DMA_STATUS_REG
 LDR r1, [r0]

 ; Check if transfer complete
 TEQ r1, #0x01
 BEQ handle_dma_complete

 ; Exit FIQ handler
 SUBS pc, lr, #4

handle_dma_complete
 ; Process completed DMA transfer
 ; Direct register access without stack operations
 LDR r8, =DMA_BUFFER_ADDR
 LDR r9, [r8], #4

 ; Clear interrupt flag
 STR r1, [r0]

 ; Return from FIQ
 SUBS pc, lr, #4
```

## NVIC in ARM Cortex-M Processors

Modern ARM Cortex-M processors (Cortex-M0, M0+, M3, M4, M7) implement an enhanced interrupt handling architecture through the Nested Vectored Interrupt Controller (NVIC). The NVIC provides substantial improvements over the traditional IRQ/FIQ model:

- **Hardware Priority Levels**: The NVIC supports up to 256 priority levels (implementation-dependent), configurable through the priority bits in the Interrupt Set-Enable and Interrupt Priority registers.

- **Automatic State Saving**: The Cortex-M processor automatically pushes R0-R3, R12, LR, PSR, and PC onto the stack upon exception entry, substantially reducing handler overhead.

- **Vector Table Relocation**: The vector table can be relocated to flash or SRAM through the Vector Table Offset Register (VTOR), enabling bootloaders and runtime firmware updates.

- **Tail Chaining**: The NVIC optimizes interrupt sequencing by skipping stack push/pop operations when servicing pending interrupts back-to-back.

- **Late Arranging**: When a higher priority interrupt arrives during state saving of a lower priority interrupt, the NVIC can optimize latency by servicing the higher priority interrupt immediately.
