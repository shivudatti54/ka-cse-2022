# History of Database Applications - Summary

## Key Definitions and Concepts

- **File Processing System**: Early data storage method using sequential flat files with significant limitations including data redundancy and program-data dependence.
- **Hierarchical Database Model**: Tree-structured model with one-to-many relationships, pioneered by IBM's IMS (1966).
- **Network Database Model**: Graph-based model supporting many-to-many relationships, developed by CODASYL (1969).
- **Relational Model**: Table-based model proposed by E.F. Codd in 1970, using set theory and first-order predicate logic.
- **NoSQL Databases**: Non-relational databases designed for scalability, flexibility, and handling unstructured data (2000s).

## Important Formulas and Theorems

- **CAP Theorem**: States that distributed databases can only guarantee two of three: Consistency, Availability, and Partition tolerance.
- **ACID Properties**: Atomicity, Consistency, Isolation, Durability—fundamental properties of relational database transactions.

## Key Points

1. Early file processing systems suffered from data redundancy, program-data dependence, and lack of security.
2. The hierarchical model organizes data in tree structures, suitable for one-to-many relationships.
3. The network model uses graph structures to represent many-to-many relationships.
4. Edgar Codd's relational model (1970) revolutionized database technology with its mathematical foundation.
5. SQL became the standard query language for relational databases.
6. Object-oriented databases emerged to handle complex data types and object persistence.
7. NoSQL databases address scalability and flexibility needs of modern web applications.
8. Modern trends include NewSQL, cloud databases, and multi-model databases.

## Common Mistakes to Avoid

- Confusing hierarchical and network models—hierarchical has one-to-many while network supports many-to-many relationships.
- Believing NoSQL means "no SQL"—many NoSQL databases support SQL-like query languages.
- Assuming newer models are always better—relational databases remain optimal for many business applications.

## Revision Tips

1. Create a timeline chart showing database evolution with key dates and models.
2. Practice comparing different models using a structured table format.
3. Understand the "why" behind each evolution—know what problems each new model solved.
4. Review university question papers to identify frequently asked questions on this topic.
