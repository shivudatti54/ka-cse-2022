# Interrupt Hardware Architecture

## 1. Introduction and Formal Definition

An **interrupt** constitutes a hardware-level mechanism that enables asynchronous notification to the CPU regarding the occurrence of significant events requiring immediate computational intervention. According to Tanenbaum and Bos (2014), "an interrupt is a signal to the CPU indicating that an event has occurred that needs immediate attention." When the processor detects an interrupt request (IRQ), it suspends execution of the current instruction stream, preserves the processor state, and transfers control to a designated **Interrupt Service Routine (ISR)** or **Interrupt Handler** to service the requesting device.

The interrupt mechanism fundamentally transforms the CPU's operational paradigm from **polling-based I/O**, wherein the processor iteratively examines device status registers, to an **event-driven model** where devices proactively signal the CPU upon requiring service. This transition yields substantial improvements in CPU utilization and response latency.

## 2. Classification of Interrupts

### 2.1 Hardware Interrupts (Asynchronous)

Hardware interrupts originate from external peripheral devices and are asynchronous with respect to the executing program flow. These interrupts are classified based on their electrical signaling characteristics:

- **Edge-Triggered Interrupts:** The interrupt is recognized on a **transition** (rising or falling edge) of the IRQ signal. The interrupt is latched within the interrupt controller until acknowledged. These are less susceptible to noise but may miss interrupts if the pulse width is insufficient.

- **Level-Triggered Interrupts:** The interrupt request is recognized as long as the IRQ line remains active (at logic high or low). This configuration supports interrupt nesting and allows multiple devices to share IRQ lines through **wired-OR** logic, though it requires careful handling to prevent repeated service calls.

### 2.2 Software Interrupts (Traps)

Software interrupts, also termed **traps**, are synchronously generated by executing a specific instruction—`INT n` in the x86 architecture. These are intentional interruptions invoked by programs to request operating system services (system calls) or for debugging purposes (breakpoints via `INT 3`).

### 2.3 Exceptions

Exceptions are synchronous interrupts generated by the CPU itself when detecting anomalous conditions during instruction execution:

- **Faults:** Page faults, division faults—recoverable conditions where the instruction can be restarted after resolution.
- **Traps:** Integer overflow, breakpoint—reported after successful instruction execution.
- **Aborts:** Severe hardware errors indicating inconsistent processor state.

## 3. Interrupt Hardware Architecture

### 3.1 Interrupt Request Lines (IRQs)

Each peripheral device requiring interrupt service is assigned a dedicated or shareable **Interrupt Request Line (IRQ)**. In the **ISA bus** architecture, 16 IRQ lines (IRQ0-IRQ15) are available. Modern systems utilizing the **PCI express** architecture support message-signaled interrupts (MSI), eliminating physical IRQ line constraints.

### 3.2 Interrupt Controllers

The **Programmable Interrupt Controller (PIC)** serves as an intermediate hardware component between multiple IRQ sources and the CPU's single interrupt pin. The **Intel 8259A PIC**, historically significant in IBM PC/AT architecture, manages eight IRQ lines through a cascaded configuration of two 8259A chips (master and slave), providing 16 IRQ channels.

#### 8259A PIC Architecture:

The 8259A incorporates the following functional components:

1. **Interrupt Request Register (IRR):** Stores pending interrupt requests from all IRQ lines.
2. **In-Service Register (ISR):** Maintains interrupts currently being serviced.
3. **Priority Resolver:** Determines the highest-priority pending interrupt when the CPU acknowledges the interrupt.
4. **Interrupt Mask Register (IMR):** Masks specific IRQs to prevent their service.

**Operation Sequence:**

1. Device raises interrupt on assigned IRQ line.
2. PIC latches the request in IRR.
3. Priority resolver identifies highest-priority unmasked request.
4. PIC asserts INT pin to CPU.
5. CPU acknowledges via INTA (Interrupt Acknowledge) cycle.
6. PIC places the interrupt vector onto the data bus.

#### Advanced PIC (APIC):

The **Advanced Programmable Interrupt Controller (APIC)** architecture, prevalent in modern multiprocessor systems, extends interrupt handling capabilities through:

- Support for 24+ IRQ lines
- **Local APIC (LAPIC)** per processor core
- **I/O APIC** for interrupt routing
- Distributed interrupt management across multiple CPUs
- Message Signaled Interrupts (MSI) support

### 3.3 Daisy-Chain Arbitration

In systems without centralized interrupt controllers, **daisy-chain arbitration** provides a distributed priority mechanism. Devices are connected in a prioritized series; a device asserting interrupt passes control down the chain until a handler acknowledges the interrupt. Higher-priority devices in the chain receive service before lower-priority devices.

## 4. Interrupt Vectoring Mechanism

### 4.1 Interrupt Vector Table (IVT)

The **Interrupt Vector Table** (in x86 real mode) or **Interrupt Descriptor Table (IDT)** (in protected/long mode) constitutes a data structure containing pointers to interrupt service routines. In real mode, the IVT occupies the first 1 KB of memory (addresses 0x00000-0x003FF), with each entry comprising 4 bytes (CS:IP pointer), yielding 256 possible interrupt vectors.

```
Memory Address Vector Number Purpose
0x00000 0 Divide Error
0x00004 1 Single Step (Debug)
0x00008 2 NMI Interrupt
...
0x00020-0x00023 8 IRQ0 (System Timer)
0x00024-0x00027 9 IRQ1 (Keyboard)
...
```

### 4.2 Vectored Interrupt Processing Sequence

The complete interrupt acknowledgment sequence involves:

1. **Device Assertion:** Device raises IRQ signal.
2. **CPU Acknowledgment:** CPU completes current instruction and acknowledges via INTA cycle.
3. **Vector Acquisition:** CPU reads interrupt vector number from the interrupt controller.
4. **State Preservation:** CPU pushes FLAGS, CS, IP onto stack.
5. **Handler Invocation:** CPU loads CS:IP from IVT/IDT entry.
6. **Execution:** ISR executes with interrupts typically disabled.
7. **Return:** IRET instruction restores state and resumes original execution.

## 5. Interrupt Priority and Handling Multiple Interrupts

### 5.1 Priority Schemes

- **Fixed Priority:** Highest-priority device always serviced first; lower-priority devices may experience extended latency (starvation).
- **Rotating Priority:** The device receiving service becomes lowest priority; subsequent requests rotate through the priority chain.
- **Programmable Priority:** Software configures priority levels through PIC control words (OCW2).

### 5.2 Nested Interrupts

When **interrupt nesting** is enabled (IF flag set in EFLAGS), a higher-priority interrupt can preempt a lower-priority ISR. This requires:

- Preserving additional state (additional registers)
- Ensuring ISR reentrancy through proper synchronization
- Maintaining stack depth considerations

### 5.3 Masking Mechanisms

The CPU's **Interrupt Flag (IF)** controls maskable interrupt acceptance globally. The PIC's **Interrupt Mask Register (IMR)** provides per-IRQ masking. Critical sections often disable interrupts (`cli`) to ensure atomicity.

## 6. Interrupt Latency and Overhead Analysis

### 6.1 Interrupt Latency Components

**Total Interrupt Latency (L)** comprises:

$$L = L_{device} + L_{controller} + L_{CPU\_ack} + L_{vector} + L_{ISR\_entry}$$

Where:

- **L_device:** Time from event occurrence to IRQ signal assertion
- **L_controller:** PIC latency for priority resolution (typically 80-200 ns)
- **L_CPU_ack:** CPU INTA cycle propagation delay
- **L_vector:** Time to fetch vector from controller
- **L_ISR_entry:** ISR prologue execution time

### 6.2 Overhead Calculation

**Interrupt Overhead (H)** for N interrupts per second:

$$H = N \times (T_{save} + T_{ISR} + T_{restore})$$

Where:

- **T_save:** Context save time (typically 10-50 instruction cycles)
- **T_ISR:** Actual interrupt service routine execution time
- **T_restore:** Context restore time

### 6.3 Polling vs Interrupt: Break-Even Analysis

For a device generating **D** interrupts per second, with **P** being polling interval overhead and **I** being interrupt handling overhead:

- **Polling:** Total overhead = P (fixed per second)
- **Interrupt:** Total overhead = D × I

Interrupts are superior when: **D × I < P**, typically when device activity is infrequent or bursty.

## 7. Summary

The interrupt hardware architecture represents a sophisticated coordination between CPU hardware, interrupt controllers, and peripheral devices. Key hardware components—IRQ lines, programmable interrupt controllers (8259A PIC, APIC), and interrupt vector tables—collectively enable efficient, prioritized event handling. Understanding the detailed mechanisms of interrupt acknowledgment, vectoring, priority resolution, and latency analysis is essential for embedded systems design and operating system development at the B.Tech/MSc level.
