# Enabling and Disabling Interrupts

## Introduction

Interrupts are fundamental to modern computing systems, serving as mechanisms by which peripheral devices and internal events notify the CPU requiring immediate attention. While interrupts are essential for efficient system operation and responsiveness, there exist critical scenarios where their temporary suppression becomes necessary to maintain data integrity and system stability. This chapter provides a comprehensive examination of interrupt enable/disable mechanisms, their implementation across processor architectures, and the formal guarantees they provide in concurrent systems.

## Theoretical Foundation of Interrupts

### Formal Definition

An **interrupt** may be formally defined as an asynchronous event that temporarily suspends the normal execution flow of a processor, transfers control to a designated handler routine, and subsequently resumes execution from the point of suspension. Mathematically, we can model the interrupt mechanism using state transitions:

```
State: (PC, PSW, Registers, Memory)
Interrupt Event e triggers transition:
S → S' where S' = (ISR_Entry, PSW', Registers', Memory)
```

The **Interrupt Service Routine (ISR)** is a special-purpose handler function invoked in response to specific interrupt conditions, characterized by its execution context, priority level, and atomicity requirements.

### Interrupt Processing Sequence

The complete interrupt handling cycle comprises five distinct phases:

1. **Interrupt Detection**: The processor samples interrupt request lines at specific points in the instruction cycle
2. **State Preservation**: Complete processor state (Program Counter, Status Word, General-Purpose Registers) is saved to stack or dedicated registers
3. **Vector Fetch**: The interrupt vector address is obtained from an interrupt controller or vector table
4. **Handler Execution**: Control transfers to the ISR, which executes device-specific operations
5. **State Restoration and Return**: Original processor state is restored, and execution resumes from the interrupted point

## Classification of Interrupts

### Hardware Interrupts (Asynchronous)

Generated by external devices via dedicated interrupt request (IRQ) lines:

- **I/O Completion Interrupts**: Disk, network interface, keyboard, mouse
- **Timer Interrupts**: System clock, programmable interval timers
- **Hardware Failure Interrupts**: Memory parity errors, bus errors, power supply warnings

### Software Interrupts (Synchronous/Traps)

Explicitly triggered by program execution:

- **System Calls**: Operating system service invocations (e.g., `INT 0x80` on x86, `ECALL` on RISC-V)
- **Exceptions**: Divide-by-zero, page faults, undefined instructions, overflow
- **Debugging Traps**: Software breakpoints (`INT 3` on x86)

### Maskable versus Non-Maskable Interrupts

The classification into maskable and non-maskable interrupts is fundamental to interrupt theory:

**Maskable Interrupts (MI):**

- Can be selectively enabled or disabled by software
- Controlled via the Interrupt Flag (IF) in the Processor Status Word
- Normal I/O device interrupts belong to this category
- Mathematical representation: `MI = {i | enable(i, t) ∈ {0, 1}, where t is time}`

**Non-Maskable Interrupts (NMI):**

- Cannot be disabled under any circumstances
- Reserved for critical events requiring guaranteed service
- Examples: power failure, hardware failure, machine check
- Guaranteed latency independent of software state

## The Interrupt Flag Mechanism

### Formal Specification

Most processors implement an **Interrupt Flag (IF)** within the Processor Status Word (PSW) or Control and Status Register (CSR):

```
IF = 1 ⟹ Interrupts are enabled (processor recognizes maskable interrupt requests)
IF = 0 ⟹ Interrupts are disabled (maskable interrupts are ignored)
```

The interrupt enable state can be modeled as a binary semaphore `S_interrupt`:

```
P(S_interrupt): if S_interrupt > 0 then S_interrupt := S_interrupt - 1
 else block current process

V(S_interrupt): if processes blocked on S_interrupt then wake one
 else S_interrupt := S_interrupt + 1
```

### Architectural Implementations

**x86 Architecture:**

```assembly
CLI ; Clear Interrupt Flag (IF ← 0)
STI ; Set Interrupt Flag (IF ← 1)
```

**ARM Architecture (ARMv7 and later):**

```assembly
CPSID I ; Disable Interrupts (set PRIMASK)
CPSIE I ; Enable Interrupts (clear PRIMASK)
```

**8085 Microprocessor:**

```assembly
DI ; Disable All Interrupts
EI ; Enable Interrupts
```

**RISC-V (Machine Mode):**

```assembly
csrr t0, mstatus ; Read machine status register
ori t0, t0, 0x8 ; Set MIE bit (Machine Interrupt Enable)
csrw mstatus, t0 ; Write back to enable interrupts
```

## Interrupt Controllers: Hardware Implementation

### Programmable Interrupt Controller (PIC)

In multi-interrupt systems, a **PIC** (such as the Intel 8259A) manages multiple interrupt sources:

- **Interrupt Request Register (IRR)**: Stores pending interrupt requests
- **In-Service Register (ISR)**: Tracks interrupts currently being serviced
- **Interrupt Mask Register (IMR)**: selectively enables/disables individual IRQ lines

The interrupt priority is determined by cascade ordering and rotating priority schemes.

### Advanced PIC (APIC)

Modern systems employ the **APIC** architecture supporting:

- Multiple processors with local APICs
- Interrupt routing via I/O APIC
- Configurable priority and affinity
- Message Signaled Interrupts (MSI)

## Critical Sections and Atomicity Guarantees

### Formal Problem Statement

A **critical section** is a code segment that accesses shared resources and must execute atomically—meaning no other process or interrupt handler may observe or modify the shared state during execution. Formally:

```
Critical Section Property:
∀ i, j, t₁, t₂: if i ≠ j and t₁, t₂ ∈ CS_i ∩ CS_j then t₁ = t₂ (no temporal overlap)
```

### Race Condition Demonstration

Consider two execution contexts (main program and ISR) accessing shared variables:

```
Initial State: A = 5, B = 5

Main Thread: ISR:
A = A + 1 → A = 6 A = A * 2 → A = 12
B = B + 1 → B = 6 B = B * 2 → B = 10

If interrupted after A assignment:
A becomes 6, then ISR: A=12, B=10
Resume: B=6 (WRONG: A≠B, invariant violated)
```

**Invariant Violation**: The relationship `A = B` that held initially is violated.

### Solution via Interrupt Disabling

```assembly
; Enter Critical Section
DI ; IF ← 0 (disable all maskable interrupts)

LDA A ; Load shared variable A
INC ; A ← A + 1
STA A ; Store updated A
STA B ; B ← A (maintain invariant)

; Exit Critical Section
EI ; IF ← 1 (re-enable interrupts)
```

**Proof of Correctness**:

1. Before `DI`: Let state be S₀ with invariant I
2. During disabled region: No interrupt can occur → state transitions only from main code
3. After `EI`: State is S₁ where I holds
4. Therefore, critical section executes atomically with respect to interrupt handlers ∎

### Nested Critical Sections

When critical sections may be nested, simple enable/disable is insufficient:

```c
void outer_critical() {
 int saved_state = save_and_disable_interrupts(); // Save current IF
 // ... outer critical section ...

 inner_critical(); // May also modify interrupt state

 restore_interrupts(saved_state); // Restore original state
}

void inner_critical() {
 int saved = save_and_disable_interrupts();
 // ... inner critical section ...
 restore_interrupts(saved);
}
```

## Interrupt Latency Analysis

### Definitions

- **Interrupt Latency**: Time between interrupt assertion and start of ISR execution
- **Interrupt Response Time**: Time from interrupt to handler completion
- **Worst-Case Latency**: Maximum latency under all possible system conditions

### Factors Affecting Latency

1. **Current Instruction Completion**: Long-running instructions (e.g., division, string operations)
2. **Interrupt Disable Duration**: Time spent in critical sections
3. **Priority Inversion**: Higher priority task waiting for lower priority
4. **Interrupt Controller Queuing**: Multiple pending interrupts

**Maximum Latency Formula**:

```
L_max = L_instruction + L_disable + L_controller + L_priority
```

Where:

- L_instruction = longest instruction execution time
- L_disable = maximum time interrupts disabled
- L_controller = PIC/APIC response time
- L_priority = priority resolution overhead

## Rationale for Disabling Interrupts

### 1. Data Structure Modification

When updating kernel data structures shared with ISRs (queues, buffers, linked lists), interruption between read-modify-write sequences causes corruption.

### 2. Hardware Initialization

During device initialization, spurious interrupts must be suppressed until the interrupt handler is properly installed.

### 3. Time-Critical Operations

Real-time systems require bounded execution times that cannot be guaranteed with asynchronous preemption.

### 4. Nested Interrupt Control

Preventing lower-priority interrupts from interrupting higher-priority handlers maintains execution order.

## Performance Implications and Trade-offs

### System Responsiveness Degradation

Prolonged interrupt disable intervals cause:

- **Input Latency**: User input processing delays
- **Timer Skew**: Missed scheduler ticks, watchdog timeouts
- **Buffer Overflow**: Serial/network receive buffers filling
- **Priority Inversion**: High-priority tasks blocked by low-priority execution

### Quantitative Analysis

For a system with:

- Interrupt rate: λ interrupts/second
- Average interrupt disable time: T_disable seconds

**Maximum Sustainable Duty Cycle**:

```
D_max = (1/λ - T_disable) / (1/λ) = 1 - λ × T_disable
```

Example: If T_disable = 1ms and λ = 1000 interrupts/sec, then:

```
D_max = 1 - 1000 × 0.001 = 0% (system deadlocked)
```

## Best Practices for Interrupt Management

### Principle 1: Minimize Critical Section Duration

```c
// OPTIMAL: Pre-compute outside critical section
complex_result = compute_expensive_function(data); // Non-critical
disable_interrupts();
shared_variable = complex_result; // Only atomic operation
enable_interrupts();
```

### Principle 2: Prefer Finer-Grained Synchronization

Modern systems provide:

- **Spinlocks**: For multiprocessor critical sections
- **Interrupt Masks**: Selective disable of specific interrupt sources
- **Priority Mechanisms**: Allow higher-priority interrupts during lower-priority critical sections

### Principle 3: Bounded Disable Time

Real-time systems must guarantee:

```
∀ t: disable_duration(t) ≤ D_max (system-specified bound)
```

### Principle 4: Measurement and Monitoring

- Profile interrupt disable intervals during development
- Configure watchdog timers with appropriate margins
- Monitor system responsiveness metrics

## Summary

Interrupt enable and disable mechanisms constitute fundamental synchronization primitives in computer systems. The interrupt flag provides atomic control over interrupt recognition, enabling atomic execution of critical sections. However, this approach introduces latency and blocks all interrupt handling, necessitating careful design to minimize disable intervals and maintain system responsiveness. Advanced techniques including interrupt priority schemes, interrupt controllers, and finer-grained synchronization mechanisms provide more sophisticated solutions for complex multi-device systems.
