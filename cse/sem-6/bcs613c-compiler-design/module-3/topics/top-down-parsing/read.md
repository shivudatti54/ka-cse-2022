# Top-Down Parsing

## Introduction

Top-down parsing is a fundamental parsing technique in compiler design that builds the parse tree for an input string starting from the root (start symbol) and working downwards towards the leaves (terminals). This approach begins with the start symbol and applies production rules to match the input, making it one of the most intuitive parsing methods. Top-down parsers are particularly significant because they form the basis for predictive parsing, which is efficient and widely used in compilers for programming languages.

In the context of compiler construction, parsing is the phase that analyzes the token stream produced by the lexical analyzer and determines whether the tokens can be generated by the grammar of the programming language. Top-down parsing techniques, particularly LL(1) parsing, are favored in many compiler implementations due to their simplicity, efficiency, and the ability to implement them using recursive descent or table-driven approaches. Understanding top-down parsing is essential for CSE students as it forms a critical component of compiler design and is frequently asked in university examinations.

## Key Concepts

### 3.1 Basics of Top-Down Parsing

Top-down parsing constructs a parse tree where the root is the start symbol of the grammar. At each step, the parser expands a non-terminal on the leftmost derivation tree using a production rule, attempting to match the input symbols. The process continues until either the parse tree is complete (all input consumed) or an error is detected.

The two main approaches to top-down parsing are:

- **Recursive Descent Parsing**: Uses a set of recursive procedures, one for each non-terminal
- **LL(1) Parsing**: Uses a predictive parsing table and a stack-based approach

### 3.2 Problems with Naive Top-Down Parsing

**Left Recursion**: A grammar is left-recursive if it contains a production of the form A → Aα. This causes infinite recursion in recursive descent parsers. For example:

```
E → E + T | T
```

This left recursion must be eliminated before parsing can proceed.

**Left Factoring**: When two productions for the same non-terminal share a common prefix, the grammar needs to be factored to allow predictive parsing. For example:

```
stmt → if expr then stmt else stmt
stmt → if expr then stmt
```

Both start with "if expr then stmt", so left factoring is applied:

```
stmt → if expr then stmt rest
rest → else stmt | ε
```

### 3.3 Elimination of Left Recursion

Immediate left recursion in productions of the form A → Aα | β can be eliminated using the following transformation:

Original: A → Aα₁ | Aα₂ | ... | Aαₘ | β₁ | β₂ | ... | βₙ

Transformed:

- A → β₁A' | β₂A' | ... | βₙA'
- A' → α₁A' | α₂A' | ... | αₘA' | ε

**Algorithm for eliminating left recursion**:

1. For each non-terminal A in the grammar:

- Group productions: A → Aα₁ | Aα₂ | ... | β₁ | β₂ | ...
- Replace with: A → β₁A' | β₂A' | ... and A' → α₁A' | α₂A' | ... | ε

### 3.4 Left Factoring

Left factoring transforms a grammar to make it suitable for predictive parsing by factoring out common prefixes.

**Algorithm**:

1. For each non-terminal A with productions A → αβ₁ | αβ₂ | ... | αβₘ | γ:

- Replace with: A → αA' | γ
- Where: A' → β₁ | β₂ | ... | βₘ

### 3.5 FIRST and FOLLOW Sets

These sets are crucial for building LL(1) parsing tables.

**FIRST(α)**: The set of terminals that can appear as the first symbol of strings derived from α.

**Rules for computing FIRST sets**:

1. If X is a terminal, FIRST(X) = {X}
2. If X is a non-terminal and X → ε is a production, add ε to FIRST(X)
3. For production X → Y₁Y₂...Yₖ:

- Add FIRST(Y₁) - {ε} to FIRST(X)
- If ε ∈ FIRST(Y₁), add FIRST(Y₂) - {ε}
- Continue until either ε is not in FIRST(Yᵢ) or i = k

**FOLLOW(A)**: The set of terminals that can appear immediately after A in some sentential form.

**Rules for computing FOLLOW sets**:

1. Add $ (end-of-input) to FOLLOW(S), where S is the start symbol
2. For production A → αBβ, add FIRST(β) - {ε} to FOLLOW(B)
3. For production A → αB or A → αBβ where ε ∈ FIRST(β), add FOLLOW(A) to FOLLOW(B)

### 3.6 LL(1) Grammar

A grammar is LL(1) if:

1. No ambiguity exists
2. No left recursion
3. For each non-terminal A with multiple productions A → α|β:

- FIRST(α) and FIRST(β) are disjoint
- If ε ∈ FIRST(β), then FIRST(α) and FOLLOW(A) are disjoint

LL(1) stands for:

- **L**: Left-to-right scanning of input
- **L**: Leftmost derivation
- **1**: One input symbol of lookahead

### 3.7 LL(1) Parsing Table Construction

The parsing table M[A, a] for grammar G is constructed as follows:

For each production A → α:

1. For each terminal a in FIRST(α), add A → α to M[A, a]
2. If ε ∈ FIRST(α), for each terminal b in FOLLOW(A), add A → α to M[A, b]

If the parsing table has no multiple entries, the grammar is LL(1).

### 3.8 Predictive Parsing Algorithm

Using the LL(1) parsing table, the predictive parser works as follows:

1. Initialize stack with $ and start symbol S
2. Read first input token
3. While stack is not empty:

- Let X be top of stack, a be current input symbol
- If X is a terminal:
- If X = a, pop stack and advance input
- Otherwise, error
- If X is a non-terminal:
- Consult M[X, a]
- If M[X, a] has production X → α, replace X with α (reversed)
- Otherwise, error

## Examples

### Example 1: Computing FIRST Sets

**Grammar**:

```
E → T E'
E' → + T E' | ε
T → F T'
T' → * F T' | ε
F → ( E ) | id
```

**Solution**:

FIRST(F) = {(, id}
FIRST(T') = {\*, ε} (since T' → ε)
FIRST(T) = FIRST(F) = {(, id}
FIRST(E') = {+, ε}
FIRST(E) = FIRST(T) = {(, id}

### Example 2: Computing FOLLOW Sets

**Same Grammar as above**

**Solution**:

FOLLOW(E) = {$, )} (start symbol, appears after ))
FOLLOW(E') = FOLLOW(E) = {$, )}
FOLLOW(T) = FIRST(E') - {ε} = {+}
∪ FOLLOW(E) = {$, ), +}
FOLLOW(T') = FOLLOW(T) = {+, $, )}
FOLLOW(F) = FIRST(T') - {ε} = {_}
∪ FOLLOW(T) = {+, $, ), _}

### Example 3: Constructing LL(1) Parsing Table

**Grammar** (after left recursion elimination):

```
E → T E'
E' → + T E' | ε
T → F T'
T' → * F T' | ε
F → ( E ) | id
```

**Solution**:

| Non-Terminal | id    | (     | )    | +       | \*       | $    |
| ------------ | ----- | ----- | ---- | ------- | -------- | ---- |
| E            | E→TE' | E→TE' |      |         |          |      |
| E'           |       |       | E'→ε | E'→+TE' |          | E'→ε |
| T            | T→FT' | T→FT' |      |         |          |      |
| T'           |       |       | T'→ε | T'→ε    | T'→\*FT' | T'→ε |
| F            | F→id  | F→(E) |      |         |          |      |

### Example 4: Predictive Parsing Trace

**Input**: id + id \* id

Using the parsing table above:

| Step | Stack    | Input      | Action   |
| ---- | -------- | ---------- | -------- |
| 1    | $E       | id+id\*id$ | E→TE'    |
| 2    | $E'T     | id+id\*id$ | T→FT'    |
| 3    | $E'T'F   | id+id\*id$ | F→id     |
| 4    | $E'T'id  | +id\*id$   | Match id |
| 5    | $E'T'    | +id\*id$   | T'→ε     |
| 6    | $E'      | +id\*id$   | E'→+TE'  |
| 7    | $E'T+    | id\*id$    | Match +  |
| 8    | $E'T     | id\*id$    | T→FT'    |
| 9    | $E'T'F   | id\*id$    | F→id     |
| 10   | $E'T'id  | \*id$      | Match id |
| 11   | $E'T'    | \*id$      | T'→\*FT' |
| 12   | $E'T'F\* | id$        | Match \* |
| 13   | $E'T'F   | id$        | F→id     |
| 14   | $E'T'id  | $          | Match id |
| 15   | $E'T'    | $          | T'→ε     |
| 16   | $E'      | $          | E'→ε     |
| 17   | $        | $          | Accept   |

## Exam Tips

1. **Remember LL(1) acronym**: L (Left-to-right scan), L (Leftmost derivation), 1 (One symbol lookahead). This is frequently asked in university exams.

2. **LEFT RECURSION ELIMINATION**: Always check and remove left recursion first. The standard transformation converts A → Aα|β to A → βA' and A' → αA'|ε.

3. **LEFT FACTORING**: Essential when two productions start with the same terminal/non-terminal. Factor out the common prefix.

4. **FIRST and FOLLOW computation order**: Compute FIRST sets first, then FOLLOW sets. Remember ε handling is critical.

5. **FOLLOW rules**: The start symbol always has $ in its FOLLOW set. If A → αB, add FOLLOW(A) to FOLLOW(B).

6. **LL(1) condition**: For a grammar to be LL(1), FIRST sets of alternatives must be disjoint, and if ε in one alternative, it must not conflict with FOLLOW.

7. **Parsing table entries**: For production A → α: add to M[A, a] for all a in FIRST(α), and if ε in FIRST(α), add to M[A, b] for all b in FOLLOW(A).

8. **Predictive parsing algorithm**: Stack-based with top-down expansion. Always match terminals, expand non-terminals using table.

9. **Common sources of errors**: Left recursion causes infinite loops. Multiple entries in parsing table indicate grammar is not LL(1).
