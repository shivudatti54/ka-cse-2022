# Apache Hadoop

## Overview

Apache Hadoop is an open-source distributed computing framework designed to store and process massive datasets across clusters of commodity hardware. Developed by Doug Cutting and Mike Cafarella in 2005, Hadoop has become the cornerstone of big data processing, enabling organizations to handle petabytes of data efficiently and cost-effectively. In the context of Internet of Things (IoT), Hadoop provides the scalable infrastructure required to store and analyze the enormous volumes of data generated by billions of connected devices, including sensor readings, telemetry data, and log streams from industrial machines, smart homes, and wearable devices.

## What is Apache Hadoop?

Apache Hadoop is a framework that enables distributed processing of large data sets across clusters of computers using simple programming models. It follows a master-worker architecture where a single NameNode serves as the master coordinating operations across multiple DataNode workers. The framework is engineered to scale horizontally from single servers to thousands of machines, each offering local computation and storage capabilities, with automatic data distribution and fault tolerance built into its core design.

### Key Characteristics

**1. Distributed Storage and Processing:**
HDFS distributes data across multiple nodes in blocks (typically 128MB or 256MB), and MapReduce processes occur in parallel across the cluster. The framework automatically handles data distribution, replication, and load balancing without requiring explicit programmer intervention.

**2. Fault Tolerance:**
Hadoop implements automatic failure detection and recovery through data replication (default factor of 3). When a DataNode fails, the NameNode redirects reads to available replicas and orchestrates re-replication to maintain the desired replication factor. The system continues processing even when nodes fail, as tasks are rescheduled on healthy nodes.

**3. Scalability:**
Hadoop achieves horizontal scalability by adding more commodity nodes to the cluster. Performance scales approximately linearly with the number of nodes, and the system can handle datasets ranging from gigabytes to petabytes through simple cluster expansion.

**4. Cost-Effective:**
Hadoop runs on commodity hardware (low-cost, industry-standard servers) rather than expensive specialized hardware. As an open-source project with no licensing costs, it offers significantly lower Total Cost of Ownership (TCO) compared to traditional enterprise data warehousing solutions.

**5. Flexibility:**
Hadoop supports processing of structured, semi-structured, and unstructured data through its schema-on-read approach, allowing data to be stored without pre-defined schemas. This flexibility accommodates diverse data formats common in IoT ecosystems, including JSON from sensors, CSV logs, and binary telemetry streams.

### Core Components

**1. Hadoop Common:**
Common utilities and libraries that support other Hadoop modules, including Java ARchive files (JAR), scripts, and configuration files required for cluster operation.

**2. HDFS (Hadoop Distributed File System):**
HDFS is the primary storage layer, providing distributed file storage across multiple nodes with high-throughput data access. It is optimized for sequential read/write operations on large files, making it suitable for batch processing of IoT sensor data.

**3. YARN (Yet Another Resource Negotiator):**
YARN is the resource management layer that decouples resource management from application processing. It enables multiple processing engines (MapReduce, Spark, Tez) to run on a shared cluster, improving resource utilization and allowing simultaneous workloads.

**4. MapReduce:**
MapReduce is the original programming model for distributed batch processing, dividing tasks into map and reduce phases that execute in parallel across the cluster.

### NameNode (Master Node)

The NameNode serves as the master server that manages the file system namespace and regulates client access to files. It maintains two critical files: the fsimage (a checkpoint of the entire file system namespace) and the edit logs (a transaction log recording all modifications to the namespace).

**Responsibilities:**

1. **Metadata Management:**

- Maintains the complete file system tree with all directories and files
- Stores file-to-block mapping and block-to-DataNode locations
- Tracks file permissions, replication factor, and modification timestamps

2. **Namespace Operations:**

- Handles file and directory creation, deletion, renaming
- Manages quota limits and access permissions

3. **Block Management:**

- Determines block placement through rack-aware algorithms
- Monitors DataNode health via periodic heartbeats
- Manages block replication to maintain configured replication factor
- Handles under-replicated and over-replicated blocks

4. **Client Request Handling:**

- Processes file open/close operations
- Provides block location information to clients
- Coordinates the distributed read/write operations

**NameNode Metadata Structure:**

```
File System Namespace:
/user/iot/sensors/
 ├── temperature/
 │ ├── 2024/
 │ │ ├── jan.csv (256MB, blocks: blk_1001-blk_1002)
 │ │ └── feb.csv (128MB, blocks: blk_1003)
 │ └── 2023/
 └── pressure/
 └── annual.parquet (512MB, blocks: blk_2001-blk_2004)

Block Mapping Table:
blk_1001 → [DataNode1: rack1, DataNode3: rack2, DataNode5: rack2]
blk_1002 → [DataNode2: rack1, DataNode4: rack1, DataNode6: rack2]
blk_1003 → [DataNode1: rack1, DataNode3: rack2, DataNode7: rack3]
```

**High Availability:**
Hadoop 2.x introduces HA configurations with automatic failover using ZooKeeper. The Secondary NameNode periodically merges fsimage with edit logs to prevent unbounded growth of edit logs, though it does not provide real-time HA.

### DataNode (Worker Node)

DataNodes are worker nodes that store actual data blocks and serve read/write requests from clients.

**Responsibilities:**

1. **Block Storage:**

- Stores data blocks on local file system
- Default block size is 128MB (Hadoop 2.x) or 256MB (Hadoop 3.x)
- Maintains multiple replicas on different nodes

2. **Data Operations:**

- Serves read requests by locating and transmitting block data
- Processes write requests through the replication pipeline
- Creates, deletes, and replicates blocks as directed by NameNode

3. **Heartbeat and Block Reports:**

- Sends periodic heartbeats (every 3 seconds) to NameNode
- Reports block inventory to NameNode (every 6 hours by default)

### HDFS Write Pipeline (Detailed)

When a client writes data to HDFS, the following pipeline executes:

```
Step 1: Client requests file creation from NameNode
 ↓
Step 2: NameNode checks permissions, creates metadata entry
 Returns list of DataNodes for first block replication
 ↓
Step 3: Client writes first packet (64KB) to first DataNode
 ↓
Step 4: First DataNode pipelines data to second DataNode
 Second DataNode pipelines to third DataNode
 ↓
Step 5: Acknowledgment flows back through pipeline to client
 ↓
Step 6: Steps 3-5 repeat until all data written
 ↓
Step 7: Client closes file, NameNode commits final metadata
```

### Rack Awareness

HDFS uses rack awareness to optimize data placement for fault tolerance and network efficiency. The default block placement policy places one replica on the local node, a second in a different node within the same rack, and the third in a different rack. This strategy ensures data availability even if an entire rack fails while minimizing inter-rack network traffic.

**Example Calculation:**
For a 1TB sensor dataset with 256MB blocks and replication factor of 3:

- Number of blocks = (1TB × 1024³) / (256MB × 1024²) = 4096 blocks
- Total storage with replication = 1TB × 3 = 3TB
- Network transfers for single block read = 1 intra-rack + 2 inter-rack (worst case)

## YARN (Yet Another Resource Negotiator)

YARN provides resource management and job scheduling capabilities, enabling multiple applications to share a common Hadoop cluster.

### YARN Components

**1. ResourceManager:**
The ResourceManager is the master daemon that arbitrates cluster resources among competing applications. It has two main components: the Scheduler (Capacity Scheduler or Fair Scheduler) allocates resources, and the ApplicationsManager accepts job submissions and negotiates the first container for the ApplicationMaster.

**2. NodeManager:**
NodeManager is the worker daemon running on each DataNode. It monitors container resource usage (CPU, memory, disk, network), reports node health, and manages the lifecycle of containers allocated by the ResourceManager.

**3. ApplicationMaster:**
The ApplicationMaster runs within a container and is responsible for negotiating resources with the ResourceManager, coordinating task execution with NodeManagers, and handling application-specific failures.

**4. Container:**
A Container represents a collection of physical resources (CPU cores, RAM, disk) on a single node. It is the fundamental unit of resource allocation in YARN.

### YARN Job Submission Lifecycle

1. Client submits application to ResourceManager
2. ApplicationsManager negotiates container for ApplicationMaster
3. ApplicationMaster registers with ResourceManager
4. ApplicationMaster requests containers from ResourceManager
5. ResourceManager allocates containers; ApplicationMaster distributes work to NodeManagers
6. Containers execute tasks and report status to ApplicationMaster
7. ApplicationMaster monitors task completion and releases containers
8. ApplicationMaster unregisters and exits; client retrieves results

## MapReduce Programming Model

MapReduce is a programming paradigm for processing large datasets in parallel across distributed clusters.

### MapReduce Word Count Example

```java
// Map Function
public class TokenizerMapper extends Mapper<LongWritable, Text, Text, IntWritable> {
 private final static IntWritable one = new IntWritable(1);
 private Text word = new Text();

 public void map(LongWritable key, Text value, Context context) {
 StringTokenizer itr = new StringTokenizer(value.toString());
 while (itr.hasMoreTokens()) {
 word.set(itr.nextToken());
 context.write(word, one);
 }
 }
}

// Reduce Function
public class IntSumReducer extends Reducer<Text, IntWritable, Text, IntWritable> {
 private IntWritable result = new IntWritable();

 public void reduce(Text key, Iterable<IntWritable> values, Context context) {
 int sum = 0;
 for (IntWritable val : values) {
 sum += val.get();
 }
 result.set(sum);
 context.write(key, result);
 }
}
```

### MapReduce ExecutionInput Phases

```
 Files → Split → Map Phase → Shuffle & Sort → Reduce Phase → Output
 │ │ │
 [Key, Value] [Key, List[V]] [Key, Result]
```

**1. Map Phase:**
The Map function processes input key-value pairs and produces intermediate key-value pairs. For IoT sensor data, a map function might extract device IDs and aggregate sensor readings.

**2. Shuffle and Sort Phase:**
This critical phase transfers map outputs to reducers and sorts them by key. The shuffle is often the performance bottleneck in MapReduce jobs, involving data transfer across the network.

**3. Reduce Phase:**
The Reduce function merges all values associated with the same intermediate key, producing the final output.

### Numerical Problem: MapReduce Performance

**Problem:** Given a cluster of 10 nodes, each with 4 map slots, processing 1TB of data with 256MB blocks and a map task taking 30 seconds:

**Solution:**

- Number of map tasks = (1TB × 1024⁴) / (256MB × 1024²) = 4096 maps
- Map waves required = ceil(4096 / (10 × 4)) = 103 waves
- Minimum execution time = 103 × 30 = 3090 seconds (ignoring reduce)

### Use Case: Sensor Data Aggregation

A manufacturing facility with 10,000 temperature sensors, each generating readings every 5 seconds (approximately 17.28 million readings per day), can use Hadoop to:

- Store daily sensor logs in HDFS (approximately 50GB/day)
- Run daily batch jobs to compute hourly/daily statistics
- Identify anomalous temperature patterns using MapReduce
- Generate compliance reports using Hive queries

## Limitations of Hadoop

- **High-latency operations:** Not suitable for real-time processing (< seconds latency)
- **Small file problem:** Too many small files overwhelm NameNode memory
- **No true random access:** Sequential access pattern only
- **Complex programming:** MapReduce requires significant development effort

## Summary

Apache Hadoop provides a robust, scalable, and cost-effective framework for distributed storage and processing of massive datasets. Its core components—HDFS for storage, YARN for resource management, and MapReduce for processing—form the foundation of modern big data architectures. For IoT applications, Hadoop excels at batch analytics on historical sensor data but requires integration with real-time frameworks like Storm or Spark Streaming for low-latency requirements.
