# Waterfall, Agile, and Scrum Methodologies - Summary

## Key Definitions and Concepts
- **Waterfall**: Linear sequential development with distinct phases
- **Agile**: Iterative development through collaborative cross-functional teams
- **Scrum**: Framework using fixed-length iterations (sprints) with roles/events
- **Velocity**: Story points completed per sprint
- **Burn-down Chart**: Work remaining vs time visualization

## Important Formulas and Theorems
- **Brooks' Law**: Adding people to late projects makes them later (Waterfall)
- **Cycle Time** = End Date - Start Date (Agile)
- **Velocity** = Σ Story Points Completed / Number of Sprints
- **Burndown Rate** = (Total Work - Work Remaining)/Days

## Key Points
- Waterfall suits projects with stable requirements (e.g., spacecraft software)
- Agile's MVP (Minimum Viable Product) approach reduces time-to-market
- Scrum Master removes impediments; Product Owner manages backlog
- Daily Standups answer: What did I do? What will I do? Blockers?
- Retrospectives drive continuous improvement
- Product Backlog must be DEEP: Detailed, Emergent, Estimated, Prioritized
- Sprint Review demonstrates working software; Retrospective improves process

## Common Mistakes to Avoid
- Confusing Agile (philosophy) with Scrum (specific framework)
- Assuming Agile means no documentation (it values "working software over comprehensive documentation")
- Using Waterfall for AI/ML projects with evolving data models
- Neglecting Definition of Ready/Done in Scrum
- Estimating in hours instead of story points

## Revision Tips
1. Create comparison tables: Waterfall vs Agile vs Scrum
2. Practice drawing Scrum process diagrams with timeboxes
3. Memorize Agile Manifesto through acronym "IWAW" (Individuals, Working, Collaboration, Responding)
4. Solve velocity calculation problems (e.g., 3 sprints completed with 20, 18, 22 points → Avg velocity=20)
5. Study real-world failures: FBI's Virtual Case File (Waterfall) vs Healthcare.gov (Agile mismatch)
