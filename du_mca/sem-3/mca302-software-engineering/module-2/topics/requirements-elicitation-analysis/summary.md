# Requirements Elicitation and Analysis - Summary

## Key Definitions and Concepts
- **Elicitation**: Systematic discovery of stakeholder needs
- **SRS**: Software Requirements Specification (IEEE 830 standard)
- **Use Case**: Sequence of actions yielding measurable value
- **Non-functional Requirement**: Quality attributes like performance

## Important Formulas and Theorems
- **MoSCoW Formula**: R = M + S + C + W (Requirement set composition)
- **Fagan's Defect Rate**: D = (Number of defects)/(KLOC) × 1000
- **Traceability Completeness**: T = (Linked requirements)/(Total requirements) × 100

## Key Points
- 68% project failures originate from poor requirements (Standish Group)
- JAD sessions require trained facilitator and scribe
- Prototypes reduce requirement ambiguity by 40%
- Always validate against original business objectives
- Traceability matrices prevent scope creep
- Context diagrams establish system boundaries
- Indian IT Act 2000 impacts security requirements

## Common Mistakes to Avoid
- Confusing wants vs needs during interviews
- Neglecting regulatory stakeholders
- Overlooking data migration requirements
- Using technical jargon in user stories

## Revision Tips
1. Practice converting user stories to use cases
2. Memorize SRS section headers through acronym VIDECAP:
   - Validity, Implementability, Decidability, Completeness, etc.
3. Use color-coding in traceability matrices
4. Solve previous DU papers on DFD creation
