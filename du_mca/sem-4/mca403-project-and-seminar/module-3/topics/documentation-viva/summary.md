# Documentation & Viva - Summary

## Key Definitions and Concepts
- **Technical Debt**: Compromises in documentation causing future rework
- **Viva Voce**: Oral examination testing practical implementation knowledge
- **Sphinx**: Python documentation generator supporting multiple output formats
- **Swagger**: API documentation standard using OpenAPI specification

## Important Formulas and Theorems
- **Documentation ROI**: (Maintenance Cost Saved) / (Documentation Effort)
- **Brooks' Law**: Adding manpower to late software projects makes them later
- **Halstead Complexity**: η = η1 + η2 (Program vocabulary metrics)

## Key Points
- Maintain separate docs for developers, testers, and end-users
- Use UML 2.5 standards for system diagrams
- Record all design decisions with rationale
- Prepare for 3 types of viva questions: technical, analytical, situational
- Implement searchable documentation using Elasticsearch
- Include failure scenarios in test documentation
- Use JIRA integration for requirement traceability

## Common Mistakes to Avoid
- Writing monolithic documents without modular structure
- Using technical jargon in user manuals
- Neglecting to document error handling procedures
- Memorizing answers instead of understanding concepts
- Overlooking non-functional requirements in viva

## Revision Tips
1. Conduct mock vivas with time constraints
2. Use Grammarly for technical writing checks
3. Create documentation checklists:
   - [ ] Version history
   - [ ] Table of contents
   - [ ] Glossary of terms
4. Study past DU project reports from department archives
5. Practice explaining concepts to non-technical audiences
