# Signal Concepts in Unix/Linux

## Introduction to Signals

Signals are software interrupts delivered to a process to notify it of an event that occurred. They are a fundamental mechanism for inter-process communication (IPC) and process control in Unix/Linux systems. When a signal is sent to a process, the operating system interrupts the process's normal flow of execution to deliver the signal.

Signals can be generated by:

- The kernel (e.g., when a child process terminates)
- Another process (with appropriate permissions)
- The process itself (self-signaling)
- The terminal driver (e.g., when user presses Ctrl+C)

## Signal Types and Their Meanings

Unix systems define a standard set of signals, each with a unique number and symbolic name. Here are some commonly used signals:

| Signal Number | Signal Name | Default Action | Description                                |
| ------------- | ----------- | -------------- | ------------------------------------------ |
| 1             | SIGHUP      | Terminate      | Hangup detected on controlling terminal    |
| 2             | SIGINT      | Terminate      | Interrupt from keyboard (Ctrl+C)           |
| 3             | SIGQUIT     | Core dump      | Quit from keyboard (Ctrl+\)                |
| 9             | SIGKILL     | Terminate      | Kill signal (cannot be caught or ignored)  |
| 11            | SIGSEGV     | Core dump      | Invalid memory reference                   |
| 15            | SIGTERM     | Terminate      | Termination signal                         |
| 17            | SIGCHLD     | Ignore         | Child process stopped or terminated        |
| 18            | SIGCONT     | Continue       | Continue if stopped                        |
| 19            | SIGSTOP     | Stop           | Stop process (cannot be caught or ignored) |
| 20            | SIGTSTP     | Stop           | Stop typed at terminal (Ctrl+Z)            |

## Signal Lifecycle

The lifecycle of a signal involves three main stages:

1. **Generation**: A signal is generated by an event
2. **Delivery**: The signal is delivered to the target process
3. **Handling**: The process handles the signal according to its disposition

```
+-------------+      +-------------+      +-------------+
|  Signal     |      |  Signal     |      |  Signal     |
| Generation  |----->|  Delivery   |----->|  Handling   |
| (Event)     |      | (OS Action) |      | (Process)   |
+-------------+      +-------------+      +-------------+
```

## Signal Disposition

Each signal has a disposition, which determines how the process responds when the signal is delivered:

1. **Default Action**: The default behavior defined by the system
2. **Ignore**: The signal is discarded and has no effect
3. **Custom Handler**: A user-defined function is called to handle the signal

## Signal Handling Functions

### The signal() Function

The `signal()` function is the traditional way to set a signal handler:

```c
#include <signal.h>

void (*signal(int sig, void (*func)(int)))(int);
```

Example:

```c
#include <stdio.h>
#include <signal.h>
#include <unistd.h>

void sigint_handler(int sig) {
    printf("Caught SIGINT (Signal #%d)\n", sig);
}

int main() {
    // Set SIGINT handler
    if (signal(SIGINT, sigint_handler) == SIG_ERR) {
        perror("signal");
        return 1;
    }

    printf("Waiting for SIGINT (Ctrl+C)...\n");
    while(1) {
        pause(); // Wait for any signal
    }

    return 0;
}
```

### The sigaction() Function

The `sigaction()` function provides more control and is the preferred method:

```c
#include <signal.h>

int sigaction(int sig, const struct sigaction *act,
              struct sigaction *oldact);
```

Example:

```c
#include <stdio.h>
#include <signal.h>
#include <unistd.h>

void sigint_handler(int sig) {
    printf("Caught SIGINT (Signal #%d)\n", sig);
}

int main() {
    struct sigaction sa;

    sa.sa_handler = sigint_handler;
    sigemptyset(&sa.sa_mask);
    sa.sa_flags = 0;

    if (sigaction(SIGINT, &sa, NULL) == -1) {
        perror("sigaction");
        return 1;
    }

    printf("Waiting for SIGINT (Ctrl+C)...\n");
    while(1) {
        pause();
    }

    return 0;
}
```

## Signal Sets and Operations

Signal sets (sigset_t) are used to represent groups of signals. Common operations include:

```c
#include <signal.h>

int sigemptyset(sigset_t *set);          // Initialize empty set
int sigfillset(sigset_t *set);           // Initialize full set
int sigaddset(sigset_t *set, int signo); // Add signal to set
int sigdelset(sigset_t *set, int signo); // Remove signal from set
int sigismember(const sigset_t *set, int signo); // Test if signal in set
```

## Signal Mask and Process Signal Context

Each process has a signal mask that defines which signals are currently blocked from delivery. Blocked signals remain pending until unblocked.

```c
#include <signal.h>

int sigprocmask(int how, const sigset_t *set, sigset_t *oldset);
```

The `how` parameter can be:

- SIG_BLOCK: Add signals to current mask
- SIG_UNBLOCK: Remove signals from current mask
- SIG_SETMASK: Replace current mask with new mask

## Signal Generation Functions

### kill() and raise()

```c
#include <sys/types.h>
#include <signal.h>

int kill(pid_t pid, int sig); // Send signal to process
int raise(int sig);           // Send signal to self
```

### alarm() and pause()

```c
#include <unistd.h>

unsigned int alarm(unsigned int seconds); // Set alarm clock
int pause(void);                         // Wait for any signal
```

Example using alarm():

```c
#include <stdio.h>
#include <signal.h>
#include <unistd.h>

void alarm_handler(int sig) {
    printf("Alarm triggered!\n");
}

int main() {
    signal(SIGALRM, alarm_handler);

    printf("Setting alarm for 3 seconds...\n");
    alarm(3);

    printf("Waiting for alarm...\n");
    pause();

    printf("Program continues after alarm\n");
    return 0;
}
```

## Signal Safety Considerations

When writing signal handlers, it's crucial to use only async-signal-safe functions. These are functions that can be safely called from within a signal handler. Most standard I/O functions are NOT async-signal-safe.

**Async-signal-safe functions** include:

- write()
- read() (with caution)
- \_exit()
- abort()
- signal()
- sigaction()
- kill()
- Most functions that manipulate signal sets

## Race Conditions and Signal Handling

Race conditions can occur when signals are handled asynchronously. To avoid them:

1. Use `sigaction()` instead of `signal()`
2. Block signals during critical sections
3. Use atomic operations when possible
4. Consider using self-pipes or signalfd() for more controlled signal handling

## Signal Inheritance Across fork() and exec()

- After `fork()`: Child inherits parent's signal dispositions and mask
- After `exec()`: Signal dispositions reset to default for most signals, but signal mask is preserved
- Signals with handlers remain caught across `exec()` if set with `sigaction()` and SA_SIGINFO flag

## Real-World Signal Usage Patterns

### Graceful Shutdown

```c
#include <stdio.h>
#include <signal.h>
#include <unistd.h>
#include <stdlib.h>

volatile sig_atomic_t shutdown_flag = 0;

void graceful_shutdown(int sig) {
    shutdown_flag = 1;
}

int main() {
    struct sigaction sa;

    sa.sa_handler = graceful_shutdown;
    sigemptyset(&sa.sa_mask);
    sa.sa_flags = 0;

    sigaction(SIGINT, &sa, NULL);
    sigaction(SIGTERM, &sa, NULL);

    while(!shutdown_flag) {
        printf("Working...\n");
        sleep(1);
    }

    printf("Initiating graceful shutdown...\n");
    // Cleanup code here
    printf("Shutdown complete\n");

    return 0;
}
```

### Signal-Driven I/O

Signals can be used for asynchronous I/O notification, particularly with SIGIO for socket programming.

## Exam Tips

1. **Remember key signal numbers**: SIGINT(2), SIGKILL(9), SIGTERM(15), SIGSTOP(19)
2. **Know the difference**: SIGKILL and SIGSTOP cannot be caught, blocked, or ignored
3. **Signal safety**: Always use async-signal-safe functions in handlers
4. **Race conditions**: Be aware of potential race conditions in signal handling
5. **Process relationships**: Understand how signals behave across fork() and exec()
6. **Default actions**: Memorize default actions for common signals
7. **sigaction vs signal**: Prefer sigaction() for better control and portability
8. **Signal masking**: Understand how to use sigprocmask() to block signals during critical sections
