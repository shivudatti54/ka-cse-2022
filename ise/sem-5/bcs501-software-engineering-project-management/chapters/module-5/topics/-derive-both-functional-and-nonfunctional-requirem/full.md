# Deriving Functional and Non-Functional Requirements

## Introduction

In software engineering, requirements engineering is a critical phase of the software development life cycle. It involves identifying, analyzing, and documenting the needs of stakeholders to create a software product that meets their needs. Requirements can be broadly categorized into two types: functional and non-functional. In this article, we will delve into the world of requirements engineering and explore how to derive both functional and non-functional requirements from a case study.

## Functional Requirements

Functional requirements describe the what of a software system, i.e., what the system is expected to do. They are specific, measurable, achievable, relevant, and time-bound (SMART) statements that describe the functionality of the system. Here are some examples of functional requirements:

- The user should be able to login to the system using their username and password.
- The system should be able to process payments for online transactions.
- The system should generate a receipt for the customer after a successful transaction.

To derive functional requirements, the following steps can be taken:

1. **Interviews and Surveys**: Conduct interviews with stakeholders, including users, customers, and end-users, to gather information about their needs and expectations.
2. **User Stories**: Use user stories to describe the functionality of the system from the user's perspective. For example, "As a user, I want to be able to view my order history so that I can keep track of my previous purchases."
3. **Use Cases**: Use use cases to describe the interactions between the system and its users. For example, "The system shall allow a user to place an order by entering their details and payment information."
4. **System Requirements Specification (SRS)**: Create a SRS that documents all the functional requirements of the system.

## Non-Functional Requirements

Non-functional requirements describe the how of a software system, i.e., how the system should behave or perform. They are often related to quality attributes such as performance, security, usability, and maintainability. Here are some examples of non-functional requirements:

- The system should respond to user input within 2 seconds.
- The system should be secure and protect sensitive customer data.
- The system should be easy to use and navigate.

To derive non-functional requirements, the following steps can be taken:

1. **Performance Tests**: Conduct performance tests to determine the system's response time, throughput, and resource utilization.
2. **Security Analysis**: Conduct a security analysis to identify potential security threats and vulnerabilities.
3. **Usability Studies**: Conduct usability studies to evaluate the system's user interface and user experience.
4. **Architecture and Design**: Consider the system's architecture and design when deriving non-functional requirements.

## Case Study: Online Shopping Platform

Let's consider a case study of an online shopping platform. The platform allows users to browse and purchase products from various retailers. The platform's functional and non-functional requirements can be derived as follows:

**Functional Requirements**

- The user should be able to browse products by category, brand, and price.
- The system should allow users to add products to their cart and checkout.
- The system should generate a receipt for the customer after a successful transaction.

**Non-Functional Requirements**

- The system should respond to user input within 2 seconds.
- The system should be secure and protect sensitive customer data.
- The system should be easy to use and navigate.

Example of Functional Requirements:

```markdown
**User Interface Requirements**

- The system should display product information, including price, description, and images.
- The system should allow users to filter products by category, brand, and price.
- The system should provide a secure checkout process to protect sensitive customer data.

**System Requirements**

- The system should be able to process payments for online transactions.
- The system should generate a receipt for the customer after a successful transaction.
- The system should be able to handle a large number of concurrent users.

**Performance Requirements**

- The system should respond to user input within 2 seconds.
- The system should be able to handle a large number of concurrent users.
- The system should have a high availability rate of 99.9%.
```

Example of Non-Functional Requirements:

```markdown
**Security Requirements**

- The system should be secure and protect sensitive customer data.
- The system should validate user input to prevent SQL injection and cross-site scripting (XSS) attacks.
- The system should implement encryption to protect data in transit and at rest.

**Usability Requirements**

- The system should be easy to use and navigate.
- The system should provide clear and concise product information.
- The system should provide a secure checkout process to protect sensitive customer data.

**Architecture Requirements**

- The system should be scalable to handle a large number of concurrent users.
- The system should be able to handle a large number of concurrent transactions.
- The system should provide a high availability rate of 99.9%.
```

## Conclusion

In conclusion, deriving functional and non-functional requirements is an essential phase of the software development life cycle. By identifying and documenting the needs of stakeholders, developers can create a software product that meets their needs and expectations. The case study of the online shopping platform demonstrates how to derive functional and non-functional requirements using various techniques and tools.
