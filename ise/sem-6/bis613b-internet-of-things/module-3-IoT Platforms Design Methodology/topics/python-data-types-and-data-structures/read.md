## Introduction

In IoT Platform Design Methodology, the effective handling of data is paramount. IoT systems generate massive volumes of heterogeneous data from diverse sensors and devices. This data must be efficiently structured, processed, stored, and analyzed to extract meaningful insights. Understanding the fundamental data types and appropriate data structures is crucial for designing scalable, efficient, and reliable IoT platforms.

## Fundamental IoT Data Types

IoT data can be categorized based on its source, structure, and temporal characteristics.

### 1. Based on Source and Nature

- **Sensor Data:** Raw measurements from physical sensors (e.g., temperature, humidity, pressure, light intensity, GPS coordinates). Typically numeric and time-series in nature.
  - Example: `25.7` (Temperature in °C), `48` (Humidity in %RH)
- **Actuator Status Data:** Data representing the state or command sent to an actuator (e.g., `ON`, `OFF`, `OPEN`, `CLOSE`, `45` degrees for a servo motor). Often boolean, enum, or numeric.
- **Device Metadata:** Information about the IoT device itself (e.g., device ID, firmware version, location, operational status). Usually structured as key-value pairs or JSON objects.
  - Example: `{"deviceId": "sensor-node-12", "firmware": "v2.1.4", "location": "Building-A-3rd-Floor"}`
- **Multimedia Data:** Data from cameras or microphones (e.g., images, audio clips, video streams). This is unstructured binary data (blobs) and requires significant storage and bandwidth.

### 2. Based on Structure

- **Structured Data:** Data adhering to a predefined schema or model, often stored in relational databases. Easy to query and analyze.
  - Example: A table with columns `timestamp`, `sensor_id`, `temperature`, `humidity`.
- **Semi-structured Data:** Data that does not have a fixed schema but contains tags or markers to separate semantic elements. Common formats are JSON, XML, and CSV. Highly flexible and widely used in IoT.
  - Example: `{"ts": "2023-10-27T10:15:30Z", "sensor_id": "temp_sensor_1", "value": 22.5, "unit": "°C"}`
- **Unstructured Data:** Data without a predefined model or organization. Most multimedia data falls into this category. Requires specialized processing and storage solutions.

### 3. Based on Temporal Characteristics

- **Streaming Data:** Continuous, real-time flow of data points generated by sensors. Requires processing as it arrives.
  - Example: A live feed of temperature readings every second.
- **Time-Series Data:** A sequence of data points indexed (or listed or graphed) in time order. This is the most common data type in IoT.
  - Example: Historical temperature data stored for a week, with a reading every minute.

## Key Data Structures for IoT

Choosing the right data structure is critical for performance in constrained IoT environments (limited memory, processing power) and for efficient cloud-based analytics.

### 1. Arrays and Buffers

Simple, contiguous blocks of memory used for storing sequences of elements. Essential for handling sensor readings in embedded C/C++ programs on microcontrollers.

```c
// C example for storing 10 temperature readings
float temperature_readings[10];
int reading_index = 0;
```

**Use Case:** Temporary storage of sensor data on a device before transmission.

### 2. Structs (C/C++) and Classes (Python/Java)

Composite data types that group together variables of different types under a single name. Perfect for representing a complete "sensor reading" or "device state" packet.

```c
// C struct example for a sensor data packet
typedef struct {
    uint16_t device_id;
    float temperature;
    float humidity;
    uint32_t timestamp;
} sensor_packet_t;
```

```python
# Python class example
class SensorReading:
    def __init__(self, device_id, temperature, humidity, timestamp):
        self.device_id = device_id
        self.temperature = temperature
        self.humidity = humidity
        self.timestamp = timestamp
```

**Use Case:** Packaging data for transmission over a network (e.g., via MQTT).

### 3. Queues

A First-In-First-Out (FIFO) data structure. Crucial for decoupling tasks in an IoT system, such as buffering data between a sensor reading task and a network transmission task.

```
[Read Task] --> [Queue: | Data3 | Data2 | Data1 | ] --> [Transmit Task]
      (Producer)          (FIFO Buffer)            (Consumer)
```

**Use Case:** Managing data flow in real-time operating systems (RTOS) on edge devices to prevent data loss.

### 4. JSON (JavaScript Object Notation)

_The de facto standard for IoT data interchange._ A lightweight, text-based, language-independent semi-structured data format. It's human-readable and easy for machines to parse and generate.

```json
{
  "device": "SN-102",
  "timestamp": "2023-10-27T14:32:01Z",
  "readings": {
    "temperature": 27.4,
    "humidity": 62,
    "pressure": 1013.2
  },
  "status": "OK"
}
```

**Use Case:** The primary format for payloads in IoT communication protocols like MQTT and HTTP REST APIs. It is the lingua franca for data moving from the edge to the cloud platform.

### 5. Protocol Buffers (Protobuf)

A Google-developed alternative to JSON. It is a binary format, meaning it is not human-readable but is significantly more efficient in terms of payload size and parsing speed.

```
# .proto schema definition
message SensorData {
  required string device_id = 1;
  required float temperature = 2;
  optional float humidity = 3;
  required uint64 timestamp = 4;
}
```

**Use Case:** Ideal for high-frequency communication between devices or in bandwidth-constrained scenarios like LPWAN (LoRaWAN, NB-IoT).

### 6. CSV (Comma-Separated Values)

A simple file format used to store tabular data (numbers and text) in plain text. Each line is a data record, and each record consists of one or more fields, separated by commas.

```
timestamp,device_id,temperature,humidity
2023-10-27T10:00:00Z,SN-101,25.7,48.0
2023-10-27T10:01:00Z,SN-101,25.8,47.8
```

**Use Case:** Exporting historical time-series data from IoT platforms for analysis in tools like Excel or Python Pandas.

### 7. Avro

A row-based remote procedure call and data serialization framework. It uses a schema (defined in JSON) for serialization, and the data is stored in a compact binary format.

```json
// Avro Schema
{
  "type": "record",
  "name": "WeatherReading",
  "fields": [
    { "name": "station_id", "type": "string" },
    { "name": "temperature", "type": "float" },
    { "name": "humidity", "type": "int" }
  ]
}
```

**Use Case:** Efficient serialization for big data frameworks like Apache Hadoop and Apache Kafka, which are often part of larger IoT analytics pipelines.

## Comparison of Data Interchange Formats

| Feature           | JSON                   | Protocol Buffers (Protobuf)                    | CSV              | Avro                              |
| ----------------- | ---------------------- | ---------------------------------------------- | ---------------- | --------------------------------- |
| **Format**        | Text (Human-readable)  | Binary (Not readable)                          | Text (Readable)  | Binary (Not readable)             |
| **Schema**        | Schema-less (implicit) | Requires external `.proto` file                | No formal schema | Schema in JSON                    |
| **Size**          | Larger (verbose)       | **Very compact**                               | Compact          | Compact                           |
| **Parsing Speed** | Slower (needs parsing) | **Very Fast**                                  | Fast (simple)    | Fast                              |
| **Flexibility**   | High (easy to modify)  | Low (schema changes need care)                 | Medium           | Medium (schema evolution)         |
| **Primary Use**   | IoT APIs, Web Comm.    | High-performance/bandwidth-constrained devices | Data Export      | Big Data Pipelines (Hadoop/Kafka) |

## Case Study: Weather Monitoring System

Consider the weather monitoring case study from the syllabus. A typical data packet sent from a sensor node to the cloud might be structured as follows in JSON:

```json
{
  "system_id": "weather-station-alpha",
  "location": { "lat": 12.9716, "lon": 77.5946, "alt": 920 },
  "sensor_readings": [
    { "sensor_type": "temperature", "value": 23.1, "unit": "celsius" },
    { "sensor_type": "humidity", "value": 65, "unit": "percent" },
    { "sensor_type": "pressure", "value": 1009, "unit": "hPa" },
    { "sensor_type": "wind_speed", "value": 12, "unit": "kmph" }
  ],
  "timestamp": "2023-10-27T05:30:00Z",
  "battery_voltage": 4.1
}
```

This semi-structured JSON object is efficient for the cloud platform to parse, validate, and insert into various databases for historical analysis and real-time dashboards.

## Exam Tips

- **Remember Key Data Types:** Be able to list and describe the primary IoT data types (Sensor, Actuator, Metadata, Multimedia) and their characteristics (Structured, Semi-structured, Unstructured, Streaming, Time-Series).
- **Compare JSON and Protobuf:** Understand the trade-offs between these two dominant formats. JSON is flexible and universal, while Protobuf is efficient and fast. Be prepared to justify the use of one over the other based on a scenario (e.g., bandwidth constraints -> Protobuf).
- **Link Structures to Layers:** Associate data structures with the correct layer in the IoT architecture (e.g., Arrays/Structs on the device, JSON for transmission, Avro/Parquet in big data storage).
- **Visualize the Flow:** Draw a simple diagram of the data journey from sensor to cloud, annotating it with the typical data structures used at each step. This demonstrates a holistic understanding.
- **Python Relevance:** Since the module includes "Python for IoT," remember that dictionaries and lists are the fundamental structures in Python for creating and manipulating these data types (especially JSON).
