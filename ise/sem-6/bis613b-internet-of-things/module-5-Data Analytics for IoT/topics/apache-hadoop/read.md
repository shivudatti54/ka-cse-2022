## Overview

Apache Hadoop is an open-source distributed computing framework designed to store and process massive datasets across clusters of commodity hardware. Developed by Doug Cutting and Mike Cafarella in 2005, Hadoop has become the cornerstone of big data processing, enabling organizations to handle petabytes of data efficiently and cost-effectively. In the IoT context, Hadoop provides the infrastructure needed to store and analyze the enormous volumes of data generated by billions of connected devices.

## What is Apache Hadoop?

Apache Hadoop is a framework that allows for the distributed processing of large data sets across clusters of computers using simple programming models. It is designed to scale up from single servers to thousands of machines, each offering local computation and storage.

### Key Characteristics

**1. Distributed Storage and Processing:**

- Data is distributed across multiple nodes
- Processing occurs in parallel across the cluster
- Automatic data distribution and replication

**2. Fault Tolerance:**

- Automatic detection and recovery from failures
- Data replication (default 3 copies)
- Continues processing even if nodes fail

**3. Scalability:**

- Horizontal scaling by adding more nodes
- Linear performance improvement
- Handles petabyte-scale datasets

**4. Cost-Effective:**

- Uses commodity hardware
- Open-source with no licensing costs
- Lower TCO compared to traditional systems

**5. Flexibility:**

- Processes structured, semi-structured, and unstructured data
- Schema-on-read approach
- Multiple processing paradigms

## Hadoop Distributed File System (HDFS)

HDFS is the primary storage system used by Hadoop applications. It is specifically designed to store very large files and provide high-throughput access to application data.

### HDFS Architecture

```
┌──────────────────────────────────────────────────────┐
│                    Client Application                │
└────────────────┬─────────────────────────────────────┘
                 │
                 ▼
         ┌───────────────┐
         │   NameNode    │  (Master)
         │  - Metadata   │
         │  - File System│
         │    Namespace  │
         └───────┬───────┘
                 │
     ┌───────────┼───────────┐
     │           │           │
     ▼           ▼           ▼
┌─────────┐ ┌─────────┐ ┌─────────┐
│DataNode1│ │DataNode2│ │DataNode3│  (Workers)
│- Blocks │ │- Blocks │ │- Blocks │
│- Storage│ │- Storage│ │- Storage│
└─────────┘ └─────────┘ └─────────┘
    │           │           │
    └───────────┴───────────┘
       Block Replication
```

### NameNode (Master Node)

The NameNode is the master server that manages the file system namespace and regulates access to files by clients.

**Responsibilities:**

1. **Metadata Management:**
   - Stores file system tree
   - Maintains file-to-block mapping
   - Tracks block locations
   - Stores file permissions and attributes

2. **Namespace Operations:**
   - Opening, closing, and renaming files
   - Directory operations (create, delete)
   - File operations (create, delete, rename)

3. **Block Management:**
   - Assigns blocks to DataNodes
   - Monitors DataNode health
   - Manages block replication
   - Handles under-replicated and over-replicated blocks

4. **Client Request Handling:**
   - Processes read/write requests
   - Provides block locations to clients
   - Coordinates data operations

**NameNode Metadata:**

```
File System Tree:
/user/data/sensors/
  ├─ temperature.txt (128MB, blocks: blk_1, blk_2)
  ├─ pressure.txt (64MB, blocks: blk_3)
  └─ humidity.txt (200MB, blocks: blk_4, blk_5, blk_6)

Block Mapping:
blk_1: [DataNode1, DataNode3, DataNode5]
blk_2: [DataNode2, DataNode4, DataNode6]
blk_3: [DataNode1, DataNode2, DataNode3]
```

**High Availability:**

- Secondary NameNode for metadata backups
- NameNode Federation for scalability
- Automatic failover with HA configuration

### DataNode (Worker Node)

DataNodes are the worker nodes that store the actual data blocks and serve read/write requests from clients.

**Responsibilities:**

1. **Block Storage:**
   - Stores data blocks on local disk
   - Each block typically 128MB or 256MB
   - Multiple replicas across different nodes

2. **Data Operations:**
   - Serves read requests from clients
   - Processes write requests
   - Performs block creation and deletion

3. **Heartbeat and Block Reports:**
   - Sends heartbeat to NameNode (every 3 seconds)
   - Reports block information periodically
   - Confirms node health and availability

4. **Replication Pipeline:**
   - Participates in block replication
   - Forwards data to other DataNodes
   - Maintains replication factor

**DataNode Operation:**

```
Client Write Process:
1. Client contacts NameNode for write
2. NameNode provides DataNode list
3. Client writes to first DataNode
4. First DataNode replicates to second
5. Second DataNode replicates to third
6. Acknowledgments sent back through pipeline
7. Client confirms write completion to NameNode
```

### HDFS Features

**1. Block-Based Storage:**

- Files divided into blocks (default 128MB)
- Each block stored independently
- Blocks distributed across cluster

**2. Replication:**

- Default replication factor: 3
- Ensures fault tolerance
- Automatic re-replication on failure

**3. Rack Awareness:**

- Considers network topology
- Optimizes data placement
- Typical placement: 2 replicas on same rack, 1 on different rack

**4. Write-Once-Read-Many:**

- Optimized for streaming reads
- Append-only writes (no random writes)
- High throughput for sequential access

**5. Large File Support:**

- Optimized for files from gigabytes to terabytes
- Not suitable for millions of small files
- High aggregate data bandwidth

## YARN (Yet Another Resource Negotiator)

YARN is Hadoop's resource management layer that enables multiple data processing engines to run on a single Hadoop cluster.

### YARN Architecture

```
┌──────────────────────────────────────────────────┐
│           Client Applications                    │
└────────────────┬─────────────────────────────────┘
                 │
         ┌───────▼────────┐
         │ ResourceManager│  (Master)
         │ - Scheduler    │
         │ - AppManager   │
         └───────┬────────┘
                 │
     ┌───────────┼───────────┐
     │           │           │
     ▼           ▼           ▼
┌─────────┐ ┌─────────┐ ┌─────────┐
│NodeMgr 1│ │NodeMgr 2│ │NodeMgr 3│  (Workers)
│         │ │         │ │         │
│┌───────┐│ │┌───────┐│ │┌───────┐│
││AppMstr││ ││ Task  ││ ││ Task  ││
│├───────┤│ │├───────┤│ │├───────┤│
││ Task  ││ ││ Task  ││ ││ Task  ││
│└───────┘│ │└───────┘│ │└───────┘│
└─────────┘ └─────────┘ └─────────┘
```

### YARN Components

**1. ResourceManager (RM):**

- Global resource scheduler
- Arbitrates resources among applications
- Components:
  - **Scheduler:** Allocates resources based on policies
  - **ApplicationsManager:** Accepts job submissions, negotiates resources

**2. NodeManager (NM):**

- Per-machine agent
- Manages resources on individual nodes
- Monitors resource usage
- Reports to ResourceManager

**3. ApplicationMaster (AM):**

- Per-application component
- Negotiates resources from ResourceManager
- Works with NodeManager to execute tasks
- Monitors application progress

**4. Container:**

- Collection of resources (CPU, memory, disk)
- Basic unit of allocation
- Runs application-specific processes

### YARN Workflow

```
Step 1: Client submits application to ResourceManager

Step 2: ResourceManager allocates container for ApplicationMaster

Step 3: ApplicationMaster registers with ResourceManager

Step 4: ApplicationMaster requests containers for tasks

Step 5: ResourceManager allocates containers on NodeManagers

Step 6: ApplicationMaster launches tasks in containers

Step 7: Tasks execute and report progress

Step 8: ApplicationMaster reports completion to ResourceManager

Step 9: ApplicationMaster unregisters and shuts down
```

## Hadoop for IoT Data Processing

### Why Hadoop for IoT?

IoT applications generate massive volumes of diverse data that require scalable storage and processing solutions. Hadoop addresses these needs:

**1. Volume Handling:**

- Stores petabytes of sensor data
- Scales horizontally with data growth
- Cost-effective storage solution

**2. Variety Support:**

- Handles structured sensor readings
- Processes semi-structured logs
- Manages unstructured images/videos from IoT devices

**3. Batch Analytics:**

- Historical trend analysis
- Periodic reporting
- Data aggregation and summarization

**4. Integration:**

- Connects with various data sources
- Supports multiple processing frameworks
- Enables comprehensive analytics pipelines

### IoT Data Storage in HDFS

**Typical IoT Data Structure:**

```
/iot-data/
├── /sensors/
│   ├── /temperature/
│   │   ├── 2024-01-01.json
│   │   ├── 2024-01-02.json
│   │   └── ...
│   ├── /pressure/
│   └── /humidity/
├── /devices/
│   ├── /smart-meters/
│   ├── /connected-vehicles/
│   └── /industrial-machines/
└── /logs/
    ├── /device-logs/
    ├── /network-logs/
    └── /error-logs/
```

**Data Ingestion Methods:**

1. **Batch Ingestion:**
   - Periodic uploads from edge devices
   - Scheduled data transfers
   - ETL pipelines with Sqoop

2. **Stream Ingestion:**
   - Real-time data collection with Flume
   - Message queues with Kafka
   - Direct streaming to HDFS

3. **Hybrid Approach:**
   - Hot data in fast storage
   - Cold data in HDFS
   - Tiered storage strategy

### IoT Use Cases with Hadoop

**1. Smart Manufacturing:**

```
Data Sources: Machine sensors, quality sensors, environmental sensors
Data Volume: 1-10 TB per day
Processing: Daily production reports, quality analysis, predictive maintenance
Hadoop Usage: Store historical sensor data, batch analysis of production metrics
```

**2. Smart City:**

```
Data Sources: Traffic cameras, air quality sensors, smart meters

Processing: Traffic pattern analysis, pollution mapping, energy consumption trends

```

## Advantages of Hadoop

### 1. Scalability

- Linear scaling with cluster size
- Handles petabyte-scale data
- Easy to add new nodes

### 2. Fault Tolerance

- Automatic failure detection
- Data replication ensures availability
- No single point of failure

### 3. Cost-Effectiveness

- Commodity hardware usage
- Open-source software
- Pay-as-you-grow model

### 4. Flexibility

- Schema-on-read approach
- Multiple data formats support
- Various processing paradigms

### 5. Data Locality

- Moves computation to data
- Reduces network traffic
- Improves processing speed

### 6. Ecosystem Integration

- Rich set of tools and frameworks
- Integration with various technologies
- Active community support

## Limitations of Hadoop

### 1. High Latency

**Issue:**

- Batch processing oriented
- Not suitable for real-time analytics
- Startup overhead for small jobs

**Impact on IoT:**

- Cannot handle urgent alerts
- Delayed insights for time-critical decisions
- Need complementary real-time systems

### 2. Small Files Problem

**Issue:**

- NameNode memory limitation
- Each file consumes ~150 bytes of NameNode memory
- Inefficient for millions of small files

**Impact on IoT:**

- IoT devices often generate small frequent messages
- Need file aggregation strategies
- May require pre-processing or buffering

### 3. No Real-Time Processing

**Issue:**

- MapReduce has high latency
- Disk-based processing
- Multiple read/write cycles

**Solution:**

- Use Apache Storm or Spark for real-time needs
- Hybrid architecture (Lambda architecture)
- Stream processing for hot path

### 4. Programming Complexity

**Issue:**

- MapReduce programming requires expertise
- Debugging distributed applications is challenging
- Steep learning curve

**Mitigation:**

- Use higher-level abstractions (Pig, Hive, Spark)
- Pre-built libraries and templates
- Better development tools

### 5. Security Challenges

**Issue:**

- Basic security in early versions
- Authentication and authorization complexity
- Data encryption overhead

**Solutions:**

- Kerberos authentication
- Apache Ranger for authorization
- Encryption at rest and in transit

## Hadoop Configuration for IoT Workloads

### Hardware Recommendations

**NameNode Server:**

- 64-128 GB RAM (depends on number of files)
- Multiple fast disks for metadata
- Redundant power and networking
- High availability setup for production

**DataNode Servers:**

- 64-256 GB RAM
- 12-24 SATA drives (4-8 TB each)
- 10 Gigabit Ethernet
- Cost-optimized commodity hardware

**Cluster Sizing:**

- Start with 5-10 nodes for pilot
- Scale to 100+ nodes for production IoT
- Plan for 30-50% annual data growth

### HDFS Configuration for IoT

```xml
<!-- Block size for IoT data -->
<property>
  <name>dfs.blocksize</name>
  <value>134217728</value> <!-- 128MB -->
</property>

<!-- Replication factor -->
<property>
  <name>dfs.replication</name>
  <value>3</value>
</property>

<!-- NameNode handler threads for concurrent operations -->
<property>
  <name>dfs.namenode.handler.count</name>
  <value>100</value>
</property>
```

### Performance Optimization

**1. Compression:**

- Reduce storage requirements
- Common codecs: Snappy, LZO, Gzip
- Balance compression ratio vs. CPU overhead

**2. Data Partitioning:**

- Organize by date/device/location
- Enables partition pruning
- Improves query performance

**3. File Format:**

- Use columnar formats (Parquet, ORC)
- Better compression and query performance
- Efficient for analytical queries

## Hadoop vs. Traditional Databases

| Aspect           | Hadoop                         | Traditional RDBMS           |
| ---------------- | ------------------------------ | --------------------------- |
| **Data Model**   | Schema-on-read                 | Schema-on-write             |
| **Scalability**  | Horizontal (add nodes)         | Vertical (upgrade hardware) |
| **Cost**         | Low (commodity hardware)       | High (specialized hardware) |
| **Processing**   | Batch, high latency            | Real-time queries           |
| **Data Types**   | Structured, semi, unstructured | Structured only             |
| **Transactions** | Not supported                  | ACID compliance             |
| **Best For**     | Large-scale analytics          | Transactional operations    |
| **IoT Fit**      | Historical analysis            | Real-time device control    |

## Summary

Apache Hadoop is a powerful distributed computing framework that provides the foundation for big data processing in IoT applications. Its core components—HDFS for distributed storage and YARN for resource management—enable scalable, fault-tolerant processing of massive IoT datasets.

HDFS architecture with NameNode managing metadata and DataNodes storing actual data blocks ensures reliable storage with automatic replication. YARN provides flexible resource management, allowing multiple processing frameworks to coexist on the same cluster.

While Hadoop excels at batch processing and cost-effective storage of petabyte-scale IoT data, it has limitations including high latency, small files problem, and lack of real-time processing capabilities. These limitations necessitate complementary technologies like Apache Storm for real-time analytics.

For IoT applications, Hadoop serves as the backbone for storing historical sensor data, performing periodic analytics, and generating long-term insights that inform business decisions and optimization strategies.

1. **Hadoop Core Components:** HDFS (storage), YARN (resource management), MapReduce (processing), Common (utilities)

3. **NameNode Functions:** Metadata management, namespace operations, block management, client request handling

4. **DataNode Functions:** Block storage, data operations, heartbeat/block reports, replication

5. **YARN Components:** ResourceManager (global scheduler), NodeManager (per-node agent), ApplicationMaster (per-app), Container (resource unit)

6. **Hadoop Advantages:** Scalability, fault tolerance, cost-effectiveness, flexibility, data locality

7. **Hadoop Limitations:** High latency, small files problem, no real-time processing, programming complexity

8. **IoT Use Cases:** Smart manufacturing (production analytics), smart cities (traffic analysis), healthcare (patient history)

9. **HDFS Features:** Block-based storage (128MB), replication (3 copies), rack awareness, write-once-read-many, large file support

10. **Hadoop vs. RDBMS:** Schema-on-read vs. schema-on-write, horizontal vs. vertical scaling, batch vs. real-time processing
