# **Component Hierarchy Revision Notes**

## **What is Component Hierarchy?**

- A hierarchical structure of components in software development, where each component is a self-contained part of a larger system
- Used in software architecture and design patterns

## **Key Concepts:**

- **Component**: A self-contained part of a larger system, with its own interface and behavior
- **Composition**: A relationship between components, where one component is contained within another
- **Container**: A component that holds other components

## **Types of Component Hierarchy:**

- **Flat Hierarchy**: A simple, linear structure with no nesting
- **Hierarchical Structure**: A structure with nesting, where components are contained within others

## **Important Formulas and Definitions:**

- **Component-Container Relationship**: A = C ∩ I, where A is the component, C is the container, and I is the interface
- **Component-Component Relationship**: A = C ∩ I ∩ B, where A is the component, C is the container, I is the interface, and B is the behavior

## **Theorems:**

- **Component Hierarchy Theorem**: A well-designed component hierarchy is a key factor in maintaining maintainable and scalable software systems

## **Key Design Patterns:**

- **Singleton Pattern**: A design pattern that ensures only one instance of a component is created
- **Factory Pattern**: A design pattern that provides a way to create components without specifying the exact type of component

## **Revision Tips:**

- Practice drawing component hierarchies to understand the relationships between components
- Review the key concepts and formulas to ensure a strong grasp of component hierarchy
- Use real-world examples to illustrate the importance of component hierarchy in software development.
