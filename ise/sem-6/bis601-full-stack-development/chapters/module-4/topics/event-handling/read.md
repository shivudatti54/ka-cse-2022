# Event Handling in Swing

## 1. Introduction to Event Handling

In Graphical User Interface (GUI) programming, an **event** is an occurrence of a user action, such as a mouse click, a key press, or a window closing. **Event Handling** is the mechanism that controls the program's flow in response to these events. It is the cornerstone of creating interactive, responsive applications.

Swing, like its predecessor AWT, uses the **Delegation Event Model**, a modern, efficient approach to handling events. This model is based on the concept of **sources** and **listeners**.

## 2. The Delegation Event Model

The Delegation Event Model defines a standard, logical way for a _source_ (a GUI component like a button) to generate events and send them to _listener_ objects that are registered to receive them. This model separates the UI component (the source) from the logic that defines its behavior (the listener), promoting cleaner, more modular code.

**Key Participants:**

- **Event Source:** The object that originates or _fires_ the event (e.g., a `JButton`).
- **Event Listener:** An object that is notified when an event occurs. It must be registered with a source to receive notifications about specific types of events.
- **Event Object:** An object that describes the event. It contains all pertinent data about the event (e.g., the x,y coordinates of a mouse click, the character of a key pressed). It is passed to the listener's callback method.

The process can be visualized as follows:

```
+----------------+    generates    +---------------+
|   Event Source |----------------->|  Event Object |
|  (e.g., Button)|                  +---------------+
+----------------+                         |
        |                                  |
        | registers & calls               v
        |                         +----------------+
        +-----------------------> | Event Listener |
                                   | (implements    |
                                   |  an interface) |
                                   +----------------+
```

## 3. Event Classes and Listener Interfaces

Events in Java are represented by classes, and the reactions to events are defined in interfaces.

### Common Event Classes (in `java.awt.event` package)

- `ActionEvent`: Generated by components like `JButton`, `JTextField` (on pressing 'Enter'), `JMenuItem`.
- `ItemEvent`: Generated when an item is selected or deselected, such as in a `JCheckBox` or `JComboBox`.
- `MouseEvent`: Generated by mouse actions (click, press, release, enter, exit, drag, move).
- `KeyEvent`: Generated by keyboard actions (key pressed, released, typed).
- `WindowEvent`: Generated by window operations (opening, closing, activating, deactivating).
- `FocusEvent`: Generated when a component gains or loses keyboard focus.

### Common Listener Interfaces (in `java.awt.event` package)

Each event type has a corresponding listener interface that declares one or more methods (callbacks).

| Listener Interface    | Key Methods                                                                   | Typical Event Source      |
| --------------------- | ----------------------------------------------------------------------------- | ------------------------- |
| `ActionListener`      | `actionPerformed(ActionEvent e)`                                              | `JButton`, `JTextField`   |
| `ItemListener`        | `itemStateChanged(ItemEvent e)`                                               | `JCheckBox`, `JComboBox`  |
| `MouseListener`       | `mouseClicked(MouseEvent e)`, `mousePressed(e)`, `mouseReleased(e)`, etc.     | Any `Component`           |
| `MouseMotionListener` | `mouseDragged(MouseEvent e)`, `mouseMoved(MouseEvent e)`                      | Any `Component`           |
| `KeyListener`         | `keyPressed(KeyEvent e)`, `keyReleased(e)`, `keyTyped(e)`                     | Any `Component`           |
| `WindowListener`      | `windowOpened(WindowEvent e)`, `windowClosing(e)`, `windowActivated(e)`, etc. | `JFrame`                  |
| `FocusListener`       | `focusGained(FocusEvent e)`, `focusLost(FocusEvent e)`                        | Any focusable `Component` |

## 4. Registering Listeners with Sources

For a listener to receive events from a source, it must be registered with that source. Each component has methods to add listeners, typically named `addXXXListener(XXXListener l)`.

**Example: Registering an `ActionListener` with a `JButton`**

```java
JButton myButton = new JButton("Click Me!");
// Create a listener (using an anonymous inner class)
myButton.addActionListener(new ActionListener() {
    @Override
    public void actionPerformed(ActionEvent e) {
        // This code runs when the button is clicked
        System.out.println("Button was clicked!");
    }
});
```

## 5. Implementing Listeners: Common Approaches

There are three primary ways to implement a listener.

### 1. Anonymous Inner Classes

This is a very common approach, especially for simple, one-off listeners. It keeps the listener code directly where the component is set up.

```java
button.addActionListener(new ActionListener() {
    public void actionPerformed(ActionEvent e) {
        label.setText("Anonymous Class handled this.");
    }
});
```

**Pros:** Concise, keeps related code together.
**Cons:** Can clutter the code if the listener logic is very long.

### 2. Named Inner Classes

Useful when the listener logic is more complex or if the same listener needs to be added to multiple sources.

```java
public class MyFrame extends JFrame {
    // ...
    public MyFrame() {
        button.addActionListener(new MyButtonListener());
    }
    // Inner class
    class MyButtonListener implements ActionListener {
        public void actionPerformed(ActionEvent e) {
            label.setText("Named Inner Class handled this.");
        }
    }
}
```

**Pros:** Better organization for complex logic, reusable within the outer class.
**Cons:** Still tied to the instance of the outer class.

### 3. Having the Container Class Implement the Interface

The container (e.g., your main `JFrame` class) can itself implement the listener interface.

```java
public class MyFrame extends JFrame implements ActionListener {
    JButton button;
    JLabel label;

    public MyFrame() {
        button = new JButton("Click Me");
        label = new JLabel("Waiting...");
        // 'this' refers to the MyFrame instance, which is an ActionListener
        button.addActionListener(this);
        add(button);
        add(label);
    }

    // Must implement the method from the interface
    @Override
    public void actionPerformed(ActionEvent e) {
        if(e.getSource() == button) { // Check which source fired the event
            label.setText("Container class handled this.");
        }
    }
}
```

**Pros:** Centralizes event handling code.
**Cons:** Can lead to a large `if-else` chain if handling many different components. Violates single-responsibility principle for complex UIs.

## 6. Adapter Classes

Listener interfaces like `MouseListener` and `WindowListener` have multiple methods. Often, you only care about one of them (e.g., `mouseClicked`). Implementing the interface forces you to provide empty implementations for all methods, which is cumbersome.

Java provides **Adapter Classes** (e.g., `MouseAdapter`, `WindowAdapter`). These are abstract classes that provide empty, do-nothing implementations for all methods in their corresponding interface. You can then extend the adapter class and override _only_ the method you need.

**Without Adapter:**

```java
component.addMouseListener(new MouseListener() {
    @Override
    public void mouseClicked(MouseEvent e) { /* My logic */ }
    @Override
    public void mousePressed(MouseEvent e) {} // Empty body
    @Override
    public void mouseReleased(MouseEvent e) {} // Empty body
    @Override
    public void mouseEntered(MouseEvent e) {} // Empty body
    @Override
    public void mouseExited(MouseEvent e) {} // Empty body
});
```

**With Adapter:**

```java
component.addMouseListener(new MouseAdapter() {
    @Override // Only override the method you need!
    public void mouseClicked(MouseEvent e) {
        // My logic
    }
});
```

This leads to much cleaner and more focused code.

## 7. Practical Example: A Simple Calculator

Let's combine these concepts into a simple application with two text fields, operation buttons, and a result label.

```java
import javax.swing.*;
import java.awt.*;
import java.awt.event.*;

public class SimpleCalculator extends JFrame implements ActionListener {
    private JTextField tf1, tf2;
    private JLabel resultLabel;
    private JButton addBtn, subBtn;

    public SimpleCalculator() {
        setLayout(new FlowLayout());
        tf1 = new JTextField(10);
        tf2 = new JTextField(10);
        resultLabel = new JLabel("Result: ");
        addBtn = new JButton("+");
        subBtn = new JButton("-");

        // Register listeners
        addBtn.addActionListener(this);
        subBtn.addActionListener(this);

        // Add components to frame
        add(tf1); add(tf2); add(addBtn); add(subBtn); add(resultLabel);
        setDefaultCloseOperation(JFrame.EXIT_ON_CLOSE);
        pack();
        setVisible(true);
    }

    @Override
    public void actionPerformed(ActionEvent e) {
        try {
            double num1 = Double.parseDouble(tf1.getText());
            double num2 = Double.parseDouble(tf2.getText());
            double result = 0;

            if (e.getSource() == addBtn) {
                result = num1 + num2;
            } else if (e.getSource() == subBtn) {
                result = num1 - num2;
            }
            resultLabel.setText("Result: " + result);
        } catch (NumberFormatException ex) {
            resultLabel.setText("Error: Invalid Input");
        }
    }

    public static void main(String[] args) {
        SwingUtilities.invokeLater(() -> new SimpleCalculator());
    }
}
```

## 8. Using Lambda Expressions (Java 8+)

For listener interfaces with a **single abstract method** (SAM interfaces), such as `ActionListener`, you can use lambda expressions for a very concise syntax.

```java
JButton button = new JButton("Click with Lambda");
button.addActionListener( (e) -> {
    System.out.println("Lambda handled the click!");
    // 'e' is the ActionEvent object
});
```

This is the modern and preferred way to implement simple listeners, as it reduces boilerplate code significantly.

## 9. Exam Tips and Common Pitfalls

- **Understand the Model:** Be able to explain the Delegation Event Model and its components (Source, Listener, Event Object) clearly.
- **Know the Methods:** Memorize the key method(s) in common listener interfaces (e.g., `actionPerformed` for `ActionListener`).
- **Registration is Key:** Remember that a listener will **not** receive events unless it is explicitly registered using `addXXXListener()`.
- **Adapter Advantage:** Explain why adapter classes (like `MouseAdapter`) are useful. They save you from writing empty method stubs.
- **Identify the Source:** In a method like `actionPerformed`, use `e.getSource()` to determine which component fired the event, especially when one listener handles multiple sources.
- **Swing Threading:** Always make sure to create and manipulate Swing components on the **Event Dispatch Thread (EDT)** using `SwingUtilities.invokeLater()`. Forgetting this can lead to thread-safety issues and unpredictable GUI behavior.
- **Lambda Readiness:** For modern Java exams, be prepared to read and write event handling code using lambda expressions.
