# Introduction to Parsing

## What is Parsing?

Parsing is the process of analyzing a string of symbols, either in natural language or computer languages, conforming to the rules of a formal grammar. In compiler design, parsing is the second phase of compilation, following lexical analysis. The parser takes the stream of tokens produced by the lexical analyzer and verifies whether this stream can be generated by the grammar of the source language.

The primary goals of parsing are:

- To determine if the input program is syntactically correct
- To build a parse tree or abstract syntax tree that represents the syntactic structure of the program
- To report syntax errors clearly and accurately

## The Role of Parser in Compilation Process

```
          Source Program
                ↓
        Lexical Analyzer (Scanner)
                ↓
           Stream of Tokens
                ↓
               Parser
                ↓
         Syntax Tree (AST)
                ↓
        Semantic Analyzer
                ↓
   Intermediate Code Generator
```

The parser sits between the lexical analyzer and the semantic analyzer. It consumes tokens and produces a hierarchical structure that represents the grammatical relationships between those tokens.

## Context-Free Grammars (CFG)

Parsing relies on context-free grammars, which formally define the syntax of programming languages. A CFG consists of:

- **Terminals**: Basic symbols (tokens) from which strings are formed
- **Nonterminals**: Syntactic variables that represent language constructs
- **Productions**: Rules that define how nonterminals can be rewritten
- **Start symbol**: A special nonterminal that begins all derivations

**Example CFG for arithmetic expressions:**

```
E → E + E
E → E * E
E → ( E )
E → id
```

## Parse Trees and Abstract Syntax Trees

**Parse Tree** (Concrete Syntax Tree): A tree representation of the syntactic structure of a program that shows all the details of the grammar rules used.

**Abstract Syntax Tree (AST)**: A condensed form of parse tree that abstracts away syntactic details while preserving the essential structure.

**Example:** For expression `a + b * c`

```
Parse Tree:            Abstract Syntax Tree:
      E                      +
     /|\                    / \
    E + E                 a     *
   /   / \                    / \
  a   E * E                 b   c
     /   \
    b     c
```

## Types of Parsers

Parsers can be classified based on their approach to building the parse tree:

### 1. Top-Down Parsers

- Begin with the start symbol and apply productions to reach the input string
- Build parse tree from root to leaves
- Examples: Recursive Descent, LL parsers

```
Working: Start symbol → Apply productions → Match input
```

### 2. Bottom-Up Parsers

- Begin with the input string and apply productions in reverse
- Build parse tree from leaves to root
- Examples: LR parsers, SLR, LALR

```
Working: Input string → Reduce using productions → Reach start symbol
```

## Comparison of Parser Types

| Aspect               | Top-Down Parsers                         | Bottom-Up Parsers                 |
| -------------------- | ---------------------------------------- | --------------------------------- |
| Direction            | Root to leaves                           | Leaves to root                    |
| Approach             | Predictive                               | Reductive                         |
| Examples             | Recursive Descent, LL                    | LR, SLR, LALR                     |
| Grammar Restrictions | No left recursion, Left factoring needed | More powerful, handle wider range |
| Implementation       | Easier to implement by hand              | Typically generated by tools      |
| Lookahead            | Limited (LL(1), LL(k))                   | More powerful (LR(1), LALR(1))    |

## Syntax-Directed Translation

Syntax-directed translation (SDT) is a method of compiling where the translation is driven by the syntax of the source language. It associates:

- **Semantic actions** with grammar productions
- **Attributes** with grammar symbols
- **Translation rules** that define how to compute attribute values

**Example SDT for expression evaluation:**

```
Production: E → E₁ + T
Semantic action: E.val = E₁.val + T.val
```

## Parsing Algorithms

### Recursive Descent Parsing

A top-down parsing technique that uses a set of recursive procedures to process the input. Each nonterminal has a corresponding procedure.

```pseudocode
procedure E()
    T()
    while lookahead = '+' or lookahead = '-'
        match(lookahead)
        T()

procedure T()
    F()
    while lookahead = '*' or lookahead = '/'
        match(lookahead)
        F()

procedure F()
    if lookahead = '('
        match('(')
        E()
        match(')')
    else if lookahead is id
        match(id)
    else
        error()
```

### Shift-Reduce Parsing

A bottom-up parsing approach that uses a stack and input buffer. The parser shifts tokens onto the stack until a handle (right side of production) appears, then reduces it to the left side.

**Operations:**

- **Shift**: Move input symbol to stack
- **Reduce**: Replace handle on stack with nonterminal
- **Accept**: Successful completion
- **Error**: Syntax error detected

## Parser Generators

Tools that automatically create parsers from grammar specifications:

- **Yacc (Yet Another Compiler Compiler)**: Creates LALR parsers
- **Bison**: GNU version of Yacc
- **ANTLR**: Creates LL(\*) parsers
- **JavaCC**: Java parser generator

## Common Parsing Problems

### 1. Ambiguous Grammars

A grammar is ambiguous if it can generate multiple parse trees for the same input string.

**Example:** The grammar `E → E + E | E * E | id` is ambiguous for `id + id * id`

**Solution:** Define operator precedence and associativity rules:

```
E → E + T | E - T | T
T → T * F | T / F | F
F → ( E ) | id
```

### 2. Left Recursion

A production where the leftmost symbol on the right side is the same as the nonterminal on the left side: `A → Aα`

**Problem:** Causes infinite recursion in top-down parsers
**Solution:** Eliminate left recursion by rewriting grammar

### 3. Left Factoring

When multiple productions for the same nonterminal start with the same symbols.

**Problem:** Causes backtracking in predictive parsers
**Solution:** Factor out common prefixes

## Error Handling in Parsing

Parsers must detect and report syntax errors effectively. Common strategies include:

- **Panic-mode recovery**: Skip tokens until a synchronizing token is found
- **Phrase-level recovery**: Local correction of erroneous input
- **Error productions**: Include error-handling in grammar
- **Global correction**: Find minimal sequence of changes

## Practical Example: Simple Expression Parser

Let's implement a simple recursive descent parser for arithmetic expressions:

**Grammar:**

```
E → T E'
E' → + T E' | ε
T → F T'
T' → * F T' | ε
F → ( E ) | id
```

**Implementation sketch:**

```python
def E():
    T()
    E_prime()

def E_prime():
    if lookahead == '+':
        match('+')
        T()
        E_prime()
    # else: epsilon production - do nothing

def T():
    F()
    T_prime()

def T_prime():
    if lookahead == '*':
        match('*')
        F()
        T_prime()
    # else: epsilon production - do nothing

def F():
    if lookahead == '(':
        match('(')
        E()
        match(')')
    elif lookahead == 'id':
        match('id')
    else:
        error()
```

## Exam Tips

1. **Understand the difference** between top-down and bottom-up parsing approaches - this is a common exam question.
2. **Practice constructing parse trees** for given expressions and grammars.
3. **Memorize the solutions** for common parsing problems: left recursion elimination and left factoring.
4. **Be able to compute FIRST and FOLLOW sets** - these are essential for predictive parsing.
5. **Understand how parser generators work** and what types of grammars they can handle.
6. **Practice tracing** through parsing algorithms step by step.
7. **Know the differences** between various bottom-up parsing techniques (SLR, LR, LALR).
8. **Remember that ambiguous grammars** cannot be used directly - you must know how to resolve ambiguity.
