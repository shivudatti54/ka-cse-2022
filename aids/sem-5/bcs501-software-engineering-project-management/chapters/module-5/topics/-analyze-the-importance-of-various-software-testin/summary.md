# Software Testing Methods and Agile Methodology

### Overview

This module covers the importance of various software testing methods and agile methodology in software engineering and project management.

### Software Testing Methods

- **Black Box Testing**: Tests software without knowing its internal workings, focusing on functionality and user interface.
  - Example: Exploratory Testing
- **White Box Testing**: Tests software by examining its internal structure, focusing on code and algorithms.
  - Example: Code Review
- **Grey Box Testing**: Combines black box and white box testing, focusing on both functionality and internal structure.
  - Example: Equivalence Partitioning
- **Acceptance Testing**: Verifies that software meets requirements and user acceptance criteria.
  - Example: User Acceptance Testing (UAT)
- **Equivalence Partitioning**: Divides input data into partitions to test all possible scenarios.
  - Formula: n = (2^n - 1) / log2(p)
  - Definition: Equivalence partitioning is a technique used to divide the input data into partitions to test all possible scenarios.
- **Boundary Value Analysis**: Tests input values at the boundaries of partitions.
  - Theorem: If n different partitions are created, then at most 2^n different test cases can be created.

### Agile Methodology

- **Agile Manifesto**: Emphasizes flexibility, collaboration, and rapid delivery.
  - Values: Individuals and interactions, working software, customer collaboration, and responding to change
- **Scrum Framework**: A popular agile framework that emphasizes iterative development and continuous improvement.
  - Components: Sprints, Daily Scrum, Sprint Review, Sprint Retrospective
- **Kanban Board**: A visual representation of the workflow, used to track progress and improve efficiency.
  - Components: Columns, Cards, WIP Limits
- **Continuous Integration**: Integrates code changes into the main codebase frequently, reducing integration errors.
  - Benefits: Faster feedback, reduced errors, improved quality

### Important Formulas and Definitions

- **Test-Effectiveness Formula**: (Number of Test Cases / Total Number of Requirements) x 100
- **Test-Coverage Formula**: (Number of Test Cases / Total Number of Code Lines) x 100
- **Definition of Done (DoD)**: A set of criteria that defines when a feature or user story is complete and ready for testing.
