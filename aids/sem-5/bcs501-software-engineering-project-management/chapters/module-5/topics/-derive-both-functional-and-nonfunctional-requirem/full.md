# Deriving Functional and Nonfunctional Requirements

### Introduction

In software engineering, requirements are the essential elements that define the scope, behavior, and constraints of a software system. These requirements can be categorized into two main types: functional requirements and nonfunctional requirements. Functional requirements describe what the software should do, while nonfunctional requirements describe the characteristics of the software that are not directly related to its functionality. In this section, we will delve into the world of requirements derivation, exploring both functional and nonfunctional requirements in depth.

### Historical Context

The concept of requirements has been around for decades, dating back to the early days of computing. However, it wasn't until the 1970s that the term "requirements" became widely accepted. The US Department of Defense's (DoD) Department of Information Processing Standards (DIPS) publication, "Information Processing Systems - Interoperability Standard," introduced a structured approach to requirements derivation.

In the 1980s, the IEEE introduced the IEEE 830-1984 standard, which provided a comprehensive framework for software requirements engineering. This standard emphasized the importance of requirements derivation, analysis, and validation.

In recent years, the development of agile methodologies and cloud computing has led to a shift towards more flexible and iterative requirements derivation processes.

### Functional Requirements

Functional requirements describe what the software should do, including its behavior, interactions, and outputs. These requirements are typically expressed in natural language, diagrams, or use cases. The following are some key characteristics of functional requirements:

- **Description**: A statement of what the software should do or provide.
- **Scope**: The extent to which the requirement applies to the system.
- **Assumptions**: The underlying assumptions made when deriving the requirement.
- **Constraints**: The limitations or boundaries that affect the requirement.

### Nonfunctional Requirements

Nonfunctional requirements describe the characteristics of the software that are not directly related to its functionality, such as performance, security, and usability. These requirements are also known as "quality attributes" or "non- functional attributes." The following are some key characteristics of nonfunctional requirements:

- **Description**: A statement of the desired characteristic or attribute of the software.
- **Scope**: The extent to which the requirement applies to the system.
- **Assumptions**: The underlying assumptions made when deriving the requirement.
- **Constraints**: The limitations or boundaries that affect the requirement.

### Deriving Functional Requirements

Deriving functional requirements involves identifying the functionality that the software should provide. This can be done using various techniques, including:

- **Use Cases**: A written description of the desired behavior of the software.
- **Use Case Diagrams**: A visual representation of the use cases.
- **State Machines**: A mathematical model of the system's behavior.
- **Prototyping**: Creating a prototype to demonstrate the functionality.

### Example

Suppose we are developing a web-based banking system that allows users to view their account balances, transfer funds, and pay bills. Some possible functional requirements for this system might include:

- "The system shall allow users to view their account balances in real-time."
- "The system shall enable users to transfer funds between accounts using a secure payment gateway."
- "The system shall allow users to pay bills using a payment processing service."

### Deriving Nonfunctional Requirements

Deriving nonfunctional requirements involves identifying the characteristics of the software that are not directly related to its functionality. This can be done using various techniques, including:

- **Performance Metrics**: Measuring the system's performance using metrics such as response time, throughput, and memory usage.
- **Security Requirements**: Identifying the security threats and vulnerabilities that the system should address.
- **Usability Requirements**: Defining the user experience and interaction patterns that the system should support.

### Example

Suppose we are developing a web-based banking system that allows users to view their account balances, transfer funds, and pay bills. Some possible nonfunctional requirements for this system might include:

- "The system shall respond to user input within 1 second."
- "The system shall use HTTPS encryption to protect user data."
- "The system shall be accessible on devices with screen sizes greater than 1024 pixels."

### Case Study

Suppose we are developing a mobile app that allows users to track their daily physical activity and set fitness goals. The following are some possible functional requirements for this system:

- "The system shall allow users to log their daily physical activity using GPS tracking."
- "The system shall enable users to set fitness goals and track their progress."
- "The system shall provide users with personalized recommendations for improving their physical activity."

Some possible nonfunctional requirements for this system might include:

- "The system shall respond to user input within 500 milliseconds."
- "The system shall use a secure payment gateway to process in-app purchases."
- "The system shall be accessible on devices with screen sizes between 480 and 1024 pixels."

### Applications

Functional requirements are essential in software development, as they define the behavior of the software and ensure that it meets the user's needs. Nonfunctional requirements, on the other hand, are essential in ensuring that the software meets the required quality attributes, such as performance, security, and usability.

### Conclusion

Deriving functional and nonfunctional requirements is a critical step in software engineering. By understanding the importance of requirements derivation, developers and stakeholders can create software systems that meet the user's needs and provide a high-quality user experience. In this section, we have explored both functional and nonfunctional requirements, including their characteristics, derivation techniques, and applications.
