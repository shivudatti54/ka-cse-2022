# Scheduling and Tasking in OpenMP

## Introduction to OpenMP Scheduling

OpenMP provides powerful mechanisms for parallelizing loops and managing tasks in shared memory programming environments. The scheduling of loop iterations and the creation of tasks are fundamental concepts that determine how work is distributed among available threads, directly impacting performance and efficiency.

When parallelizing loops in OpenMP, the runtime system must decide how to distribute loop iterations across multiple threads. This distribution process is called scheduling. Different scheduling strategies offer various trade-offs between load balancing, overhead, and data locality.

## Loop Scheduling in OpenMP

### The schedule Clause

OpenMP provides the `schedule` clause to control how loop iterations are assigned to threads:

```c
#pragma omp parallel for schedule(kind[, chunk_size])
for (int i = 0; i < n; i++) {
    // Loop body
}
```

### Static Scheduling

Static scheduling divides iterations into fixed-size chunks before execution begins. The `schedule(static, chunk_size)` directive assigns chunks of `chunk_size` iterations to each thread in a round-robin fashion.

```
Thread 0: [0, 1, 2, 3, 4]   [20, 21, 22, 23, 24]
Thread 1: [5, 6, 7, 8, 9]   [25, 26, 27, 28, 29]
Thread 2: [10, 11, 12, 13, 14] [30, 31, 32, 33, 34]
Thread 3: [15, 16, 17, 18, 19] [35, 36, 37, 38, 39]
```

**Advantages:**

- Minimal runtime overhead
- Good data locality when chunk_size aligns with cache lines
- Predictable execution pattern

**Disadvantages:**

- Poor load balancing if iteration times vary significantly

### Dynamic Scheduling

Dynamic scheduling (`schedule(dynamic, chunk_size)`) assigns chunks to threads as they become available, using a shared work queue.

```
Initial state:
Work queue: [0-4], [5-9], [10-14], [15-19], [20-24], [25-29], [30-34], [35-39]

Thread 0 takes [0-4]
Thread 1 takes [5-9]
Thread 2 takes [10-14]
Thread 3 takes [15-19]

After completion:
Thread 0 takes [20-24]
Thread 1 takes [25-29]
Thread 2 takes [30-34]
Thread 3 takes [35-39]
```

**Advantages:**

- Excellent load balancing for uneven workloads
- Adapts to varying iteration times

**Disadvantages:**

- Higher runtime overhead due to synchronization
- Poor data locality

### Guided Scheduling

Guided scheduling (`schedule(guided, chunk_size)`) starts with larger chunks and gradually decreases chunk size as work is completed. The chunk size is proportional to the number of remaining iterations divided by the number of threads.

```
Initial chunks: Large
Remaining iterations decrease → chunk size decreases
```

**Advantages:**

- Good compromise between static and dynamic
- Reduces synchronization overhead compared to dynamic
- Better load balancing than static

**Disadvantages:**

- More complex implementation
- Moderate runtime overhead

### Auto Scheduling

Auto scheduling (`schedule(auto)`) lets the OpenMP implementation choose the scheduling strategy, which may be based on hardware characteristics or runtime analysis.

**Runtime Scheduling**

The `schedule(runtime)` directive uses the scheduling policy specified in the `OMP_SCHEDULE` environment variable, allowing flexibility without recompilation.

| Dynamic         | High     | Excellent      | Poor          | Irregular iterations |

## Tasking in OpenMP

### Introduction to Tasks

Tasks represent units of work that can be executed by any thread. Unlike loop-based parallelism, tasking allows for irregular and recursive parallelism patterns.

```c
#pragma omp parallel
{
    #pragma omp single
    {
        for (int i = 0; i < n; i++) {
            #pragma omp task
            {
                process_item(i);
            }
        }
    }
}
```

### Task Construct Syntax

The basic task construct creates an explicit task:

```c
#pragma omp task [clause[[,] clause]...]
{
    // Task code
}
```

### Task Synchronization

**Taskwait Directive:**
The `#pragma omp taskwait` directive ensures all child tasks generated by the current task are completed before proceeding.

```c
#pragma omp task
{
    // Some work
}

#pragma omp taskwait  // Wait for the above task to complete
// Continue after task completion
```

**Barrier Synchronization:**
The `#pragma omp barrier` directive synchronizes all threads in a parallel region.

### Task Dependencies

OpenMP 4.0 introduced task dependencies using the `depend` clause:

```c
#pragma omp task depend(in: x)  // Task reads x
{
    // Use x
}

#pragma omp task depend(out: y)  // Task writes y
{
    // Compute y
}

#pragma omp task depend(inout: z)  // Task reads and writes z
{
    // Update z
}
```

Dependencies create ordering constraints between tasks, enabling efficient pipeline patterns and dataflow programming.

### Task Scheduling Constraints

The `if` clause conditionally creates tasks:

```c
#pragma omp task if(condition)
{
    // Task created only if condition is true
}
```

The `final` clause prevents further task generation from a task:

```c
#pragma omp task final(condition)
{
    // If condition is true, tasks inside this task are executed immediately
}
```

### Task Groups

Task groups (OpenMP 5.0) provide structured task synchronization:

```c
#pragma omp taskgroup
{
    #pragma omp task
    { /* Task 1 */ }

    #pragma omp task
    { /* Task 2 */ }
}
// Wait for all tasks in the taskgroup to complete
```

## Practical Examples

### Example 1: Matrix Multiplication with Static Scheduling

```c
#pragma omp parallel for schedule(static, 16)
for (int i = 0; i < N; i++) {
    for (int j = 0; j < N; j++) {
        C[i][j] = 0;
        for (int k = 0; k < N; k++) {
            C[i][j] += A[i][k] * B[k][j];
        }
    }
}
```

### Example 2: Tree Processing with Tasks

```c
void process_tree(Node* node) {
    if (node == NULL) return;

    #pragma omp task firstprivate(node)
    {
        process_node(node);
    }

    #pragma omp task firstprivate(node)
    {
        process_tree(node->left);
    }

    #pragma omp task firstprivate(node)
    {
        process_tree(node->right);
    }

    #pragma omp taskwait
}
```

### Example 3: Pipeline with Task Dependencies

```c
#pragma omp parallel
#pragma omp single
{
    for (int i = 0; i < n; i++) {
        #pragma omp task depend(in: data[i]) depend(out: processed[i])
        {
            process_data(data[i], processed[i]);
        }

        #pragma omp task depend(in: processed[i])
        {
            analyze_result(processed[i]);
        }
    }
}
```

## Performance Considerations

### Load Balancing

Choose scheduling strategy based on workload characteristics:

- Use static scheduling for uniform iteration times
- Use dynamic or guided scheduling for irregular workloads
- Consider chunk size carefully to balance overhead and load balancing

### Overhead Analysis

Different scheduling strategies incur different overheads:

- Static: Minimal overhead, fixed at compile time
- Dynamic: Runtime overhead due to shared queue management
- Guided: Moderate overhead with adaptive chunk sizing

### Data Locality

Optimize for cache performance:

- Use larger chunks in static scheduling for better temporal locality
- Consider data access patterns when choosing chunk size
- Use `affinity` clauses (OpenMP 4.5+) for better data placement

## Advanced Topics

### Task Loops (OpenMP 4.5)

The `taskloop` construct combines tasks and loops:

```c
#pragma omp taskloop grainsize(G)
for (int i = 0; i < n; i++) {
    // Each iteration becomes a task
}
```

### Task Reduction (OpenMP 5.0)

Task reduction enables reduction operations across tasks:

```c
#pragma omp parallel
#pragma omp single
#pragma omp taskgroup task_reduction(+: sum)
{
    for (int i = 0; i < n; i++) {
        #pragma omp task in_reduction(+: sum)
        {
            sum += compute_value(i);
        }
    }
}
```

### Task Priority (OpenMP 5.0)

Task priority hints help schedulers:

```c
#pragma omp task priority(high_priority ? 10 : 1)
{
    // High priority task
}
```
