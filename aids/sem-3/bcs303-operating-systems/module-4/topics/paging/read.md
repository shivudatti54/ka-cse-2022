## Introduction to Paging

Paging is a memory management scheme that eliminates the need for contiguous allocation of physical memory. This method solves the external fragmentation problem inherent in earlier memory management techniques like fixed and variable partitioning.

In paging, the **physical address space** is divided into fixed-size blocks called **frames**, while the **logical address space** (of a process) is divided into blocks of the same size called **pages**. The size of pages and frames is typically a power of 2, determined by the hardware architecture (e.g., 4KB, 8KB).

The primary advantage of paging is that it allows the physical memory corresponding to a process's pages to be non-contiguous. This means the operating system can store a process's pages in any available memory frames, dramatically increasing memory utilization and eliminating external fragmentation.

## Basic Concepts of Paging

When a process is to be executed, its pages are loaded into any available memory frames. The operating system maintains a data structure for each process called a **page table**. This table maps the process's logical pages to the physical frames where they are stored.

Key terms:

- **Page Number (p)**: The part of a logical address that specifies the page.
- **Page Offset (d)**: The part of a logical address that specifies the displacement (or location) within the page.
- **Frame Number (f)**: The physical frame number where a page is stored.

The logical address generated by the CPU is divided into two parts:

1. A page number (p)
2. A page offset (d)

The page number is used as an index into the page table to find the corresponding frame number. The frame number is combined with the page offset to produce the physical address.

```
Logical Address: [ p | d ]
 | |
 | +--> Used directly as offset in physical frame
 |
 +--> Indexes into Page Table to get Frame Number (f)

Physical Address: [ f | d ]
```

### Example of Address Translation

Assume a system with:

- Page size = 4 bytes
- Physical memory = 32 bytes (8 frames)
- A process with 4 pages (logical addresses 0-15)

Page Table for the process:
| Page | Frame |
|------|-------|
| 0 | 5 |
| 1 | 2 |
| 2 | 7 |
| 3 | 0 |

Let's translate logical address 10:

- Page size is 4 bytes, so the page number is `10 / 4 = 2` (integer division)
- The offset is `10 % 4 = 2`
- Looking up page 2 in the page table gives frame 7
- Physical address = `(7 * 4) + 2 = 28 + 2 = 30`

## Page Table Structure

The page table is the fundamental data structure that enables paging. It's typically stored in the main memory, and a dedicated hardware register called the **Page Table Base Register (PTBR)** points to the start of the page table for the currently executing process.

### Simple Page Table

A simple page table is essentially a linear array of page table entries (PTEs). Each PTE contains several fields of information:

```
+-------------------------------+
| Frame Number | Present Bit | Other Bits |
+-------------------------------+
```

Common bits in a Page Table Entry:

- **Frame Number**: The physical frame number where the page is stored.
- **Present/Absent Bit**: Indicates whether the page is currently in memory (1) or not (0).
- **Protection Bits**: Specify read, write, and execute permissions for the page.
- **Modified (Dirty) Bit**: Indicates if the page has been modified since it was loaded.
- **Referenced (Accessed) Bit**: Indicates if the page has been accessed recently.
- **Caching Disabled Bit**: Important for memory-mapped I/O.

### Performance Consideration and TLB

The major issue with page tables is performance. Since the page table is stored in main memory, each memory reference requires two memory accesses: one for the page table entry and one for the actual data/instruction.

To overcome this performance penalty, computer systems use a special, small, fast lookup hardware cache called a **Translation Lookaside Buffer (TLB)**. The TLB stores recently used page-to-frame mappings.

```
CPU generates logical address
 |
 v
Check TLB for page number
 |
 +--> Hit: Get frame number immediately (fast)
 |
 +--> Miss: Look up in page table (slow), then update TLB
```

TLB hit ratio is crucial for performance. A high hit ratio means most translations are found in the TLB, minimizing the need to access the page table in memory.

## Hierarchical Paging Schemes

As modern systems support very large address spaces (e.g., 64-bit addresses), the page table itself can become excessively large. For example, a 32-bit system with 4KB pages would need 1 million page table entries (2²⁰), requiring 4MB of memory per process just for the page table.

Several solutions address this problem:

### Two-Level Paging

The page number is split into two parts (p₁ and p₂), creating a hierarchical page table structure.

```
Logical Address: [ p1 | p2 | d ]
 | | |
 | | +--> Page offset
 | |
 | +--> Index into inner page table
 |
 +--> Index into outer page table (page directory)
```

The outer page table (page directory) points to inner page tables, which then point to actual frames.

### Multi-Level Paging

Extended further for 64-bit systems, with three or more levels of paging (e.g., x86-64 uses 4-level paging).

### Hashed Page Tables

Common in address spaces larger than 32 bits. Virtual page numbers are hashed into a hash table. Each entry contains a linked list of elements that hash to the same location (to handle collisions).

### Inverted Page Tables

Rather than having one page table per process, the system has a single inverted page table that has one entry for each physical frame. Each entry stores which page of which process is stored in that frame.

This approach saves significant memory but increases lookup time, as searching the table may be required. It's often combined with hashing for better performance.

## Comparison of Paging with Other Schemes

| Technique                 | Fragmentation                                 | Memory Utilization | Implementation Complexity              |
| ------------------------- | --------------------------------------------- | ------------------ | -------------------------------------- |
| **Fixed Partitioning**    | External fragmentation                        | Poor               | Simple                                 |
| **Variable Partitioning** | External fragmentation                        | Moderate           | Moderate                               |
| **Paging**                | No external fragmentation (internal possible) | Excellent          | More complex (hardware support needed) |
| **Segmentation**          | External fragmentation                        | Good               | Complex                                |

## Internal Fragmentation in Paging

While paging eliminates external fragmentation, it can suffer from **internal fragmentation**. This occurs because processes are allocated in fixed-size pages, and the last page of a process may not be completely filled.

The average internal fragmentation is approximately half a page per process. With smaller page sizes, internal fragmentation decreases, but the page table size increases.

## Protection and Sharing in Paging

Paging provides excellent memory protection through the protection bits in each page table entry. Different pages can have different permissions (read-only, read-write, execute).

Paging also facilitates **memory sharing** through shared pages. Common code (like libraries) can be mapped into the address space of multiple processes, but stored only once in physical memory. This is achieved by having page table entries in different processes point to the same physical frame.

## Exam Tips

1. **Remember the address translation formula**: Physical Address = (Frame Number × Page Size) + Offset
2. **TLB is crucial for performance**: Understand that without TLB, each memory access would take at least twice as long.
3. **Differentiate fragmentation types**: Paging eliminates external fragmentation but may cause internal fragmentation.
4. **Know the purpose of each bit in a PTE**: Present bit, protection bits, modified bit, etc., all serve important functions.
5. **Understand hierarchical paging**: Be able to explain why it's necessary for large address spaces and how it works.
6. **Practice numerical problems**: Calculate logical/physical addresses given page tables, and compute hit ratios for TLBs.
7. **Compare techniques**: Be prepared to contrast paging with segmentation and other memory management schemes.
