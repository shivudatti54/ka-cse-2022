## What is Virtual Memory?

Virtual memory is a technique that allows the execution of processes that are **not completely in main memory**. It provides an abstraction of the main memory as a large, uniform array of storage, separating logical memory as viewed by the programmer from physical memory.

**Key Idea:** Programs can be larger than physical memory. Only the parts of a program that are currently needed reside in memory; the rest stay on disk.

```
+---------------------------------------------+
| VIRTUAL MEMORY CONCEPT |
+---------------------------------------------+
| |
| Program Size: 2 GB |
| Physical RAM: 512 MB |
| Disk (Swap): Available as backing store |
| |
| Virtual memory bridges the gap — |
| the process "sees" 2 GB, but only |
| active portions occupy physical RAM. |
+---------------------------------------------+
```

## Why is Virtual Memory Needed?

1. **Programs larger than physical memory:** Many real-world applications (databases, IDEs, games) require more memory than the available RAM. Without virtual memory, such programs simply cannot run.

2. **Multiprogramming efficiency:** If each process had to be loaded entirely into memory, only a few processes could run simultaneously. Virtual memory allows many processes to share RAM by keeping only active pages in memory.

3. **Simplified programming model:** Programmers do not need to worry about physical memory size or overlays. The OS handles placement transparently.

4. **Process isolation:** Each process gets its own virtual address space, preventing one process from accessing another's memory.

| Problem Without Virtual Memory       | Solution With Virtual Memory              |
| ------------------------------------ | ----------------------------------------- |
| Program must fit entirely in RAM     | Only active parts loaded; rest on disk    |
| Few processes can run simultaneously | Many processes share RAM efficiently      |
| Programmer must manage overlays      | OS handles memory placement automatically |
| Memory fragmentation wastes RAM      | Paging eliminates external fragmentation  |
| No isolation between processes       | Each process has its own address space    |

## Separation of Logical Memory from Physical Memory

Virtual memory **decouples** the logical address space (what the process sees) from the physical address space (actual RAM). This separation is fundamental:

- **Logical (Virtual) Address Space:** The set of all addresses generated by the CPU for a process. It can be much larger than physical memory.
- **Physical Address Space:** The set of all addresses available in main memory (RAM).

```
 LOGICAL ADDRESS SPACE PHYSICAL ADDRESS SPACE
 (per process, virtual) (actual RAM)

 +-------------------+ +-------------------+
 | Page 0 |----+ | Frame 0 |
 +-------------------+ | +-------------------+
 | Page 1 |----+-----+--->| Frame 1 |
 +-------------------+ | | +-------------------+
 | Page 2 | +-----+--->| Frame 2 |
 +-------------------+ | +-------------------+
 | Page 3 (on disk)| | | Frame 3 |
 +-------------------+ | +-------------------+
 | ... | | | ... |
 +-------------------+ | +-------------------+
 | Page N |----------+ | Frame M |
 +-------------------+ +-------------------+

 N >> M (Virtual space is much larger than physical space)
 Pages not in RAM reside on disk (backing store / swap space)
```

The **Memory Management Unit (MMU)** performs the translation from logical addresses to physical addresses at runtime using a **page table**.

Every process has a virtual address space organized into well-defined regions:

### Regions Explained

| Region | Contents                        | Growth Direction |
| ------ | ------------------------------- | ---------------- |
| Text   | Compiled machine code           | Fixed size       |
| Data   | Global and static variables     | Fixed size       |
| Heap   | Dynamically allocated memory    | Grows upward     |
| Stack  | Function parameters, local vars | Grows downward   |

## Sparse Address Spaces

The gap between the heap and the stack is called a **hole** or sparse region. This space is part of the virtual address space but has **no physical memory or disk backing** until it is actually used.

**Why is this useful?**

- The heap can grow upward into the hole as `malloc()` allocates memory.
- The stack can grow downward into the hole as functions are called.
- The OS allocates physical frames only when these virtual pages are actually accessed (via demand paging).
- Sparse address spaces allow the virtual address space to be very large without wasting physical resources.

```
 Virtual Address Space (sparse)

 +----------+
 | Stack | Backed by physical frames
 +----------+
 | Hole | NO physical memory allocated
 | (sparse)| Pages exist only in the page table
 | | with "invalid" bits
 +----------+
 | Heap | Backed by physical frames
 +----------+
 | Data | Backed by physical frames
 +----------+
 | Text | Backed by physical frames
 +----------+
```

## Shared Libraries and Shared Pages

Virtual memory enables **shared libraries** (e.g., libc, DLLs) to be loaded once into physical memory and mapped into the virtual address space of every process that needs them.

```
 Process A Physical Memory Process B
 +-----------+ +---------------+ +-----------+
 | Text | | | | Text |
 | Data | | Shared | | Data |
 | Heap | | Library | | Heap |
 | | | (libc.so) | | |
 | Shared Lib|---------->| ONE COPY |<------| Shared Lib|
 | Stack | | | | Stack |
 +-----------+ +---------------+ +-----------+

 Both processes map the same physical frames for the shared library.
 Saves significant memory when many processes use the same library.
```

**Advantages of shared pages:**

- Only **one copy** of a shared library exists in physical memory.
- Multiple processes map their virtual pages to the **same physical frames**.
- Reduces total memory consumption significantly (e.g., 50 processes sharing libc save ~49 copies worth of RAM).

## Benefits of Virtual Memory

| Benefit                        | Explanation                                                      |
| ------------------------------ | ---------------------------------------------------------------- |
| Larger logical address space   | Process can address more memory than physically available        |
| More programs in memory        | Only active pages are loaded, allowing more concurrent processes |
| Efficient process creation     | `fork()` uses copy-on-write; child shares parent's pages         |
| Less I/O for loading programs  | Only needed pages are loaded, not the entire program             |
| Simplified programming         | No need for programmer-managed overlays                          |
| Memory-mapped files            | Files can be accessed as memory, simplifying I/O                 |
| Shared memory via shared pages | Efficient inter-process communication and shared libraries       |

### Process Creation Efficiency

When a new process is created using `fork()`:

1. The child process does **not** get a full copy of the parent's memory.
2. Both parent and child share the same physical pages (marked read-only).
3. A page is copied **only when one of them tries to write** (Copy-on-Write).
4. This makes process creation extremely fast.

```
 Before Write (pages shared):
 Parent --> [Page A] [Page B] [Page C]
 Child --> [Page A] [Page B] [Page C] (same physical frames)

 After Child Writes to Page B:
 Parent --> [Page A] [Page B ] [Page C]
 Child --> [Page A] [Page B'] [Page C] (only Page B is copied)
```

## How Virtual Memory is Implemented — Demand Paging

Virtual memory is most commonly implemented using **demand paging**.

**Demand Paging:** A page is loaded into physical memory **only when it is needed** (i.e., when the CPU references an address on that page). Pages that are never accessed are never loaded.

### How Demand Paging Works

1. Process starts executing. Initially, few or no pages are in memory.
2. CPU generates a virtual address.
3. MMU checks the page table:

- **Valid bit = 1:** Page is in memory. Translate and access.
- **Valid bit = 0:** Page is NOT in memory. A **page fault** occurs.

4. On page fault:
   a. OS traps and saves the process state.
   b. OS finds the page on disk (backing store / swap space).
   c. OS loads the page into a free frame in physical memory.
   d. OS updates the page table (sets valid bit = 1, records frame number).
   e. Process resumes the interrupted instruction.

```
 CPU generates Page Table Physical Backing Store
 virtual address Lookup Memory (Disk)
 | | | |
 v v v v
 +---------+ +-----------+ +---------+ +---------+
 | Virt. |-->| Valid=1? |--Y-->| Frame | | |
 | Address | | Frame# | | (data) | | Page |
 +---------+ +-----------+ +---------+ | Image |
 | +---------+
 N (Page Fault!) |
 | |
 +---------> OS loads page ---------+
 from disk into
 a free frame
```

### Demand Paging vs. Swapping

| Aspect             | Demand Paging                     | Swapping                         |
| ------------------ | --------------------------------- | -------------------------------- |
| Unit of transfer   | Individual pages                  | Entire process                   |
| Granularity        | Fine-grained                      | Coarse-grained                   |
| Speed              | Faster (less data moved)          | Slower (entire process moved)    |
| Memory utilization | Better (only needed pages loaded) | Worse (entire process in memory) |
| Modern usage       | Standard technique                | Rarely used in pure form         |

### Pure Demand Paging

In **pure demand paging**, no pages are loaded when a process starts. Every first access to every page causes a page fault. This is the extreme form of demand paging.

- **Advantage:** No unnecessary pages are ever loaded.
- **Disadvantage:** High initial page fault rate; performance improves over time as the working set is loaded.

### Locality of Reference

Demand paging works well because programs exhibit **locality of reference**:

- **Temporal Locality:** If an address is accessed now, it is likely to be accessed again soon.
- **Spatial Locality:** If an address is accessed, nearby addresses are likely to be accessed soon.

This means that at any given time, a process typically uses only a small subset of its pages (the **working set**), making demand paging efficient.

## Exam Tips

1. **Frequently asked:** "Explain virtual memory and its advantages" — cover definition, separation of logical/physical memory, and the benefits table above.

2. **Diagram questions:** Be ready to draw the virtual address space layout (Text, Data, Heap, Hole, Stack) and the mapping from virtual pages to physical frames. These appear regularly in exams.

3. **Demand paging steps:** The 5-step page fault handling sequence is a very common 5-mark or 10-mark question. Memorize the sequence: trap, find page on disk, load into frame, update page table, restart instruction.

4. **Differentiate clearly:** Virtual memory vs. physical memory, demand paging vs. swapping, logical address vs. physical address. Comparison tables score well.

5. **Numerical awareness:** For a 32-bit system, the virtual address space is 4 GB (2^32 bytes). Physical memory is typically much smaller. This distinction is important for numerical problems.

6. **Copy-on-Write:** Often asked as part of process creation. Explain how `fork()` uses shared pages and copies only on write.

7. **Shared libraries:** A short-answer favorite. Explain how one physical copy serves multiple processes via virtual memory mapping.

8. **Key formula to remember:** Virtual Address = (Page Number, Offset). The page number indexes into the page table to get the frame number. Physical Address = (Frame Number, Offset).
