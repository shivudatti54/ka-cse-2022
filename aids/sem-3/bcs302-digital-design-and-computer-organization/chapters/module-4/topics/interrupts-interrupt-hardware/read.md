# Interrupts and Interrupt Handling

## Introduction to Interrupts

An **interrupt** is a signal sent to the processor by hardware or software indicating an event that needs immediate attention. It alerts the processor to a high-priority condition requiring the interruption of the current code being executed. The processor responds by suspending its current activities, saving its state, and executing a function called an **interrupt handler** (or interrupt service routine - ISR) to deal with the event. This mechanism is crucial for efficient I/O operations, allowing the CPU to perform other tasks while waiting for slower I/O devices.

### Why are Interrupts Needed?
Without interrupts, a processor would have to **poll** (repeatedly check) the status of all devices to see if they need attention. This is highly inefficient as it wastes CPU cycles. Interrupts provide a mechanism for devices to notify the CPU only when they require service, enabling:
*   Efficient CPU utilization
*   Faster response to critical events
*   Simplified software design for handling asynchronous events

## Types of Interrupts

Interrupts can be broadly classified based on their source.

### 1. Hardware Interrupts
These are generated by external hardware devices. They are asynchronous to the current program execution.
*   **Examples:** Keyboard press, mouse movement, data arrival from a network card, timer signal, disk I/O completion.
*   **Process:** An electrical signal is sent on a dedicated interrupt request line (IRQ) to the processor's interrupt pin.

### 2. Software Interrupts (Traps)
These are generated by software, typically by executing a special instruction (`INT n` in x86). They are synchronous because they are caused directly by the executing program.
*   **Examples:** System calls (e.g., requesting a service from the operating system kernel like reading a file), debugging breakpoints, division by zero errors.

### 3. Exceptions
These are a type of interrupt generated by the CPU itself when it encounters an error or an unusual condition during instruction execution. They are also synchronous.
*   **Examples:** Page fault (requested memory page is not in RAM), segmentation fault (invalid memory access), privilege violation.

**Comparison Table: Types of Interrupts**

| Feature          | Hardware Interrupt                              | Software Interrupt / Trap                      | Exception                                       |
| :--------------- | :---------------------------------------------- | :--------------------------------------------- | :---------------------------------------------- |
| **Source**       | External hardware device                        | Executing program (special instruction)        | CPU (error or special condition during execution) |
| **Synchronicity** | Asynchronous (to program flow)                  | Synchronous (caused by program)                | Synchronous (caused by program)                 |
| **Purpose**      | Handle external events (I/O)                    | Request OS services, debugging                 | Signal error conditions, handle special cases   |
| **Example**      | Key press, data received                        | System call (`read`, `write`)                  | Division by zero, page fault                    |

## The Interrupt Handling Process

Handling an interrupt is a multi-step process managed jointly by the CPU hardware and the operating system software.

### 1. Interrupt Occurrence
A device raises an interrupt signal on its assigned Interrupt Request (IRQ) line.

### 2. CPU Acknowledgement
The CPU finishes its current instruction (to maintain atomicity). It then acknowledges the interrupt, signaling the device that the request has been received. The device lowers the interrupt signal.

### 3. State Saving (Context Preservation)
The CPU automatically saves the current **program state** (or context) onto the stack. This minimally includes:
*   **Program Counter (PC):** The address of the next instruction to be executed.
*   **Processor Status Word (PSW)/Flags Register:** Contains condition codes (flags) like zero, carry, overflow, and interrupt enable bits.

This allows the CPU to resume the original program exactly where it left off after handling the interrupt.

### 4. Identifying the Interrupt Source (Vectored Interrupts)
The CPU needs to determine which interrupt handler to execute. This is most efficiently done through **vectored interrupts**. Each interrupt source is assigned a unique number called an **interrupt vector**. This vector is an index into the **Interrupt Vector Table (IVT)**.

*   **Interrupt Vector Table (IVT):** A pre-defined array in low memory (e.g., the first 1KB of RAM) that holds the memory addresses of the Interrupt Service Routines (ISRs).
*   The interrupting device provides its vector number.
*   The CPU uses this number as an index to look up the ISR address from the IVT.

```
+---------------------+      +-----------------------+
|   Hardware Device   |      | Interrupt Vector Table|
| (e.g., Keyboard)    |      | (in memory)           |
| IRQ = 1             |      +-----------------------+ 
| Vector = 0x21       |      | 0x00: ISR_0_Addr     |
+---------------------+      | 0x01: ISR_1_Addr     |
         |                   | ...                  |
         | (Sends IRQ & Vec) | 0x21: ISR_21_Addr   |
         +------------------>| ...                  |
                             | 0xFF: ISR_255_Addr   |
                             +-----------------------+
                                          |
                                          | (CPU fetches ISR address)
                                          v
                             +-----------------------+
                             | Interrupt Service     |
                             | Routine (ISR)         |
                             | for Keyboard          |
                             +-----------------------+
```

### 5. Executing the Interrupt Service Routine (ISR)
The CPU jumps to the address retrieved from the IVT and begins executing the ISR. The ISR is a software routine written to handle the specific interrupt.
*   **ISR Responsibilities:**
    *   Save any CPU registers it will use (beyond what the hardware saved).
    *   Determine the cause of the interrupt (if multiple devices share a vector).
    *   Perform the required processing (e.g., read the key pressed from the keyboard's data register).
    *   Send an End of Interrupt (EOI) signal to the Programmable Interrupt Controller (PIC) if one is present.
    *   Restore the saved registers.
    *   Execute a **Return from Interrupt (RTI)** instruction.

### 6. State Restoration and Resumption
The `RTI` instruction pops the saved Program Counter and Flags register from the stack. The CPU resumes execution of the interrupted program from the exact point where it stopped, unaware that it was interrupted.

**ASCII Diagram of the Complete Interrupt Handling Flow**

```
+----------------------------------------------------------------------+

```

## Key Concepts in Interrupt Handling

### Interrupt Masking
The CPU has an **Interrupt Enable (IE)** flag in its flags register.
*   If this flag is **cleared** (disabled), the CPU will ignore all maskable interrupt requests. This is crucial for protecting critical sections of code (e.g., when the OS kernel is updating a data structure) from being interrupted.
*   **Non-Maskable Interrupts (NMIs)** cannot be disabled by this flag and are reserved for critical events like power failure or memory parity errors.
*   The ISR itself often disables interrupts at the beginning (to avoid being interrupted itself) and re-enables them later.

### Priority and Nesting
Not all interrupts are equally urgent. A system needs a way to prioritize them.
*   **Hardware Priority:** The interrupt controller hardware (like a PIC or APIC) resolves simultaneous interrupts based on pre-assigned priorities (often IRQ0 is highest priority).
*   **Nested Interrupts:** If the ISR re-enables interrupts, a higher-priority interrupt can "interrupt" a lower-priority ISR. This is called interrupt nesting. It adds complexity but improves response time for critical events.

### Direct Memory Access (DMA) and Interrupts
DMA and interrupts work together. A device using DMA to transfer a block of data to memory will generate an interrupt only when the **entire transfer is complete**, not after every byte. This drastically reduces the number of interrupts and frees the CPU for other work during the transfer.

```
CPU Program: "Read 1024 bytes from disk"
     |
     v                              (DMA Controller manages data transfer)
CPU -> Initiates DMA Transfer -------------------------------------------+
     |                                                                   |
     |                                                                   |
     v                                                                   v
CPU continues other work...                                    Disk reads data directly to RAM
     |                                                                   |
     |                                                                   |
     |<-------------------------- Interrupt (Transfer Complete) ---------+
     |
     v
CPU handles interrupt, knows data is ready
```

## Exam Tips

1.  **Understand the Sequence:** Memorize the steps of the interrupt handling cycle (Finish instruction -> Save state -> Get vector -> Execute ISR -> RTI). This is a common question.
2.  **Differentiate Types:** Be able to clearly explain the difference between hardware, software interrupts, and exceptions with examples.
3.  **Vector Table Purpose:** Don't just say "it holds addresses." Explain *why* it's used: to efficiently route any number of interrupts to their correct handler code.
4.  **Context is Key:** Emphasize *what* is saved (PC, Flags) and *why* it's saved (to resume transparently). Mention that the ISR must save any other registers it uses.
5.  **Connect to DMA:** Be prepared to explain how DMA reduces interrupt load. A question might contrast programmed I/O (byte-by-byte with an interrupt per byte) with DMA (one interrupt per block).
6.  **Watch the Language:** Use precise terms like "asserts an interrupt," "interrupt vector," "ISR," and "RTI instruction."
