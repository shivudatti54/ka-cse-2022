# The CYK Parsing Algorithm

## Introduction to CYK Parsing

The Cocke-Younger-Kasami (CYK) algorithm is a dynamic programming parsing technique for context-free grammars (CFGs). Named after its inventors John Cocke, Daniel Younger, and Tadao Kasami, it is particularly significant in Natural Language Processing (NLP) because it can efficiently determine whether a given string can be generated by a given context-free grammar and, if so, how. Its primary application in the NLP syllabus context is for **constituency parsing**, which is the task of identifying the hierarchical syntactic structure of a sentence according to a phrase structure grammar.

A key requirement for the CYK algorithm is that the grammar must be in **Chomsky Normal Form (CNF)**, a restricted form of CFG. This constraint simplifies the parsing process and ensures the algorithm's correctness and efficiency.

## Chomsky Normal Form (CNF)

A context-free grammar is in Chomsky Normal Form if all its production rules are of one of the following two forms:
1.  **Non-terminal → Terminal** (e.g., `Noun → "dog"`)
2.  **Non-terminal → Non-terminal Non-terminal** (e.g., `VP → Verb NP`)

Rules of the form `A → ε` (producing the empty string) are only allowed if `A` is the start symbol and it does not appear on the right-hand side of any rule.

**Why CNF?** This standardized form eliminates chain rules (e.g., `A -> B`) and rules with more than two non-terminals on the right-hand side. This structural simplicity is what the CYK algorithm's logic is built upon, allowing it to systematically combine smaller parsed segments into larger ones.

### Converting a CFG to CNF

The conversion process involves several steps:
1.  **Eliminate ε-productions** (except for the start symbol).
2.  **Eliminate unit productions** (chain rules like `A -> B`).
3.  **Eliminate useless symbols** (non-terminals that don't derive any terminal string or are unreachable from the start symbol).
4.  **Break up long rules** so that no right-hand side has more than two symbols. For a rule like `A -> B C D`, introduce a new non-terminal (e.g., `X`) and create two rules: `A -> B X` and `X -> C D`.

**Example Conversion:**
Original CFG:
```
S → NP VP
NP → Det N | PN
VP → V NP
Det → "the"
N → "dog" | "cat"
PN → "John"
V → "chased"
```
Converted to CNF:
*   `S → NP VP` is already valid.
*   `NP → Det N` and `NP → PN` are valid.
*   `VP → V NP` is valid.
*   All other rules (`Det → "the"`, `N → "dog"`, `N → "cat"`, `PN → "John"`, `V → "chased"`) are valid.
This grammar is already in CNF.

## The CYK Algorithm: Core Idea and Data Structure

The central idea behind the CYK algorithm is **dynamic programming**. It solves the complex problem of parsing a sentence by breaking it down into smaller, overlapping subproblems (parsing substrings), solving them, and storing the solutions in a table to avoid redundant calculations.

The algorithm uses a **triangular table** (a 2D matrix) often denoted as `dp[i][j]` or `table[i][j]`. For a sentence of `n` words, the table has `n` rows and `n` columns.

*   `table[i][j]`: This cell contains the set of all non-terminals in the grammar that can generate the substring spanning from word `i` to word `j` (where indexing often starts at 1).
*   The algorithm fills this table from the bottom (single words) to the top (the entire sentence).

## The CYK Algorithm: Step-by-Step Procedure

Let's parse the sentence "the dog chased John" using our example CNF grammar.

**Input:**
*   Sentence: `[the, dog, chased, John]` (length `n=4`)
*   CNF Grammar rules:
    *   `S → NP VP`
    *   `NP → Det N | PN`
    *   `VP → V NP`
    *   `Det → "the"`
    *   `N → "dog" | "cat"`
    *   `PN → "John"`
    *   `V → "chased"`

**Step 1: Initialize the diagonal (substrings of length 1)**
For each word position `i`, determine which non-terminals can directly produce that terminal symbol (`A → terminal`).

| j=1<br>'the' | j=2<br>'dog' | j=3<br>'chased' | j=4<br>'John' |
| :--- | :--- | :--- | :--- |
| **i=1** | `{Det}` | ∅ | ∅ | ∅ |
| **i=2** | | `{N}` | ∅ | ∅ |
| **i=3** | | | `{V}` | ∅ |
| **i=4** | | | | `{PN}` |

**Step 2: Fill the table for longer spans (substrings of length L)**
For all substring lengths `L` from 2 to `n`:
    For all possible starting indices `i` of such substrings (from 1 to `n-L+1`):
        Set `j = i + L - 1` (the ending index).
        For every possible split point `k` between `i` and `j-1`:
            For each grammar rule `A → B C`:
                If `B` is in `table[i][k]` and `C` is in `table[k+1][j]`, then add `A` to `table[i][j]`.

**Filling the table for L=2:**
*   Span [1,2]: "the dog"
    *   Split at k=1: Check rules where RHS is two non-terminals.
    *   `NP → Det N`. Is `Det` in [1,1]? Yes. Is `N` in [2,2]? Yes. Therefore, add `NP` to `table[1][2]`.
*   Span [2,3]: "dog chased"
    *   No rule `X → N V` exists. Stays ∅.
*   Span [3,4]: "chased John"
    *   No rule `X → V PN` exists. Stays ∅.

| j=1<br>'the' | j=2<br>'dog' | j=3<br>'chased' | j=4<br>'John' |
| :--- | :--- | :--- | :--- |
| **i=1** | `{Det}` | `{NP}` | ∅ | ∅ |
| **i=2** | | `{N}` | ∅ | ∅ |
| **i=3** | | | `{V}` | ∅ |
| **i=4** | | | | `{PN}` |

**Filling the table for L=3:**
*   Span [1,3]: "the dog chased"
    *   Split at k=1: Check `X → Det Y`. `Det` in [1,1] is true. `Y` would need to generate [2,3] ("dog chased"), but `table[2][3]` is ∅. Fails.
    *   Split at k=2: Check `X → Z V`. `Z` must generate [1,2] ("the dog"). `table[1][2]` is `{NP}`. `V` in [3,3] is true. Is there a rule `X → NP V`? `VP → V NP` exists, but the order is wrong. No rule `X → NP V` exists. Fails. Stays ∅.
*   Span [2,4]: "dog chased John"
    *   Split at k=2: Check `X → N W`. `N` in [2,2] is true. `W` must generate [3,4] ("chased John"). `table[3][4]` is ∅. Fails.
    *   Split at k=3: Check `X → Q PN`. `Q` must generate [2,3] ("dog chased"). `table[2][3]` is ∅. `PN` in [4,4] is true. Fails. Stays ∅.

| j=1<br>'the' | j=2<br>'dog' | j=3<br>'chased' | j=4<br>'John' |
| :--- | :--- | :--- | :--- |
| **i=1** | `{Det}` | `{NP}` | ∅ | ∅ |
| **i=2** | | `{N}` | ∅ | ∅ |
| **i=3** | | | `{V}` | ∅ |
| **i=4** | | | | `{PN}` |

**Filling the table for L=4 (the whole sentence): "the dog chased John"**
*   Split at k=1: Check `X → Det Y`. `Det` in [1,1] is true. `Y` must generate [2,4] ("dog chased John"). `table[2][4]` is ∅. Fails.
*   Split at k=2: Check `X → Z W`.
    *   `Z` generates [1,2] ("the dog") -> `table[1][2] = {NP}`
    *   `W` generates [3,4] ("chased John") -> `table[3][4]` is ∅. Fails.
*   Split at k=3: Check `X → M N`.
    *   `M` generates [1,3] ("the dog chased") -> `table[1][3]` is ∅. Fails.
    *   This would fail, but wait! Let's check the rules. We have `VP → V NP`. We need to see if this can be applied backwards.
    *   Actually, we missed a valid parse. Let's re-examine the split at k=2 more carefully for L=4.
    *   The substring [1,2] is `NP`. The substring [3,4] needs to be parsed. We have a rule `VP → V NP`. To create an `NP` from [3,4], we need to see if it's possible. `table[3][4]` is currently ∅. But we can create an `NP` from [4,4] ("John") because `PN → "John"` and `NP → PN`. So we should have added `NP` to `table[4][4]` initially. Let's correct the initial table.

**Correction and Final Parse:**
Our initial table for `table[4][4]` was wrong. It should contain `{PN, NP}` because `NP → PN` is a valid rule. Let's rebuild the table correctly.

**Corrected Table Initialization (Length 1):**
| j=1<br>'the' | j=2<br>'dog' | j=3<br>'chased' | j=4<br>'John' |
| :--- | :--- | :--- | :--- |
| **i=1** | `{Det}` | ∅ | ∅ | ∅ |
| **i=2** | | `{N}` | ∅ | ∅ |
| **i=3** | | | `{V}` | ∅ |
| **i=4** | | | | `{PN, NP}` | // `NP → PN`

**Length 2:**
*   [1,2]: `NP → Det N` -> `table[1][2] = {NP}`
*   [2,3]: No valid combination -> `∅`
*   [3,4]: `VP → V NP` -> `table[3][4] = {VP}` // Check: `V` in [3,3] is true, `NP` in [4,4] is true.

**Length 3:**
*   [1,3]: Check splits.
    *   k=1: `X → Det Y`. `Det` in [1,1] ok. `Y` must be in [2,3] -> `table[2][3]` is `∅`. Fail.
    *   k=2: `X → Z V`. `Z` in [1,2] is `{NP}`. `V` in [3,3] is `{V}`. Is there a rule `X → NP V`? No. The rule is `VP → V NP`, not `VP → NP V`. Fail. -> `table[1][3] = ∅`
*   [2,4]: Check splits.
    *   k=2: `X → N W`. `N` in [2,2] ok. `W` in [3,4] is `{VP}`. Is there a rule `X → N VP`? No.
    *   k=3: `X → Q PN`. `Q` in [2,3] is `∅`. `PN` in [4,4] is part of `NP`, but not directly. Fail. -> `table[2][4] = ∅`

**Length 4 (The whole sentence):**
*   Split at k=1: `X → Det Y`. `Det` in [1,1] ok. `Y` in [2,4] is `table[2][4] = ∅`. Fail.
*   Split at k=2: `X → Z W`.
    *   `Z` in [1,2] is `{NP}`.
    *   `W` in [3,4] is `{VP}`.
    *   Check rules: `S → NP VP` exists! Therefore, add `S` to `table[1][4]`.
*   Split at k=3: `X → M N`. `M` in [1,3] is `∅`. `N` in [4,4] is `{NP, PN}`. Fail.

**Final Corrected Table:**
| j=1<br>'the' | j=2<br>'dog' | j=3<br>'chased' | j=4<br>'John' |
| :--- | :--- | :--- | :--- |
| **i=1** | `{Det}` | `{NP}` | `∅` | `{S}` |
| **i=2** | | `{N}` | `∅` | `∅` |
| **i=3** | | | `{V}` | `{VP}` |
| **i=4** | | | | `{PN, NP}` |

Because the start symbol `S` is present in `table[1][4]`, the sentence "the dog chased John" is accepted by the grammar.

## Complexity Analysis

*   **Time Complexity:** O(n³ ⋅ |G|)
    *   The three nested loops for `L`, `i`, and `k` give the O(n³) factor.
    *   For each cell, we check all grammar rules (|G|), leading to the overall cubic complexity.
*   **Space Complexity:** O(n²)
    *   This is due to the storage requirement of the n x n parsing table.

This makes CYK one of the most efficient general-purpose CFG parsing algorithms, especially compared to backtracking methods.

## Advantages and Limitations

| Advantage | Limitation |
| :--- | :--- |
| **Efficiency:** Guaranteed polynomial time (O(n³)) due to dynamic programming. | **Grammar Restriction:** Requires the grammar to be in CNF. Conversion to CNF can make the grammar larger and less intuitive. |
| **Completeness:** Can parse any string belonging to a CFG (theoretically). | **No Ambiguity Handling:** The standard algorithm only checks membership. Extensions are needed to retrieve all possible parse trees for ambiguous sentences, which can be exponential. |
| **Deterministic:** Always produces the same result for the same input and grammar. | **Practical Use:** While foundational, many modern NLP parsers use more sophisticated algorithms (e.g., Earley, chart parsing) that don't require CNF and can handle left-recursion more naturally. |

## Exam Tips

1.  **CNF is Key:** Always ensure the grammar is in CNF before starting. If an exam question provides a CFG, your first step should be to convert it. Memorize the two allowed rule forms.
2.  **Trace Carefully:** When asked to trace the algorithm, draw the table clearly. Label your rows (`i`) and columns (`j`) correctly. Double-check the initialization step—it's a common source of error (remember to add all non-terminals that produce a terminal, including those from chain rules like `NP -> PN`).
3.  **Understand the Logic:** Be prepared to explain *why* the algorithm works—the principle of combining solutions to smaller subproblems (dynamic programming) and how the table indices relate to word positions in the sentence.
4.  **Membership vs. Parsing:** Know that the basic CYK algorithm answers "yes/no" for membership. Understand that building parse trees is an extension where you store back-pointers in the table cells.
5.  **Complexity:** Remember the time and space complexity (O(n³) and O(n²)) and be able to justify them based on the algorithm's loops and data structure.
