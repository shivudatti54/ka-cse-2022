# Module 2: Top-down and Bottom-up Parsing

## Introduction

In the process of compiling a program, **parsing** (or **syntax analysis**) is the phase where the compiler verifies the syntactic structure of the source code. It uses the tokens generated by the lexical analyzer and checks them against the rules of a context-free grammar (CFG). Parsing strategies are broadly classified into two categories: **Top-down Parsing** and **Bottom-up Parsing**. Understanding these strategies is fundamental to compiler design and natural language processing.

---

## Core Concepts

### 1. Top-down Parsing

A top-down parser starts with the root node of the parse tree (the start symbol of the grammar) and works its way downward to the leaves (the input tokens). It attempts to construct the parse tree from the top, applying grammar rules to expand non-terminals until it matches the entire input string. The most common and intuitive form of top-down parsing is **recursive descent parsing**.

**Key Characteristics:**
*   **Direction:** Starts from the start symbol (root) and moves towards the input string (leaves).
*   **Approach:** It is a goal-oriented approach. The goal is to find a derivation `S =>* input_string`.
*   **Types:** Recursive Descent Parsing, LL Parsers (e.g., LL(1)).

**Example:**
Consider the grammar for addition:
`E -> E + T | T`
`T -> id`

This grammar is left-recursive and cannot be directly used by a top-down parser. We first eliminate left-recursion to get:
`E -> T E'`
`E' -> + T E' | ε`
`T -> id`

Now, to parse the input `id + id`:

1.  Start with `E`.
2.  Apply `E -> T E'`. Now we have `T E'`.
3.  Apply `T -> id`. Now we have `id E'`. The first token `id` matches.
4.  Apply `E' -> + T E'`. Now we have `id + T E'`. The `+` token matches.
5.  Apply `T -> id`. Now we have `id + id E'`. The `id` token matches.
6.  Apply `E' -> ε` (epsilon, meaning empty). The parse is complete: `id + id`.

**Challenge:** Top-down parsers struggle with **left-recursive grammars** (like the original one above) because they can lead to infinite loops. They also require **left-factoring** to avoid backtracking in many cases.

### 2. Bottom-up Parsing

A bottom-up parser starts with the input tokens (the leaves of the parse tree) and works its way upward to the root (the start symbol). It attempts to reduce the input string step-by-step back to the start symbol by applying grammar rules in reverse. The most common type of bottom-up parsing is **shift-reduce parsing**, which forms the basis for powerful LR parsers (e.g., SLR, CLR, LALR).

**Key Characteristics:**
*   **Direction:** Starts from the input string (leaves) and moves towards the start symbol (root).
*   **Approach:** It is a data-oriented approach. The parser shifts tokens onto a stack and reduces them when a handle (the right side of a production) is found on top of the stack.
*   **Types:** Shift-Reduce Parsing, LR Parsers, Operator-Precedence Parsing.

**Example:**
Let's use the same left-recursive grammar (which bottom-up parsers can handle naturally):
`E -> E + T | T`
`T -> id`

To parse the input `id + id`:

| Stack      | Input       | Action                 |
| :--------- | :---------- | :--------------------- |
| `$`        | `id + id $` | Shift                  |
| `$ id`     | `+ id $`    | **Reduce** by `T -> id` |
| `$ T`      | `+ id $`    | **Reduce** by `E -> T`  |
| `$ E`      | `+ id $`    | Shift                  |
| `$ E +`    | `id $`      | Shift                  |
| `$ E + id` | `$`         | **Reduce** by `T -> id` |
| `$ E + T`  | `$`         | **Reduce** by `E -> E + T` |
| `$ E`      | `$`         | Accept                 |

The actions are: shift tokens onto the stack, and when the top of the stack matches the right-hand side (RHS) of a grammar rule, reduce it to the rule's left-hand side (LHS). This builds the parse tree from the bottom up.

**Advantage:** Bottom-up parsers are more powerful and can handle a wider range of grammars, including left-recursive ones, which are common in programming languages.

## Key Points and Summary

| Aspect              | Top-down Parsing                              | Bottom-up Parsing                               |
| ------------------- | --------------------------------------------- | ----------------------------------------------- |
| **Direction**       | From root (start symbol) to leaves (tokens)   | From leaves (tokens) to root (start symbol)     |
| **Approach**        | Goal-oriented (Expand non-terminals)          | Data-oriented (Reduce terminals to non-terminals) |
| **Grammar**         | Prefers non-left-recursive, left-factored grammars | Can handle left-recursive and wider range of grammars |
| **Power**           | Less powerful (e.g., LL parsers)              | More powerful (e.g., LR parsers)                |
| **Implementation**  | Relatively simpler (Recursive descent)        | Complex, often requires automated parser generators (e.g., YACC/Bison) |
| **Example**         | Recursive Descent Parser, LL(1) Parser        | Shift-Reduce Parser, LR Parser, SLR Parser      |

**Summary:**
*   **Top-down parsing** is intuitive and easier to implement by hand but is restricted to a narrower class of grammars.
*   **Bottom-up parsing** is more powerful and used in most real-world compilers due to its ability to handle complex grammars efficiently, though it requires more sophisticated construction algorithms.
*   The choice between them depends on the grammar of the language and the required efficiency and power of the compiler.
