# Memory Management

## Introduction

Memory management is one of the most critical functions of any operating system. In a computer system, the main memory (RAM) is a finite, volatile resource that must be shared among multiple processes. The operating system acts as a memory manager, responsible for allocating and deallocating memory space to processes, protecting memory regions from unauthorized access, and ensuring efficient utilization of available memory.

The importance of memory management cannot be overstated in modern computing. When multiple programs run concurrently on a system, they compete for the limited physical memory available. Without proper memory management, processes could overwrite each other's data, leading to system crashes, security vulnerabilities, and inefficient resource utilization. Memory management techniques enable virtualization of memory, giving each process the illusion of having a large, contiguous address space even though physical memory may be fragmented across different locations.

This chapter explores the fundamental concepts of memory management, including the distinction between logical and physical addresses, various memory allocation strategies, paging and segmentation mechanisms, and the revolutionary concept of virtual memory that enables programs to execute even when they exceed the available physical memory.

## Key Concepts

### 1. Address Space and Address Binding

In any computer system, programs are stored on disk as executable files. When a program is executed, its code and data must be loaded into memory. The program references memory using addresses, and the process of converting these program-level addresses to actual physical memory addresses is called address binding.

There are three types of addresses encountered in memory management:
- **Symbolic addresses**: Variable names and function names used in source code
- **Relative addresses**: Addresses expressed as offsets from some base value
- **Physical addresses**: The actual hardware addresses in memory

Address binding can occur at three different times:
- **Compile time**: If the memory location is known beforehand, absolute addresses are generated. This is inflexible and rarely used in modern systems.
- **Load time**: The compiler generates relocatable code, and the loader binds these to physical addresses when the program is loaded into memory.
- **Execution time**: Binding is delayed until the program is actually running. This allows processes to be moved in memory during execution, which is essential for virtual memory systems.

### 2. Logical vs Physical Address Space

Modern operating systems distinguish between logical (virtual) addresses and physical addresses. A logical address is generated by the CPU and used by programs, while a physical address is the actual location in RAM.

The range of logical addresses that a process can use is called its logical address space, while the range of physical addresses available for storing data is the physical address space. In systems with virtual memory, the logical address space is typically much larger than the physical address space, allowing programs to use more memory than physically available.

The Memory Management Unit (MMU) is a hardware component that translates logical addresses to physical addresses dynamically during program execution. This translation is transparent to the user program, which simply uses logical addresses as if they were physical.

### 3. Memory Allocation Methods

Operating systems employ several strategies for allocating memory to processes:

**Contiguous Memory Allocation**
In this simplest approach, each process is stored in a single, contiguous block of memory. The operating system maintains a list of free memory holes and allocates memory using one of three strategies:
- **First-fit**: Allocate the first hole large enough to accommodate the process
- **Best-fit**: Allocate the smallest hole that is large enough, minimizing wasted space
- **Worst-fit**: Allocate the largest available hole, leaving larger remaining holes

Contiguous allocation suffers from external fragmentation, where free memory is broken into small non-contiguous pieces that cannot satisfy large requests. Internal fragmentation occurs when allocated memory is slightly larger than requested, wasting space within allocated blocks.

### 4. Paging

Paging is a memory management scheme that eliminates external fragmentation by dividing both logical memory and physical memory into fixed-sized blocks called pages and frames respectively.

The logical address space is divided into pages of equal size (typically 4KB), and physical memory is divided into frames of the same size. When a process needs memory, whole pages are allocated to it regardless of the actual data being stored. The page table, maintained by the operating system, maps each virtual page number to a physical frame number.

A logical address in a paged system consists of two parts: the page number and the offset within that page. The MMU uses the page number as an index into the page table to find the corresponding frame number, then combines this with the offset to produce the physical address.

Paging introduces internal fragmentation because a page must be allocated even if the process needs only part of it. However, since page sizes are typically small (4KB-8KB), the internal fragmentation is limited to less than one page per allocation.

### 5. Structure of Page Table

The page table is a data structure that maintains the mapping between virtual pages and physical frames. Several optimization techniques exist for page table storage:

**Hierarchical Paging**
For large address spaces, the page table itself can be paged. A two-level page table divides the virtual address into three parts: the outer page table index, the inner page table index, and the offset. This reduces the memory overhead of maintaining large page tables.

**Inverted Page Table**
Instead of maintaining a page table entry for every virtual page, an inverted page table maintains one entry for each physical frame. This significantly reduces memory overhead for large address spaces but increases the time to look up a virtual address.

**Hashed Page Tables**
A hashed page table uses a hash function to map virtual page numbers to entries in the page table. This is particularly useful for sparse address spaces where only a small fraction of virtual addresses are used.

### 6. Translation Lookaside Buffer (TLB)

Every memory access in a paged system requires a page table lookup, which would significantly slow down memory operations if performed for every access. The Translation Lookaside Buffer (TLB) is a special hardware cache that stores recent page table translations.

When the CPU generates a logical address, the MMU first checks the TLB for the corresponding physical address. If found (a TLB hit), the translation is completed quickly. If not found (a TLB miss), the page table must be consulted, and the new translation is stored in the TLB for future use. TLB hit rates of 95% or higher are common in practice.

### 7. Segmentation

Segmentation is a memory management technique that divides memory into variable-sized segments based on logical divisions of a program, such as code, data, stack, and heap. Each segment has a name (or number) and a length, and logical addresses are specified as segment name plus offset.

Unlike paging, which is invisible to the programmer, segmentation reflects the logical structure of a program. Each segment can have its own protection attributes (read, write, execute permissions), and segments can be shared between processes.

The physical address in a segmented system is computed by adding the base address of the segment (obtained from the segment table) to the offset. Segmentation can be combined with paging for optimal memory management.

### 8. Virtual Memory

Virtual memory is perhaps the most important innovation in memory management. It allows programs to execute even when they require more memory than is physically available in the system. Virtual memory creates the illusion of a large, contiguous address space by using disk space as an extension of RAM.

The key mechanism behind virtual memory is demand paging, where pages are loaded into memory only when they are needed. When a process accesses a page that is not currently in memory (a page fault), the operating system must bring that page from disk into a free frame. If no free frames exist, a page must be evicted (using a replacement algorithm like LRU or FIFO) to make room.

Virtual memory provides several crucial benefits:
- Programs can be larger than physical memory
- More processes can run concurrently since each uses only a fraction of its address space
- Memory protection is enforced between processes
- Programs can be loaded at any memory location (relocation becomes trivial)

## Examples

### Example 1: Address Translation in Paging

Consider a system with 16-bit logical addresses and physical memory of 64KB. If we use a page size of 4KB, determine:
- Number of bits for page number
- Number of bits for offset
- Size of page table if each entry is 4 bytes

**Solution:**

Page size = 4KB = 2^12 bytes, so offset requires 12 bits.

Logical address = 16 bits, so page number bits = 16 - 12 = 4 bits

Number of pages = 2^4 = 16 pages

Each page table entry = 4 bytes

Page table size = 16 × 4 = 64 bytes

Physical memory = 64KB = 2^16 bytes, so physical address has 16 bits.

Number of frames = 2^16 / 2^12 = 2^4 = 16 frames

### Example 2: Logical to Physical Address Translation

A system uses paging with page size 1024 bytes. The page table for a process is as follows (page number : frame number):

| Page | Frame |
|------|-------|
| 0    | 3     |
| 1    | 7     |
| 2    | 1     |
| 3    | 5     |

Convert the logical address 1529 to its physical address.

**Solution:**

Page size = 1024 = 2^10 bytes, so offset bits = 10

Logical address = 1529

Page number = 1529 / 1024 = 1 (integer division)

Offset = 1529 % 1024 = 1529 - 1024 = 505

From page table: Page 1 is in frame 7

Physical address = (7 × 1024) + 505 = 7168 + 505 = 7673

### Example 3: Calculating Page Faults

A process accesses pages in the following order: 1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5

Using the FIFO page replacement algorithm with 3 frames, calculate the number of page faults.

**Solution:**

Initially all frames are empty.

Access 1: Page 1 loaded into frame 1 (Page fault: 1)
Access 2: Page 2 loaded into frame 2 (Page fault: 2)
Access 3: Page 3 loaded into frame 3 (Page fault: 3)
Access 4: Page 4 needed, no free frames. Replace oldest (page 1). Page 4 loaded into frame 1 (Page fault: 4)
Access 1: Page 1 needed, already in memory (No fault)
Access 2: Page 2 already in memory (No fault)
Access 5: Page 5 needed. Replace oldest (page 2 in frame 2). Page 5 loaded (Page fault: 5)
Access 1: Page 1 already in memory (No fault)
Access 2: Page 2 needed. Replace oldest (page 3 in frame 3). Page 2 loaded (Page fault: 6)
Access 3: Page 3 needed. Replace oldest (page 4 in frame 1). Page 3 loaded (Page fault: 7)
Access 4: Page 4 needed. Replace oldest (page 1 in frame 3 - loaded at time 4). Page 4 loaded (Page fault: 8)
Access 5: Page 5 already in memory (No fault)

Total page faults = 8

## Exam Tips

1. UNDERSTAND THE DIFFERENCE between logical and physical addresses clearly. In exams, always identify whether a given address is logical or physical before attempting translation.

2. FOR PAGING PROBLEMS, remember that page number = logical address divided by page size (integer division), and offset = logical address modulo page size.

3. KNOW THE PAGE TABLE ENTRY FORMAT typically includes: valid/invalid bit, protection bits, modified (dirty) bit, referenced bit, and frame number.

4. PAGE FAULT CALCULATIONS are common exam questions. Remember that a page fault occurs when the requested page is not in memory (not in any frame).

5. FOR SEGMENTATION, remember that a logical address consists of segment number and offset, and the segment table contains base address and limit for each segment.

6. WHEN COMBINING SEGMENTATION AND PAGING, the logical address has three parts: segment number, page number within segment, and offset within page.

7. VIRTUAL MEMORY BENEFITS are frequently asked: program size independence, process isolation, simplified memory allocation, and efficient process creation.

8. TLB IS A HARDWARE CACHE - it does not store actual data, only address translations. A TLB miss requires accessing the page table in memory.

9. REMEMBER THAT PAGE SIZE affects both internal fragmentation and page table size. Larger pages mean fewer page table entries but more internal fragmentation.
