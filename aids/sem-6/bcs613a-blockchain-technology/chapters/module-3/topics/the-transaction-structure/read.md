# Module 3: Blockchain Transaction Structure

## Introduction

In traditional systems, a transaction is simply a record of a debit or credit. However, in blockchain, a transaction is the fundamental unit that drives the entire ecosystem. It represents a state change—a transfer of value (like cryptocurrency) or data (like a smart contract function call) from one entity to another. Understanding the intricate structure of a transaction is crucial to grasping how blockchain achieves its core properties of decentralization, immutability, and transparency. This note deconstructs the anatomy of a typical transaction in a cryptocurrency-based blockchain like Bitcoin.

## Core Concepts of a Transaction Structure

A transaction is not a single monolithic block of data; it is a structured message containing specific fields. Every transaction must be digitally signed to prove ownership of the funds being spent and must follow a strict set of rules to be considered valid by the network.

### 1. Core Components of a Transaction

A standard transaction (like in Bitcoin) consists of the following key parts:

*   **Transaction ID (TxID):** This is a unique hash identifier generated by cryptographically hashing the entire transaction data. It acts as a digital fingerprint for that specific transaction. Any alteration in the transaction data would completely change this hash, making tampering evident.
*   **Version Number:** This specifies which set of consensus rules this transaction follows. It allows the protocol to evolve over time with new features (e.g., SegWit) while maintaining backward compatibility.
*   **Input(s):** Inputs prove the origin of the funds. You cannot create value out of thin air; you must prove you received coins from a previous transaction.
    *   Each input refers to the **TxID** of a previous transaction where you received funds.
    *   It also contains an **Index (vOut)**, which specifies *which output* from that previous transaction is being spent (as a transaction can have multiple outputs).
    *   It includes an **Unlocking Script (ScriptSig)**, which is a cryptographic puzzle that provides authorization to spend the funds. This typically contains a digital signature and a public key, proving ownership of the private key that can unlock the output.
*   **Output(s):** Outputs define the destination and amount of the funds being sent.
    *   Each output contains an **Amount** (the quantity of cryptocurrency being sent).
    *   It also contains a **Locking Script (ScriptPubKey)**, which is a cryptographic puzzle that specifies the conditions that must be met to spend this output in the future. It is essentially the recipient's address (which is a hash of their public key).
*   **Witness Data (in SegWit transactions):** Introduced as a upgrade, this field segregates the unlocking script (signature) from the transaction data. This helps mitigate transaction malleability and increases block capacity.
*   **Locktime:** This field defines the earliest time or block number at which the transaction can be added to the blockchain. If set to zero, it is valid immediately.

### 2. The Lifecycle of a Transaction: An Example

Let's follow a simple example where **Alice sends 1 BTC to Bob**.

1.  **Creation:** Alice's wallet software constructs a new transaction.
    *   **Input:** It references a previous transaction where Alice received 2 BTC (TxID: `abc123...`), specifically the first output (Index: `0`).
    *   **Output 1:** `1 BTC` locked to Bob's public key hash (his address).
    *   **Output 2:** `0.99 BTC` locked back to Alice's own address (this is her **change**; the `0.01 BTC` difference is the network **transaction fee**).
    *   The transaction is signed with Alice's private key, creating the `ScriptSig`.

2.  **Propagation:** Alice's wallet broadcasts this signed transaction to the peer-to-peer network.

3.  **Validation:** Network nodes (miners) receive the transaction and validate it against a long list of rules:
    *   Is the signature valid? Does it prove ownership of the input?
    *   Are the input UTXOs (Unspent Transaction Outputs) real and unspent? (Prevents double-spending).
    *   Is the sum of inputs >= the sum of outputs? (Prevents creating coins).
    *   Does it follow the correct script rules?

4.  **Mining:** Once validated, miners include the transaction in a new candidate block. They perform the Proof-of-Work to find a valid hash for this block.

5.  **Confirmation:** When a miner solves the block, it is propagated and added to the blockchain. After subsequent blocks are built on top of it, the transaction is considered confirmed.

### 3. UTXO Model: The Foundation

It's vital to understand the **Unspent Transaction Output (UTXO)** model. The blockchain does not track account balances like a bank. Instead, it tracks chains of transactions. Your "balance" is the sum of all UTXOs (outputs from previous transactions) that are locked to your address and have not yet been spent as an input in a new transaction. A transaction simply consumes existing UTXOs and creates new ones.

## Key Points & Summary

| Concept | Description |
| :--- | :--- |
| **Purpose** | A transaction is a signed data structure that facilitates the transfer of value/data and effects a state change on the ledger. |
| **Structure** | Comprises Metadata (Version, TxID), Inputs (referencing previous outputs), Outputs (creating new locked outputs), and Locktime. |
| **Inputs** | Prove the source of funds by referencing and unlocking previous UTXOs via a signature (`ScriptSig`). |
| **Outputs** | Create new UTXOs by specifying an amount and a locking condition (`ScriptPubKey`) for the recipient. |
| **Validation** | Nodes check cryptographic signatures, UTXO status, and consensus rules to prevent double-spending and invalid transactions. |
| **UTXO Model** | The accounting model where a user's balance is the sum of all unspent outputs addressed to them. |
| **Immutability** | Once confirmed and buried under subsequent blocks, altering a transaction is computationally infeasible. |

In conclusion, the transaction is the atomic unit of a blockchain. Its meticulously designed structure, combined with cryptographic proofs and a decentralized consensus mechanism, is what enables trustless, secure, and transparent peer-to-peer transactions without the need for a central authority.
