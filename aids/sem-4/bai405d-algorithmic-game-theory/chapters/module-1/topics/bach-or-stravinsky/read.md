# Batch vs Streaming Processing

## Introduction to Data Processing Paradigms

In the realm of Big Data, efficiently handling vast amounts of information is paramount. Two fundamental data processing paradigms have emerged to tackle this challenge: **Batch Processing** and **Stream Processing**. Understanding the distinction between them, their use cases, advantages, and limitations is a cornerstone of Big Data architecture. The choice between batch and streaming often hinges on the specific requirements of an application, particularly concerning the **Velocity** aspect of Big Data's 5 Vs.

## What is Batch Processing?

Batch processing is a method of processing data where a large volume of data (a "batch") is collected over a period of time, and then the entire dataset is processed together in a single, scheduled operation.

### Key Characteristics of Batch Processing

- **Bounded Data:** Batch jobs operate on finite, complete datasets. The data has a defined beginning and end.
- **High Latency:** There is a significant delay between data being generated and the results of its processing being available. This latency can range from minutes to hours or even days.
- **High Throughput:** Designed to handle very large volumes of data efficiently by leveraging distributed computing frameworks.
- **Scheduled Execution:** Jobs are typically initiated on a schedule (e.g., hourly, daily, weekly) or triggered once a certain amount of data has accumulated.

### How Batch Processing Works

1.  **Data Accumulation:** Data is generated by various sources (application logs, user transactions, sensor readings) and stored in a persistent storage system like a database, data lake (e.g., HDFS, S3), or files.
2.  **Job Scheduling:** A processing job is scheduled to run at a specific time or interval.
3.  **Job Execution:** The processing engine (e.g., Hadoop MapReduce, Spark) reads the entire dataset, performs the required computations (filtering, aggregation, transformation), and writes the results to an output sink.
4.  **Result Availability:** The output is available after the job completes.

```
+---------------+      +------------------+      +-----------------+      +---------------+
| Data Sources  | ---> | Persistent Storage| ---> | Batch Processing| ---> | Results /     |
| (Logs, DB, etc)|      | (HDFS, S3, DB)   |      | Job (Spark, MR) |      | Data Warehouse|
+---------------+      +------------------+      +-----------------+      +---------------+
       |                       ^                       |                         ^
       |                       |                       |                         |
       | (Continuous Data Ingestion)           (Scheduled Trigger)       (Available after job completion)
       |                       |                       |                         |
       +-----------------------+-----------------------+-------------------------+
```

**Example:** A retail company aggregates all sales transactions from the day. Every night at 2:00 AM, a batch job runs to calculate total daily sales, update inventory levels, and generate daily sales reports.

### Batch Processing Frameworks

- **Apache Hadoop MapReduce:** The original batch processing framework in the Hadoop ecosystem.
- **Apache Spark:** While capable of streaming, Spark's core engine is designed for high-performance batch processing using in-memory computation.
- **Apache Hive:** A data warehouse software built on Hadoop that facilitates querying and managing large datasets using SQL-like language (HQL), often executed as batch jobs.
- **Apache Pig:** A high-level platform for creating MapReduce programs used for data transformations.

## What is Streaming Processing?

Stream processing, also known as real-time processing, is a method where data is processed continuously as it is generated, often within milliseconds or seconds. It deals with unbounded, continuous data streams.

### Key Characteristics of Streaming Processing

- **Unbounded Data:** Data streams are theoretically infinite and have no predefined end.
- **Low Latency:** Results are available almost immediately after the data is created, enabling real-time insights and actions.
- **Continuous Execution:** Applications are always running, processing events as they arrive.
- **Stateful Processing:** Often requires maintaining state (e.g., counts, averages) over time to perform computations like windowing.

### How Streaming Processing Works

1.  **Data Generation:** Data is emitted as a continuous stream of events from sources like IoT sensors, user activity on a website, or financial trading systems.
2.  **Ingestion:** The data stream is ingested by a messaging/streaming platform (e.g., Apache Kafka, Amazon Kinesis) which acts as a buffer and ensures durability.
3.  **Continuous Processing:** A streaming engine (e.g., Apache Flink, Spark Streaming) consumes the events from the messaging platform, processes them incrementally, and updates the results.
4.  **Real-Time Results:** Results are updated in real-time and can be sent to dashboards, databases, or other downstream systems for immediate action.

```
+----------------+      +-----------------------+      +---------------------+      +---------------+
| Data Sources   | ---> | Messaging/Streaming   | ---> | Streaming Processing | ---> | Real-Time     |
| (IoT, Clicks)  |      | Platform (Kafka)      |      | Engine (Flink, Spark)|      | Dashboard/DB  |
+----------------+      +-----------------------+      +---------------------+      +---------------+
       |                         ^                           |                             ^
       | (Continuous Event Flow) | (Continuous Consumption)  | (Continuous Result Update)  |
       |                         |                           |                             |
       +-------------------------+---------------------------+-----------------------------+
```

**Example:** A fraud detection system for a credit card company. Every transaction is analyzed as it happens. The system checks the transaction amount, location, and purchase history in real-time to score the likelihood of fraud and can block a transaction within milliseconds if it's deemed suspicious.

### Streaming Processing Frameworks

- **Apache Flink:** A true stream-processing framework designed for low latency and high performance, handling state and time elegantly.
- **Apache Kafka Streams:** A client library for building applications and microservices that process data stored in Kafka topics.
- **Spark Streaming:** An extension of the core Spark API that processes data in micro-batches, providing a batch-like API for streaming.
- **Apache Samza:** A framework for stream processing that is often used with Kafka.

## Key Differences: Batch vs. Streaming

| Aspect              | Batch Processing                             | Streaming Processing                          |
| ------------------- | -------------------------------------------- | --------------------------------------------- |
| **Data Scope**      | Bounded (finite dataset)                     | Unbounded (continuous stream)                 |
| **Data Size**       | Large volumes (Terabytes-Petabytes)          | Individual events or small micro-batches      |
| **Latency**         | High (minutes to hours)                      | Low (milliseconds to seconds)                 |
| **Throughput**      | Higher overall throughput                    | Slightly lower per-event overhead             |
| **Analysis Type**   | Complex, comprehensive analysis              | Simple, real-time monitoring, alerting        |
| **Cost**            | Generally more cost-effective for large jobs | Can be more expensive due to constant compute |
| **Use Cases**       | ETL, nightly reports, historical analysis    | Fraud detection, alerts, live dashboards      |
| **Fault Tolerance** | Easier; recompute entire batch on failure    | More complex; requires state checkpointing    |

## The Lambda and Kappa Architectures

These are architectural patterns designed to harness the strengths of both batch and streaming systems.

### Lambda Architecture

The Lambda Architecture is designed to handle massive quantities of data by taking advantage of both batch and stream-processing methods. This approach attempts to balance latency, throughput, and fault tolerance.

- **Batch Layer (Apache Hadoop/Spark):** Manages the master dataset and pre-comutes batch views. Provides accurate and comprehensive views of historical data. High latency.
- **Speed Layer (Apache Flink/Storm):** Compensates for the high latency of the batch layer by processing data streams in real-time to provide real-time views. Low latency but may be approximate.
- **Serving Layer:** Merges the results from the batch and speed layers to answer queries.

```
                    +-----------------+
                    |    Query       |
                    +-----------------+
                            |
            +---------------+---------------+
            |                               |
    +----------------+             +------------------+
    |   Batch Layer  |             |   Speed Layer    |
    | (Pre-computation)|             | (Real-Time       |
    +----------------+             +------------------+
            |                               |
    +----------------+             +------------------+
    | Master Dataset |             |   Stream Data    |
    | (Data Lake)    |             |   (Kafka)        |
    +----------------+             +------------------+
```

### Kappa Architecture

The Kappa Architecture is a simplification of the Lambda Architecture. It proposes a single pipeline using a stream processing system for all data processing.

- All data is treated as a stream.
- Historical data is re-processed by re-playing it from the streaming platform (e.g., Kafka, which retains data for long periods).
- This avoids maintaining two separate codebases for batch and speed layers.

```
                    +-----------------+
                    |    Query       |
                    +-----------------+
                            |
                    +-----------------+
                    | Stream Processor|
                    | (e.g., Flink)   |
                    +-----------------+
                            |
                    +-----------------+
                    | Streaming Platform|
                    | (e.g., Kafka)    |
                    +-----------------+
                            |
                    +-----------------+
                    |   All Data      |
                    |   as a Stream   |
                    +-----------------+
```

## Choosing Between Batch and Streaming

The decision is not always mutually exclusive. Many modern data systems use a hybrid approach.

**Choose Batch Processing when:**

- Processing very large datasets where latency is not critical.
- Performing complex analytics that require full data visibility (e.g., training a machine learning model).
- Cost-effectiveness is a primary concern.

**Choose Stream Processing when:**

- Immediate action is required based on new data (e.g., fraud detection, system alerts).
- The application involves real-time monitoring or live dashboards.
- The data has a very short shelf-life and loses value quickly.

## Exam Tips

1.  **Focus on Latency:** The most straightforward differentiator is latency. If the question implies a need for immediate results, think "streaming". If it talks about end-of-day reports or large historical datasets, think "batch".
2.  **Know the Frameworks:** Be able to associate popular frameworks with their primary processing type (e.g., MapReduce -> Batch, Flink -> Streaming, Spark -> Both).
3.  **Understand Bounded vs. Unbounded:** This is the core conceptual difference. Batch handles bounded data; streaming handles unbounded data.
4.  **Architecture Patterns:** Be prepared to explain the Lambda and Kappa architectures, their components, and their trade-offs (complexity vs. simplicity).
5.  **Real-World Examples:** Memorize a few clear examples for each type (e.g., Batch: daily sales report; Streaming: live traffic updates) to use in explanation questions.
