## 1. Introduction and Real-World Relevance

An **interrupt** is a signal generated by hardware or software that temporarily halts the normal execution of a CPU, compelling it to divert its attention to a high-priority event. The **Program Interrupt Cycle** represents the complete sequence of operations that occurs from the moment an interrupt is detected until the CPU resumes normal execution. This mechanism is fundamental to modern computing systems, enabling efficient handling of asynchronous events without continuous polling.

### Real-World Applications

- **Operating System Functioning**: System calls, context switching, and process management rely heavily on interrupts
- **I/O Operations**: Disk controllers, network cards, and peripheral devices use interrupts to signal data readiness
- **Real-Time Systems**: Mission-critical applications (medical devices, automotive control systems) require predictable interrupt response
- **Multitasking**: CPU scheduler uses timer interrupts to implement time-sharing
- **Error Handling**: Divide-by-zero, page faults, and memory protection violations trigger exceptions

## 2. Fundamentals of the Interrupt Cycle

### 2.1 What is an Interrupt Cycle?

The interrupt cycle is not a separate execution phase but rather a mechanism that interrupts the normal fetch-decode-execute cycle. When an interrupt occurs:

1. **CPU completes the current instruction**
2. **Interrupt detection and acknowledgment**
3. **Save current state** (program counter, status flags)
4. **Transfer control to interrupt service routine (ISR)**
5. **Execute ISR**
6. **Restore state and resume execution**

### 2.2 The Complete Interrupt Flow

```
┌─────────────────────────────────────────────────────────────────┐
│                    NORMAL EXECUTION                            │
│  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐                       │
│  │Fetch │→ │Decode│→ │Execute│→ │Store │                       │
│  └──────┘  └──────┘  └──────┘  └──────┘                       │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼ (Interrupt Occurs)
┌─────────────────────────────────────────────────────────────────┐
│                 INTERRUPT CYCLE                                │
│  1. Complete Current Instruction                               │
│  2. Save PC and Status Register                                │
│  3. Load Interrupt Vector into PC                             │
│  4. Fetch ISR (Interrupt Service Routine)                    │
│  5. Execute ISR                                                │
│  6. Restore PC and Status Register                            │
│  7. Resume Normal Execution                                    │
└─────────────────────────────────────────────────────────────────┘
```

### 2.3 Key Components Involved

| Component | Description |
|-----------|-------------|
| **Interrupt Request Line (IRQ)** | Physical line connecting devices to CPU |
| **Interrupt Controller** | Hardware that manages multiple interrupt sources |
| **Interrupt Vector Table** | Memory table containing addresses of ISRs |
| **Program Counter (PC)** | Holds address of next instruction to execute |
| **Status Register/Flags** | Contains condition codes and interrupt enable bits |

## 3. Classification of Interrupts

### 3.1 Hardware Interrupts (Asynchronous)

Generated by external hardware devices:

- **External Interrupts**: Keyboard, mouse, disk I/O, network cards
- **Timer Interrupts**: System clock (used for task scheduling)
- **I/O Completion Interrupts**: Data transfer completion signals

### 3.2 Software Interrupts (Synchronous)

Generated by the CPU itself during instruction execution:

- **Traps**: Intentional interrupts for system calls (e.g., `INT 0x80` on x86)
- **Exceptions**: Unplanned events like:
  - **Faults**: Page fault, divide-by-zero (can be corrected)
  - **Traps**: Debug breakpoints (INT 3)
  - **Aborts**: Fatal hardware errors (machine check)

### 3.3 Differences: Hardware vs Software Interrupts

| Aspect | Hardware Interrupt | Software Interrupt |
|--------|-------------------|-------------------|
| **Source** | External devices | CPU executing INT instruction |
| **Timing** | Asynchronous | Synchronous (predictable) |
| **Priority** | Configurable via PIC/APIC | Fixed (determined by vector) |
| **Examples** | IRQ0-IRQ15, IRQ0-IRQ23 | INT n, SYSCALL, SYSENTER |
| **Stack** | Kernel stack (usually) | Can use user stack or kernel stack |

## 4. Interrupt Controllers: PIC and APIC

### 4.1 Programmable Interrupt Controller (PIC)

The **Intel 8259A PIC** (used in older x86 systems) manages up to 8 interrupt request lines. Cascading two 8259A chips provides 15 IRQ lines (IRQ0-IRQ7, IRQ8-IRQ15).

**Key Features:**
- **Interrupt Mask Register (IMR)**: Disables individual interrupt sources
- **Interrupt Service Request Register (IRRR)**: Tracks pending interrupts
- **In-Service Register (ISR)**: Tracks interrupts being serviced
- **Priority resolver**: Determines highest priority pending interrupt

**Working Principle:**
1. Device raises interrupt on IRQ line
2. PIC identifies the highest-priority request
3. PIC sends signal to CPU (INTR pin)
4. CPU acknowledges via INTA (Interrupt Acknowledge)
5. PIC provides interrupt vector number
6. CPU jumps to corresponding ISR

```assembly
; Example: PIC Programming (x86 Real Mode)
; Initialize PIC with ICW1 (Initialization Command Word 1)
mov al, 0x11        ; ICW1: Edge-triggered, cascaded, ICW4 needed
out 0x20, al        ; Send to PIC1 (IRQs 0-7)
out 0xA0, al        ; Send to PIC2 (IRQs 8-15)

; ICW2: Set vector offset
mov al, 0x20        ; IRQ0 maps to vector 0x20
out 0x21, al        ; PIC1
mov al, 0x28        ; IRQ8 maps to vector 0x28
out 0xA1, al        ; PIC2

; ICW3: Configure cascading
mov al, 0x04        ; PIC1 has slave on IRQ2
out 0x21, al
mov al, 0x02        ; PIC2 is slave, connected to IRQ2
out 0xA1, al
```

### 4.2 Advanced PIC (APIC)

Modern systems use **APIC (Advanced Programmable Interrupt Controller)** to overcome PIC limitations:

**Components:**
- **Local APIC (LAPIC)**: Integrated in CPU, handles local interrupts (timer, thermal, performance counters)
- **I/O APIC**: Manages external interrupts, supports 24+ IRQ lines
- **System APIC (SAPIC)**: Enhanced version for large systems

**Advantages over PIC:**
- Supports **many more interrupt sources** (24-224+)
- **Message Signaled Interrupts (MSI)** - eliminates physical IRQ lines
- **Distributed interrupt handling** for multi-processor systems
- **Programmable priority** and **interrupt routing**
- Supports **interrupt affinity** (binding IRQs to specific CPUs)

```c
// Linux Kernel: Setting Interrupt Affinity (simplified)
#include <pthread.h>
#include <stdio.h>
#include <sched.h>

int set_irq_affinity(unsigned int irq, cpu_set_t *mask) {
    char irq_path[64];
    sprintf(irq_path, "/proc/irq/%d/smp_affinity", irq);
    
    FILE *f = fopen(irq_path, "w");
    if (!f) return -1;
    
    // Write CPU mask (e.g., "4" means CPU 2)
    fprintf(f, "%lx", *mask);
    fclose(f);
    return 0;
}

int main() {
    cpu_set_t cpuset;
    CPU_ZERO(&cpuset);
    CPU_SET(2, &cpuset);  // Bind to CPU 2
    
    // Set IRQ 46 (common for network cards) to CPU 2
    set_irq_affinity(46, &cpuset);
    return 0;
}
```

## 5. Vectored Interrupts

In **vectored interrupt systems**, each interrupt source is assigned a unique **interrupt vector** (an address or offset) that points to its specific **Interrupt Service Routine (ISR)**.

### 5.1 Interrupt Vector Table

The **Interrupt Vector Table (IVT)** is a data structure containing pointers to ISRs:

- **x86 Real Mode**: 256 entries, each 4 bytes (segment:offset), located at memory address 0x00000-0x0003FF
- **x86 Protected Mode**: **Interrupt Descriptor Table (IDT)** with 256 entries, each 8 bytes (gate descriptor)
- **ARM**: Vector table at fixed address (0x00000000 or 0xFFFF0000), can be relocated

### 5.2 Vector Assignment (x86 Example)

| Vector | IRQ | Interrupt Source |
|--------|-----|------------------|
| 0x00 | - | Divide Error |
| 0x01 | - | Debug Exception |
| 0x02 | - | NMI Interrupt |
| 0x03 | - | Breakpoint (INT 3) |
| 0x0D | - | General Protection Fault |
| 0x0E | - | Page Fault |
| 0x20 | IRQ0 | Programmable Interval Timer |
| 0x21 | IRQ1 | Keyboard Controller |
| 0x2E | IRQ6 | Floppy Disk Controller |

### 5.3 Interrupt Descriptor Table (IDT) - x86 Protected Mode

```c
// IDT Gate Descriptor Structure (64-bit mode)
struct idt_entry {
    uint16_t offset_low;    // Lower 16 bits of ISR address
    uint16_t selector;      // Code segment selector
    uint8_t  ist;           // Interrupt Stack Table
    uint8_t  type_attr;     // Type (32/64-bit interrupt/trap gate)
    uint16_t offset_mid;    // Middle 16 bits of ISR address
    uint32_t offset_high;   // Upper 32 bits of ISR address
    uint32_t reserved;      // Reserved
} __attribute__((packed));
```

## 6. Nested Interrupts and Priority Levels

### 6.1 Interrupt Priority

Interrupts are assigned **priority levels** to determine which interrupt can preempt another:

| Priority Level | Interrupt Type | Examples |
|----------------|---------------|-----------|
| 0 (Highest) | Machine Check/Abort | Hardware failure |
| 1 | Non-Maskable Interrupt (NMI) | Memory/parity errors |
| 2 | Hardware Faults | Page fault, TLB miss |
| 3 | External Interrupts | I/O devices |
| 4 | Software Interrupts | INT n instructions |
| 5 (Lowest) | Normal Program Execution | User applications |

### 6.2 Nested Interrupts

**Nested interrupts** occur when a higher-priority interrupt preempts a lower-priority ISR:

```c
// Pseudocode: Nested Interrupt Handling
void keyboard_ISR() {
    save_context();              // Save current process state
    enable_interrupts();         // Allow higher-priority interrupts
    
    // Read keyboard data
    char key = read_keyboard_buffer();
    process_key(key);
    
    disable_interrupts();        // Disable for restoration
    restore_context();           // Restore state and return
}

void timer_ISR() {
    // This can preempt keyboard_ISR if timer has higher priority
    save_context();
    update_system_clock();
    schedule_next_process();
    restore_context();
}
```

### 6.3 Interrupt Stack Frames

**x86 64-bit Interrupt Stack Frame:**
```
┌──────────────────────────────┐
│        RFLAGS               │ ← Pushed by CPU
├──────────────────────────────┤
│         RIP                  │ ← Pushed by CPU
├──────────────────────────────┤
│         CS                   │ ← Pushed by CPU
├──────────────────────────────┤
│         RBP                  │ ← Pushed by ISR (optional)
├──────────────────────────────┤
│    Function Arguments        │
├──────────────────────────────┤
│    Local Variables           │
└──────────────────────────────┘
```

### 6.4 Priority Handling Mechanisms

- **Fixed Priority**: Higher priority always preempts lower (simple, but can cause starvation)
- **Rotating Priority**: Priority rotates after each interrupt (fair distribution)
- **Polling**: CPU checks devices in fixed order (lowest latency control)
- ** Daisy Chain**: Devices prioritized by position in chain

## 7. Architecture-Specific Interrupt Handling

### 7.1 x86 Architecture

**Key Features:**
- **INTR pin**: Maskable interrupt input
- **NMI pin**: Non-maskable interrupt
- **INTA (Interrupt Acknowledge)**: CPU signals to acknowledge interrupt
- **APIC**: Modern interrupt controller
- **MSI**: Message Signaled Interrupts for PCIe devices

**x86 Linux Kernel Interrupt Flow:**
```
Hardware → LAPIC/I/O APIC → IDT → ISR (kernel function) → Device Driver Handler
```

### 7.2 ARM Architecture

**Key Features:**
- **Vector Table**: Contains branch instructions to handlers at fixed addresses
- **PSR (Program Status Register)**: Contains interrupt disable bits (I, F flags)
- **CPSR/SPSR**: Current/Stacked Program Status Registers
- **GIC (Generic Interrupt Controller)**: Standard interrupt controller for ARM

**ARM Interrupt Vector Table (Example):**
```assembly
; ARM Vector Table (at 0x00000000 or 0xFFFF0000)
.org 0x00
b reset_handler          ; Reset
b undefined_handler     ; Undefined instruction
b svc_handler           ; Software interrupt (SWI)
b prefetch_abort_handler
b data_abort_handler
nop                      ; Reserved
b irq_handler           ; IRQ (external interrupt)
b fiq_handler           ; FIQ (fast interrupt)
```

### 7.3 Comparison: x86 vs ARM Interrupt Handling

| Feature | x86 | ARM |
|---------|-----|-----|
| Interrupt Controller | PIC/APIC | GIC |
| Vector Table | IDT (flexible location) | Fixed base address |
| Non-Maskable Interrupt | NMI pin | FIQ (Fast IRQ) |
| Maskable Interrupts | INTR pin | IRQ pin |
| Priority Levels | Programmable (APIC) | Programmable (GIC) |
| Interrupt Types | Vectored | Vectored |

## 8. Interrupt Latency

**Interrupt Latency** is the time between interrupt signal assertion and the start of ISR execution.

### 8.1 Components of Interrupt Latency

1. **Hardware detection delay**: Time for interrupt controller to recognize
2. **Interrupt acknowledge**: CPU response to interrupt
3. **Context saving**: Saving registers and state
4. **Vector fetch**: Loading ISR address from IVT/IDT
5. **Branch to ISR**: Jump to interrupt handler

### 8.2 Factors Affecting Latency

- **Interrupt disable time**: When interrupts are masked
- **Current instruction completion**: Long-running instructions (e.g., string operations)
- **Priority inversion**: Low-priority ISR blocked by higher-priority wait
- **Cache effects**: Cache misses increase fetch time
- **Bus contention**: Multiple devices competing for bus

### 8.3 Reducing Interrupt Latency

```c
// Techniques for reducing interrupt latency in embedded systems

// 1. Use FIQ (Fast Interrupt Request) in ARM
void __attribute__((interrupt("FIQ"))) fast_irq_handler() {
    // Minimal processing - only critical operations
    // FIQ has dedicated registers (r8-r12)
}

// 2. Deferred interrupt handling (Linux approach)
void hardirq_handler() {
    // Only acknowledge interrupt, schedule softirq
    schedule_softirq();
}

void softirq_handler() {
    // Heavy processing deferred here
    process_network_packet();
}

// 3. Interrupt coalescing (for high-frequency interrupts)
void disk_irq_handler() {
    // Don't process each interrupt immediately
    // Batch multiple events before processing
    if (++event_count >= BATCH_SIZE) {
        process_batched_events();
    }
}
```

### 8.4 Latency Comparison by System Type

| System Type | Typical Latency |
|-------------|-----------------|
| Real-time systems | < 10 μs |
| General-purpose OS | 10-100 μs |
| Embedded Linux | 50-500 μs |
| Windows/macOS | 100 μs - 10 ms |

## 9. Examples with Code

### Example 1: Writing a Simple ISR in C (Linux Kernel)

```c
// Linux Kernel Module: Custom Interrupt Handler

#include <linux/kernel.h>
#include <linux/module.h>
#include <linux/interrupt.h>

#define IRQ_NO 1  // Keyboard IRQ on x86

// Interrupt handler function
static irqreturn_t my_interrupt_handler(int irq, void *dev_id) {
    printk(KERN_INFO "Interrupt triggered! IRQ: %d\n", irq);
    
    // For shared IRQs, return IRQ_NONE if not our device
    // return IRQ_HANDLED if we handled the interrupt
    
    return IRQ_HANDLED;
}

static int __init my_module_init(void) {
    int result;
    
    // Register interrupt handler
    result = request_irq(IRQ_NO, my_interrupt_handler, IRQF_SHARED,
                         "my_interrupt", "my_device");
    
    if (result) {
        printk(KERN_ERROR "Failed to request IRQ %d\n", IRQ_NO);
        return result;
    }
    
    printk(KERN_INFO "Interrupt handler registered successfully\n");
    return 0;
}

static void __exit my_module_exit(void) {
    // Free the interrupt
    free_irq(IRQ_NO, "my_device");
    printk(KERN_INFO "Interrupt handler unregistered\n");
}

module_init(my_module_init);
module_exit(my_module_exit);
MODULE_LICENSE("GPL");
```

### Example 2: System Call Implementation using Software Interrupts (x86 Real Mode)

```assembly
; Simple DOS-like System Call using INT instruction

; Define system call numbers
%define SYS_READ    0x01
%define SYS_WRITE   0x02
%define SYS_EXIT    0x04

section .data
    message db "Hello, World!", 0x0A
    msg_len equ $ - message

section .text
    global _start

_start:
    ; System call: Write to stdout (int 0x80 - Linux syscalls)
    ; For demonstration, using DOS INT 21h
    
    ; DOS Function: Display string
    mov ah, 0x09        ; Function 09h - Display String
    mov dx, message     ; DS:DX -> String
    int 0x21            ; Call DOS interrupt
    
    ; DOS Function: Exit
    mov ah, 0x4C        ; Function 4Ch - Terminate
    mov al, 0x00        ; Return code 0
    int 0x21            ; Call DOS interrupt

; Linux System Call Example (32-bit)
section .text
    global _start2

_start2:
    ; sys_write(fd=1, buf=message, count=13)
    mov eax, 4          ; sys_write syscall number
    mov ebx, 1          ; stdout
    mov ecx, message    ; buffer address
    mov edx, msg_len    ; length
    int 0x80            ; Generate software interrupt
    
    ; sys_exit(0)
    mov eax, 1          ; sys_exit syscall number
    mov ebx, 0          ; exit code
    int 0x80            ; Generate software interrupt
```

```c
// ARM Cortex-M Interrupt Handler Example

// Register definitions for ARM Cortex-M
#define NVIC_ISER (*(volatile unsigned int *)0xE000E100)
#define NVIC_ICPR (*(volatile unsigned int *)0xE000E280)

void enable_timer_interrupt(void) {
    // Enable Timer 0 interrupt in NVIC
    NVIC_ISER = (1 << 4);  // IRQ 4 for Timer 0
}

void disable_timer_interrupt(void) {
    NVIC_ICPR = (1 << 4);  // Clear pending bit
}

// Timer interrupt handler
void __attribute__((interrupt)) TIMER0_IRQHandler(void) {
    static unsigned int tick_count = 0;
    
    // Clear timer interrupt flag
    // (Device-specific, shown conceptually)
    // TIMER0->IRQ = 1;
    
    tick_count++;
    
    // Example: Toggle LED every 1000 ticks
    if (tick_count % 1000 == 0) {
        toggle_led();
    }
    
    // Exit from interrupt (ARM Cortex-M specific)
    // This instruction causes exception return

}
```

## 10. Key Takeaways

1. **Interrupt Mechanism**: Interrupts allow CPUs to respond to asynchronous events without continuous polling, improving overall system efficiency and responsiveness.

2. **Interrupt Cycle**: The complete sequence involves detecting the interrupt, saving the current state, jumping to the ISR, executing it, and restoring the state to resume normal execution.

3. **Hardware vs Software**: Hardware interrupts are asynchronous (external devices), while software interrupts are synchronous (INT instruction, system calls).

4. **Interrupt Controllers**: PIC (8259A) manages legacy systems with limited IRQ lines; APIC supports modern multi-core systems with many interrupt sources and advanced routing.

5. **Vectored Interrupts**: Each interrupt source has a unique vector that points to its ISR in the Interrupt Vector Table (x86) or Vector Table (ARM).

6. **Priority & Nesting**: Priority levels determine which interrupts can preempt others. Nested interrupts allow higher-priority ISRs to preempt lower-priority ones.

7. **Architecture-Specific**: x86 uses IDT with APIC; ARM uses GIC with configurable vector tables. Both support priority-based handling and MSI.

8. **Interrupt Latency**: The total delay from interrupt occurrence to ISR execution; affected by interrupt disable time, current instruction completion, and hardware design.

9. **Real-World Importance**: Interrupts are fundamental to OS functionality, I/O operations, real-time systems, and multitasking.
