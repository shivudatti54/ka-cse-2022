# Software Requirement Specification (SRS)

## Introduction

Software Requirement Specification (SRS) is a crucial document that outlines the requirements of a software system. It serves as a communication tool between the stakeholders, including customers, developers, and project managers, to ensure that everyone is on the same page regarding the software's functionality, performance, and quality. A well-written SRS document helps prevent misunderstandings, reduces errors, and ensures that the software meets the customer's expectations.

The importance of SRS cannot be overstated. It provides a clear understanding of the software's requirements, which helps developers create a system that meets the customer's needs. Moreover, SRS serves as a reference point for testing and validation, ensuring that the software is thoroughly tested and validated before its release.

## Key Concepts

### 1. Functional Requirements

Functional requirements describe the software's functionality, including its inputs, processing, and outputs. They outline what the software should do, how it should behave, and what features it should have. Functional requirements are typically written in a specific format, using verbs such as "shall" or "must."

Example: "The system shall allow users to log in using their username and password."

### 2. Non-Functional Requirements

Non-functional requirements describe the software's quality attributes, such as performance, security, usability, and reliability. They outline how the software should behave, rather than what it should do. Non-functional requirements are often written in a more subjective format, using adjectives such as "fast" or "secure."

Example: "The system should respond to user input within 2 seconds."

### 3. User Requirements

User requirements describe the needs and expectations of the software's users. They outline what the users want to achieve with the software and how they want to interact with it. User requirements are often written in a more informal format, using natural language.

Example: "The user wants to be able to search for products by name or category."

### 4. System Requirements

System requirements describe the technical specifications of the software, including its hardware, software, and network requirements. They outline what the software needs to run, including processor speed, memory, and operating system.

Example: "The system requires a minimum of 4GB RAM and a 2.4GHz processor."

## Examples

### Example 1: Writing a Functional Requirement

Suppose we are developing a software system for managing a library's book collection. One of the functional requirements might be:

"The system shall allow librarians to add new books to the collection."

To write this requirement, we might use the following steps:

1. Identify the actor: The actor is the librarian.
2. Identify the action: The action is adding new books to the collection.
3. Identify the object: The object is the book collection.
4. Write the requirement: "The system shall allow librarians to add new books to the collection."

### Example 2: Writing a Non-Functional Requirement

Suppose we are developing a software system for online shopping. One of the non-functional requirements might be:

"The system should respond to user input within 2 seconds."

To write this requirement, we might use the following steps:

1. Identify the quality attribute: The quality attribute is responsiveness.
2. Identify the metric: The metric is response time.
3. Identify the target value: The target value is 2 seconds.
4. Write the requirement: "The system should respond to user input within 2 seconds."

## Exam Tips

1. When writing functional requirements, use verbs such as "shall" or "must" to indicate what the software should do.
2. When writing non-functional requirements, use adjectives such as "fast" or "secure" to describe the software's quality attributes.
3. User requirements should be written in a more informal format, using natural language.
4. System requirements should be written in a technical format, outlining the software's technical specifications.
5. When writing requirements, use specific language and avoid ambiguity.
6. Requirements should be testable and measurable.
7. Requirements should be prioritized based on their importance and urgency.
