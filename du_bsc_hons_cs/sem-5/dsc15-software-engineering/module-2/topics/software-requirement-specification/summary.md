# Software Requirement Specification (SRS) - Summary

## Key Definitions and Concepts

- **Software Requirement Specification (SRS)**: A formal document that completely describes the functions, performance, and constraints of the software system to be developed. It serves as a contract between developers and stakeholders.

- **Requirements Engineering**: The systematic process of identifying, documenting, validating, and managing software requirements throughout the project lifecycle.

- **Functional Requirements**: Describe specific behaviors and functions the system must perform—what the system should do.

- **Non-Functional Requirements**: Specify quality criteria (performance, security, reliability, usability, maintainability) that judge system operation rather than specific behaviors.

## Important Formulas and Theorems

**IEEE 830-1998 Characteristics of Good SRS**:
- Correct, Unambiguous, Complete, Consistent
- Ranked (prioritized), Verifiable, Modifiable, Traceable

## Key Points

1. SRS serves as the primary communication medium between stakeholders and developers, forming the basis of the development contract.

2. The eight IEEE 830 characteristics ensure SRS quality: correctness, unambiguity, completeness, consistency, ranking, verifiability, modifiability, and traceability.

3. Functional requirements define system functions (inputs, processing, outputs); non-functional requirements define quality attributes.

4. Non-functional requirements include: Performance (response time, throughput), Security (encryption, authentication), Reliability (uptime, MTBF), Usability (learning time), Maintainability (modifiability).

5. Requirements engineering process has five phases: Elicitation → Analysis → Specification → Validation → Management.

6. User requirements are written in natural language for stakeholders; system requirements are technical specifications for developers.

7. SRS document structure follows IEEE 830: Introduction, Overall Description, Specific Requirements, Appendices.

8. Requirements traceability enables impact analysis and helps manage changes throughout development.

9. Ambiguous terms like "user-friendly" or "fast" must be replaced with measurable criteria in specifications.

10. Requirements prioritization is essential for incremental development and scope management.

## Common Mistakes to Avoid

1. **Confusing functional and non-functional requirements**: Remember that "the system shall calculate GST at 18%" is functional, while "the calculation shall complete within 1 second" is non-functional.

2. **Using vague language**: Avoid terms like "efficient," "fast," or "user-friendly" without specific measurable criteria.

3. **Incomplete specifications**: Missing edge cases or error handling scenarios lead to incomplete SRS.

4. **Ignoring non-functional requirements**: Students often focus only on functional requirements, but non-functional requirements are equally important for system quality.

5. **Writing requirements as features rather than behaviors**: Requirements should describe what the system does, not how it should be implemented.

## Revision Tips

1. Practice writing requirements for simple systems like library management, student attendance systems, or online shopping carts.

2. Memorize the IEEE 830 eight characteristics and their meanings—they appear frequently in exams.

3. Create a table comparing functional vs non-functional requirements with examples from different domains.

4. Review previous DU question papers to understand the pattern and weightage of SRS questions.

5. When answering scenario questions, always structure your answer with proper requirement IDs and clear, testable specifications.
