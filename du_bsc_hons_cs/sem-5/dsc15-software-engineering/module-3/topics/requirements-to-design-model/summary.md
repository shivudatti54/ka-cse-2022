# Requirements To Design Model — Software Engineering

## Introduction

The **Requirements to Design Model** is a critical phase in the software development life cycle (SDLC) that bridges the gap between what stakeholders need (requirements) and how the system will be built (design). In the Delhi University BSc (Hons) Computer Science (NEP 2024 UGCF) syllabus, this topic covers the transformation of user requirements into a structured design model—forming the foundation for implementation.

---

## Key Concepts

### 1. **Requirements Engineering**
- Systematic process of **eliciting, analyzing, validating, and managing** user requirements
- Goal: Produce a complete requirements document (SRS) that serves as a contract between developers and stakeholders

### 2. **Types of Requirements**
- **Functional Requirements**: Features/functions the system must perform (e.g., user login, data processing)
- **Non-Functional Requirements**: Quality attributes—performance, security, reliability, usability, scalability
- **Domain Requirements**: Industry-specific rules and regulations

### 3. **Requirements Gathering Techniques**
- Interviews, questionnaires, surveys
- Observation, prototyping
- Brainstorming sessions
- Document analysis
- Use case modeling

### 4. **Requirements Analysis**
- **E-R (Entity-Relationship) Modeling**: Identifies entities, attributes, and relationships
- **Use Case Diagrams**: Visual representation of system interactions
- **Data Flow Diagrams (DFD)**: Shows how data moves through the system
- **Class Diagrams**: Object-oriented analysis for identifying classes and their relationships

### 5. **Requirements Specification (SRS)**
- Standard document (IEEE 830 format) containing:
  - Introduction, Functional requirements, Non-functional requirements
  - System models, Glossary
- Characteristics: Correct, complete, unambiguous, verifiable, traceable

### 6. **Requirements Validation**
- Reviews, inspections, prototyping
- Feasibility checks
- Consistency and conflict resolution
- Acceptance criteria definition

### 7. **Requirements Management**
- **Traceability Matrix**: Links requirements to design components and test cases
- Change management process
- Version control of requirements

### 8. **Transition from Requirements to Design**
- **Architectural Design**: High-level structure, modules, layers, technology stack
- **Detailed Design**: Class diagrams, sequence diagrams, algorithm specifications, database schema
- Design principles: **Modularity, Coupling, Cohesion, Abstraction**

### 9. **Design Models**
- **Structural Models**: Class diagrams, object diagrams, component diagrams
- **Behavioral Models**: Sequence diagrams, state diagrams, activity diagrams
- **Architectural Patterns**: MVC, Layered, Client-Server, Microservices

---

## Conclusion

The **Requirements to Design Model** process is essential for building quality software. A well-defined requirements specification directly leads to a robust design model, minimizing rework, cost overruns, and stakeholder dissatisfaction. For exams, remember: **Gather → Analyze → Specify → Validate → Design** is the core workflow. Focus on types of requirements, SRS components, and the transformation from analysis models (E-R, Use Cases) to design models (Class, Sequence diagrams).
