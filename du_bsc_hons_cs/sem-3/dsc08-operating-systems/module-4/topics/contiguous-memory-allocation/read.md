# Contiguous Memory Allocation

## Introduction

Memory management is one of the most critical functions of any operating system, and contiguous memory allocation represents one of the earliest and most fundamental approaches to organizing main memory. In this technique, each process is allocated a single, continuous block of physical memory addresses during its entire execution lifetime. This method, while conceptually simple, forms the foundation upon which more complex memory management strategies are built.

Contiguous memory allocation was prevalent in early computing systems and remains relevant today in embedded systems, real-time operating systems, and certain specialized environments where simplicity and predictability are paramount. Understanding this topic is essential for computer science students because it illuminates the fundamental challenges of memory management—fragmentation, allocation efficiency, and memory protection—that continue to influence modern operating system design.

In the University of Delhi's BSc (Hons) Computer Science curriculum, this topic appears in the Operating Systems paper and is crucial for understanding how operating systems manage physical RAM. The concepts learned here directly relate to other memory management techniques like paging and segmentation, making it a cornerstone topic for exam preparation.

## Key Concepts

### 1. Definition and Basic Principle

Contiguous memory allocation is a memory management technique where each process is stored in a single, uninterrupted segment of physical memory. The operating system maintains a table (often called the memory partition table) that tracks which memory locations are allocated to which processes and which remain free.

For example, if a process requires 100KB of memory and the operating system decides to load it at physical address 5000, the process will occupy addresses 5000 through 5099. All instructions and data for that process must fit within this continuous range.

### 2. Types of Contiguous Allocation

**Fixed Partitioning (Static Partitioning)**

In fixed partitioning, the operating system divides main memory into a predetermined number of partitions at system startup. These partitions can be of equal size or unequal sizes. Each partition can accommodate exactly one process at a time.

Advantages:
- Simple to implement and manage
- Minimal hardware support required
- Provides memory protection through partition boundaries

Disadvantages:
- Internal fragmentation: wasted space within allocated partitions
- Limited degree of multiprogramming (maximum number of processes equals number of partitions)
- Inflexible partition sizes may not match process requirements

**Variable Partitioning (Dynamic Partitioning)**

In variable partitioning, partitions are created dynamically based on the exact memory requirements of incoming processes. Initially, the entire memory is a single free block. When a process arrives, exactly the required amount of memory is allocated.

This approach eliminates internal fragmentation but introduces external fragmentation—the accumulation of small, non-contiguous free memory blocks that cannot satisfy large process requests.

### 3. Memory Allocation Algorithms

When a process requires memory, the operating system must select which free block to allocate. Several algorithms exist:

**First Fit**
- Allocates the first free block that is large enough to accommodate the process
- Fast allocation (doesn't require searching entire list)
- Tends to allocate toward the beginning of memory
- Time Complexity: O(n) where n is number of free blocks

**Best Fit**
- Allocates the smallest free block that is large enough
- Minimizes wasted space within the allocated block
- Requires searching entire free list
- Creates smallest possible leftover hole
- Time Complexity: O(n)
- Disadvantage: Tends to create many small fragments

**Worst Fit**
- Allocates the largest available free block
- Creates the largest possible leftover hole, increasing chances of satisfying future requests
- Time Complexity: O(n)
- Less commonly used as it depletes large blocks quickly

**Next Fit**
- Similar to first fit but starts searching from where the last allocation ended
- Distributes allocations more uniformly across memory
- Time Complexity: O(n)

### 4. Fragmentation

**Internal Fragmentation**
- Occurs in fixed partitioning when the allocated partition is larger than the requested memory
- The unused space within the partition is "wasted"
- Example: Process needs 50KB but gets placed in a 64KB partition → 14KB wasted

**External Fragmentation**
- Occurs in variable partitioning when total free memory is sufficient but not contiguous
- Small, scattered free blocks cannot satisfy large process requests
- Can be addressed through compaction (moving all allocated blocks together)

### 5. Address Binding and Memory Protection

**Address Binding** is the process of converting logical addresses (generated by programs) to physical addresses (actual memory locations). In contiguous allocation, this typically occurs at:
- Compile time: Absolute addresses (rare in modern systems)
- Load time: Relocatable addresses (common in early systems)
- Execution time: Dynamic address translation (most flexible)

**Relocation Register** (also called base register) stores the starting physical address of a process's partition. The **Limit Register** stores the size of the partition. Every memory access is checked against these registers:

```
Physical Address = Base Register + Logical Address
Logical Address must be < Limit Register
```

This mechanism provides:
- Memory protection: Process cannot access memory outside its partition
- Relocation: Same program can be loaded at different physical addresses

### 6. Compaction

Compaction (or garbage collection in memory context) is a technique to combat external fragmentation. It involves moving all allocated memory blocks to one end of memory, consolidating all free space into a single contiguous block.

Challenges:
- Requires stopping all process execution (brief downtime)
- Must update all relocation registers
- Time-consuming for large memory
- Only practical in certain systems

## Examples

### Example 1: Fixed Partitioning Problem

**Question**: Memory has 4 partitions of sizes 100KB, 200KB, 300KB, and 400KB (in that order). Processes arrive with memory requirements: P1 (180KB), P2 (300KB), P3 (50KB), P4 (200KB). Show how each process is allocated using first-fit approach.

**Solution**:

Initial state: All partitions are free.

**P1 (180KB)**: First-fit checks partition 1 (100KB) - too small. Partition 2 (200KB) - sufficient. Allocate P1 to Partition 2.
- Partition 1: Free (100KB)
- Partition 2: P1 (180KB) - internal fragmentation = 200 - 180 = 20KB
- Partition 3: Free (300KB)
- Partition 4: Free (400KB)

**P2 (300KB)**: First-fit checks Partition 1 - too small. Partition 2 - occupied. Partition 3 (300KB) - exact fit. Allocate P2 to Partition 3.
- Partition 1: Free (100KB)
- Partition 2: P1 (180KB)
- Partition 3: P2 (300KB)
- Partition 4: Free (400KB)

**P3 (50KB)**: First-fit checks Partition 1 (100KB) - sufficient. Allocate P3 to Partition 1.
- Partition 1: P3 (50KB) - internal fragmentation = 100 - 50 = 50KB
- Partition 2: P1 (180KB)
- Partition 3: P2 (300KB)
- Partition 4: Free (400KB)

**P4 (200KB)**: First-fit checks Partition 1 - occupied. Partition 2 - occupied. Partition 3 - occupied. Partition 4 (400KB) - sufficient. Allocate P4 to Partition 4.
- Partition 1: P3 (50KB)
- Partition 2: P1 (180KB)
- Partition 3: P2 (300KB)
- Partition 4: P4 (200KB) - internal fragmentation = 400 - 200 = 200KB

Total Internal Fragmentation = 20 + 50 + 200 = 270KB

### Example 2: Variable Partitioning with Best Fit

**Question**: Consider a memory with free holes: H1(100KB), H2(300KB), H3(200KB), H4(150KB). Processes arrive: P1(180KB), P2(100KB), P3(50KB). Using Best Fit, show the allocation and remaining holes.

**Solution**:

Initial free holes: [H1:100], [H2:300], [H2:200], [H4:150]

**P1 (180KB)**: Find smallest hole ≥ 180KB
- H1(100KB) - too small
- H4(150KB) - too small  
- H3(200KB) - sufficient, smallest
- H2(300KB) - sufficient but larger

Allocate P1 to H3 (200KB)
- Remaining in H3: 200 - 180 = 20KB (new hole H3')
- New free list: [H1:100], [H4:150], [H3':20], [H2:300]

**P2 (100KB)**: Find smallest hole ≥ 100KB
- H1(100KB) - exact fit
- H4(150KB) - larger
- H3'(20KB) - too small
- H2(300KB) - larger

Allocate P2 to H1 (100KB)
- H1 is fully used, remove from free list
- New free list: [H3':20], [H4:150], [H2:300]

**P3 (50KB)**: Find smallest hole ≥ 50KB
- H3'(20KB) - too small
- H4(150KB) - sufficient, smallest
- H2(300KB) - larger

Allocate P3 to H4 (150KB)
- Remaining in H4: 150 - 50 = 100KB (new hole H4')
- Final free list: [H3':20], [H4':100], [H2:300]

### Example 3: Address Translation

**Question**: A process has base register = 10000 and limit register = 2000. For each logical address, determine if it is valid and find the physical address: (a) 500, (b) 1500, (c) 2200

**Solution**:

Valid logical addresses: 0 to (limit register - 1) = 0 to 1999

**(a) Logical Address 500**:
- Is 500 < 2000? Yes, valid
- Physical Address = Base + Logical = 10000 + 500 = 10500

**(b) Logical Address 1500**:
- Is 1500 < 2000? Yes, valid
- Physical Address = Base + Logical = 10000 + 1500 = 11500

**(c) Logical Address 2200**:
- Is 2200 < 2000? No, INVALID
- This would generate a trap/interrupt (segmentation fault)
- The process is attempting to access memory outside its allocated partition

## Exam Tips

1. **Remember the difference between internal and external fragmentation**: Internal occurs in fixed partitioning (wasted space within partitions), external occurs in variable partitioning (scattered free holes). This is a frequently asked exam question.

2. **Algorithm characteristics**: First fit is fastest; best fit minimizes internal fragmentation but creates more external fragmentation; worst fit creates larger holes but is rarely optimal.

3. **Calculation of fragmentation**: For fixed partitioning, total internal fragmentation = sum of (partition size - process size) for all occupied partitions.

4. **Address translation formula**: Always remember Physical Address = Base Register + Logical Address. Verify logical address < limit register before translation.

5. **Memory protection mechanism**: The combination of base and limit registers prevents processes from accessing memory belonging to other processes or the OS.

6. **Compaction purpose**: Understand that compaction (or memory packing) is used to combat external fragmentation by moving all allocated blocks together.

7. **Degree of multiprogramming**: In fixed partitioning, maximum number of concurrent processes equals number of partitions. This is a key limitation.

8. **Real-world relevance**: While contiguous allocation has limitations, it's still used in embedded systems and scenarios where simplicity and deterministic memory access time are critical.

9. **Comparison questions**: Be prepared to write short notes comparing fixed vs variable partitioning, or comparing first fit, best fit, and worst fit algorithms.

10. **Numerical problems**: Practice allocation problems similar to Examples 1 and 2—they frequently appear in DU exams.
